# M3-06 设计说明（GM↔Gate/Game 启动控制与集群就绪状态消息）

1. 当前不再定义独立的“配置下发”消息。所有节点都从进程启动时传入的统一配置文件读取整个服务器组配置以及自身实例配置，因此 GM 无需在Inner 网络重复下发同一份静态或启动期配置快照。
2. `M3-06` 现在在协议登记层固定一组稳定的启动控制消息：`Inner.ClusterReadyNotify`（`1201`）、`Inner.ServerStubOwnershipSync`（`1202`）、`Inner.GameServiceReadyReport`（`1203`）、`Inner.GameDirectoryQuery`（`1204`）与 `Inner.GameDirectoryNotify`（`1205`）。这组消息共同覆盖“Stub ownership 分配”“Game 本地 ready 上报”“Gate 查询 / 订阅路由目录”“GM 推送目录变化”以及“GM 下发集群 ready 结论”这几类启动控制交互。
3. `Inner.ClusterReadyNotify` 仍保持最小而稳定的单向通知体，只固定包含 `readyEpoch(uint64)`、`clusterReady(bool)`、`statusFlags(uint32)` 与 `serverNowUnixMs(uint64)`。其中 `clusterReady = true` 表示 GM 当前已确认服务器组满足对外服务前提；对 Gate 来说，这直接决定后续客户端入口是否允许开放。
4. `Inner.ServerStubOwnershipSync` 与 `Inner.GameDirectoryNotify` 采用“全量快照 + epoch”语义：接收方只保留最新 `assignmentEpoch` / `routeEpoch` 的结果，旧版本重复到达时应忽略。`Inner.GameServiceReadyReport` 必须显式携带它所基于的 `assignmentEpoch`，避免 GM 把旧 ownership 下的 ready 误聚合进新一轮启动结论。
5. 共享编码规则继续复用现有Inner 网络基础设施：所有整数字段使用网络字节序，`bool` 使用 `uint8(0/1)`，保留字段或保留位当前必须为 `0`。协议基础层只负责消息体编解码与最小语义校验，不把 `PacketHeader` 封包、ZeroMQ 收发或节点本地应用逻辑耦合进来。
6. 本条目先把 `msgId`、方向、字段与时序约束稳定下来；具体运行时落点仍由后续条目承接：`M3-12` / `M3-13` 负责路由目录查询与维护，`M3-14` 负责 `GameServiceReadyReport`，`M3-15` 负责 GM 侧 ready 聚合与路由目录变化通知，`M4-02` 负责消费 `ClusterReadyNotify` 控制客户端入口。
7. 当前 native codec 实现仍以 `ClusterReadyNotify` 为主，并已具备 roundtrip、精确 wire layout、非法布尔值、非零 `statusFlags`、输入截断、尾部多余字节与空输出参数等失败路径验证。其余新登记消息先完成稳定协议登记，待对应运行时条目实现时再补齐具体 codec 与自动化测试。
