# M2-03 设计说明（时间工具与定时器封装）

1. 在 `src/native/core` 内新增独立时间模块，统一暴露 `steady_clock` / `system_clock` 的类型别名、当前时间获取、毫秒换算与截止时间辅助函数，避免后续事件循环、心跳与超时逻辑各自重复拼装 `chrono` 细节。
2. 时间工具优先围绕单调时钟组织：超时、重试、Tick 与内部调度全部使用 `steady_clock` 计算，只有日志、监控或外部协议需要绝对时间时才使用 `system_clock` 与 Unix 毫秒时间戳互转。
3. 定时器封装基于 standalone `asio::steady_timer` 实现，但不自带线程或事件循环；它只绑定外部传入的 `asio::io_context`，为后续 `M2-04` / `M2-05` 的线程与主循环骨架提供可复用的最小调度单元。
4. `SteadyTimer` 统一提供单次触发与周期触发两种模式；重复模式要求正间隔，重新启动同一个定时器时应替换旧等待，而不是并存多个待触发任务。
5. 取消与析构语义必须稳定：显式 `Cancel()`、重新启动覆盖旧任务，或对象析构时，都不应再把旧回调误投递给业务层，避免后续心跳、超时与 Tick 驱动中出现悬挂回调。
6. 内部实现使用代次号和共享状态来屏蔽过期等待完成，确保“重置后旧回调失效”“回调内再次启动/取消不产生重复触发”“析构后异步完成不访问悬空对象”这三类基础生命周期问题一次解决。
7. 当前阶段不承诺跨线程并发安全；定时器的启动、取消与回调默认由其所属 `io_context` 线程串行驱动，跨线程调度与工作线程封装留给后续条目处理。
8. 自动化验证使用最小 native 测试可执行程序，至少覆盖：毫秒换算、单次触发、重复触发、自取消、取消后不触发、析构后不触发，以及重启覆盖旧等待这几类核心语义。
