# M2-03 设计说明（时间工具与定时器封装）

1. 在 `src/native/core` 内新增独立时间模块，统一暴露 `steady_clock` / `system_clock` 的类型别名、当前时间获取、毫秒换算与截止时间辅助函数，避免后续事件循环、心跳与超时逻辑各自重复拼装 `chrono` 细节。
2. 时间工具优先围绕单调时钟组织：超时、重试、Tick 与内部调度全部使用 `steady_clock` 计算，只有日志、监控或外部协议需要绝对时间时才使用 `system_clock` 与 Unix 毫秒时间戳互转。
3. 对外只暴露 `TimerManager`、`TimerID` 与错误码辅助函数；单个 `asio::steady_timer` 对象完全收敛到 manager 内部管理，后续 `M2-04` / `M2-05` 以及更高层超时逻辑只和 manager 交互。
4. 定时器创建接口统一返回“成功时大于 `0` 的 `TimerID`，失败时负错误码”的结果值；调用方先判断结果是否为有效 `TimerID`，再通过错误码解析函数主动获取失败原因文本。
5. `TimerManager` 统一提供单次创建、周期创建、按 `TimerID` 重置和取消四类基础操作；定时器一旦创建成功即视为激活，生命周期直接绑定到 manager 内部登记项，不再额外维护可见的 `active` 状态位。
6. manager 内部用 `TimerID -> timer entry` 的映射组织生命周期，并继续用代次号屏蔽过期等待完成，确保“重置后旧回调失效”“回调内再次重置/取消不产生重复触发”“manager 析构后异步完成不访问悬空对象”这三类问题一次解决。
7. 当前阶段不承诺跨线程并发安全；定时器的创建、重置、取消与回调默认由其所属 `io_context` 线程串行驱动，跨线程调度与工作线程封装留给后续条目处理。
8. 自动化验证使用最小 native 测试可执行程序，至少覆盖：毫秒换算、创建成功后按 `TimerID` 取消/重置、单次触发、重复触发、自取消、manager 析构后不触发，以及错误码与错误消息映射这几类核心语义。
