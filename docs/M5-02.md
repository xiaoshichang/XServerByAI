# M5-02 设计说明（C++→C# 入口绑定与函数指针获取）

1. 在 `src/native/host/` 现有 `ManagedRuntimeHost` 之上补一层“托管入口绑定”能力，职责是基于 `M5-01` 已缓存的 `load_assembly_and_get_function_pointer` 委托，从 `XServer.Managed.GameLogic.dll` 解析 `XServer.Managed.GameLogic.Interop.GameNativeExports` 的首版导出集合，并输出稳定的已类型化函数指针表；`M5-02` 只解决“入口能否被绑定并校验通过”，不提前承担 `Game` 业务调用编排。
2. native 侧需要把 `docs/MANAGED_INTEROP.md` 中 ABI v1 的常量、`ManagedInitArgs` / `ManagedMessageView` 布局，以及 `ManagedGetAbiVersionFn`、`ManagedInitFn`、`ManagedOnMessageFn`、`ManagedOnTickFn` 四个函数指针 typedef 落到 `host` 模块可复用的头文件中，并统一保存 assembly path、assembly-qualified type name 与方法名常量，避免后续 `M6-01` / `M6-02` 重复拼接字符串或手写签名。
3. 绑定顺序固定为：先解析并调用 `GameNativeGetAbiVersion`，再比对 `XS_MANAGED_ABI_VERSION`，版本一致后继续解析 `GameNativeInit`、`GameNativeOnMessage` 与 `GameNativeOnTick`。若出现 runtime 尚未加载、导出类型/方法解析失败、返回空函数指针、或 ABI 版本不匹配等情况，立即返回稳定 interop 错误码并保持“未绑定”状态，禁止留下部分成功的残缺入口集。
4. `src/managed/GameLogic/` 需要补齐最小可解析导出实现：新增 `Interop` 命名空间下的 blittable 结构与 `GameNativeExports` 静态导出类，统一使用 `UnmanagedCallersOnly` + `CallConvCdecl`，并为项目开启 `unsafe` 编译支持。当前阶段导出方法体保持最小占位成功返回，只保证 native 可以解析并调用，不提前引入 `ServerEntity` / `ServerStubEntity` 等后续业务框架。
5. 公开 API 与错误码需要覆盖绑定阶段新增失败面：至少区分“宿主尚未加载”“导出解析失败”“ABI 版本不匹配”“入口尚未绑定不可调用”等场景，并同步更新 `docs/ERROR_CODE.md`、`ManagedHostErrorCode` 及其名称/消息映射，确保数值、规范名和错误消息保持一致，继续落在 `4000-4099` interop 号段。
6. 宿主侧应缓存一份只读的已绑定导出表，并在 `Unload` 时一并清空；外部调用方只能通过查询接口获知“是否已绑定”及取得函数指针视图，不再允许在多个位置各自解析导出，避免后续 `Game` 进程重复绑定、拿到不一致入口集，或绕过 ABI 校验直接调用 managed 导出。
7. 自动化验证需要同时覆盖成功与失败路径：成功路径通过构建后的 `GameLogic` 程序集完成 runtime 装载、导出解析、ABI 校验和四个函数指针非空断言；失败路径至少覆盖未先 `Load` 就尝试绑定、ABI 返回值与常量不匹配、以及导出方法名/导出类型错误时返回稳定错误码。测试产物应尽量通过最小额外程序集或可控错误导出构造失败场景，而不是手工修改正式业务程序集。
