# M3-08 设计说明（Gate 角色启动骨架与 GM Inner 链路接入）

1. `M3-08` 的目标是把 `GateNode` 从“启动后立刻自停的占位节点”升级为“能沿统一 `ServerNode` 生命周期持续运行、并与 `GM` 建立稳定 Inner 主动连接的骨架”。本条目不提前实现注册、心跳、目录查询或客户端会话逻辑；这些仍分别留给 `M3-09`、`M3-12` / `M3-13` 与 `M4-*`。
2. 现有 `InnerNetwork` 只覆盖 `GM` 使用的被动监听模式，因此本条目需要把它扩展为同时支持基于 `M2-07` `ZmqActiveConnector` 的主动连接模式，或引入等价的 Gate/Game 共用 active-side Inner 封装；但对节点层暴露的边界仍应保持 `Init()` / `Run()` / `Uninit()`、连接状态查询、原始消息回调和 `Send()` 这组稳定语义。这样 `M3-10` 的 `Game` 也能直接复用同一套 GM 链路骨架。
3. Gate 侧配置来源改为完全依赖统一配置查询：`cluster_config().gm.inner_network_listen_endpoint` 是 Gate 实际要连接的 `GM` 远端地址，而 Gate 自身后续若需要暴露给其他节点使用的地址，也由统一配置文件中的 `gate.<NodeId>.innerNetwork.listenEndpoint` 查询得到，而不是在运行时向 `GM` 主动上报。`M3-08` 先完成这些配置的校验、标准化 `tcp://host:port` 文本和启动期日志，不提前把 Gate 自身的 `innerNetwork` 监听真正打开成被动监听端。
4. `GateNode::OnInit()` 继续装配 `ClientNetwork` 占位组件，但新增核心职责转为 GM Inner 主动链路：创建 active-mode inner 组件、绑定连接状态与接收回调，并缓存 Gate 运行时所需的节点基础元数据与配置派生地址。`GateNode::OnRun()` 启动后不再立刻 `RequestStop()`，而是进入等待外部停止的长期运行态，让后续注册/心跳逻辑能够直接挂接在现有生命周期上。
5. 主动链路状态模型至少稳定收敛为“未启动 / 连接中 / 已连接 / 已断开”，并通过日志和轻量状态访问对外暴露，避免 `GateNode` 直接感知 ZeroMQ monitor 细节。对 `M3-08` 来说，只要求 Gate 能识别自己是否已经接入 `GM` Inner 链路、能接收原始单帧 payload，并对当前尚未消费的消息做记录后忽略，不提前落地注册/心跳/目录协议处理。
6. 为了让 `M3-09` 能直接接上注册与心跳，本条目需要把 Gate 侧运行时上下文固定下来：至少保留 `GM` 远端 endpoint、由配置查询得到的 Gate 本地 Inner endpoint、当前连接状态、最近错误文本，以及统一的 packet send helper 或原始消息回调挂点。下一条应只需要追加 codec、请求序号与定时器，而不必返工底层连接生命周期；如果后续确认注册协议也不再携带节点 Inner 地址，则应在 `M3-09` 同步收敛该消息语义。
7. `ClientNetwork` 在 `M3-08` 仍保持占位实现，但启动语义必须与 `M4-02` 的“受控开放客户端入口”兼容：当前可以完成配置解析与对象初始化，不应把“是否对外开放客户端入口”的决策提前固化在本条目里。也就是说，`M3-08` 的重点是让 Gate 进程稳定存活并接入 `GM`，而不是提前完成客户端监听开放策略。
8. 自动化验证围绕“Gate 骨架可持续运行并成功接入 GM”展开。测试至少覆盖：缺失 `GM` Inner 配置或 Gate 自身 `innerNetwork.listenEndpoint` 时初始化失败；Gate 在后台运行时能与 `GM` Inner 监听建立 active connection；连接状态变化和关键 endpoint 被写入日志；`RequestStop()` 后 Gate 能稳定退出并清理 connector / timer / client placeholder。若实现新增 active-side Inner 封装，也应补对应单测验证启动、状态回落与原始消息接收回调。
