# M3-03 设计说明（GM 注册表数据结构）

1. 在 `src/native/node` 内新增独立的 GM 控制面注册表模块，例如 `ProcessRegistry`，专门负责维护当前服务器组内 Gate/Game 的活动注册信息。`M3-03` 只交付可复用的数据结构与查询/更新接口，不在本条目内直接接入 ZeroMQ 消息收发。
2. 注册表条目统一沉淀为一个稳定结构，例如 `ProcessRegistryEntry`。它至少保存 `processType`、`nodeId`、`pid`、`startedAtUnixMs`、`serviceEndpoint`、`buildVersion`、`capabilityTags`、最新 `LoadSnapshot`，并额外保留 `lastHeartbeatAtUnixMs`、`serviceReady` 与控制链路关联标识（如 `routingId`）等运行时字段，为后续心跳、就绪聚合和路由下发复用。
3. 注册表内部以稳定 `nodeId` 作为唯一主键维护活动实例，并可选提供按当前控制链路标识（如 `routingId`）的辅助索引。同一时刻同一 `nodeId` 只允许存在一个活动注册；当前阶段不再引入单独的租约号或恢复标识，把节点身份和活动会话都直接收敛到 `nodeId + 当前链路` 语义。
4. 注册表对外提供最小但稳定的操作集合：注册新节点、按 `nodeId` 或链路标识移除节点、按 `nodeId` 或链路标识查询、更新心跳时间与负载快照、更新服务就绪状态、导出当前活动快照。所有可失败接口统一返回专属错误码，详细文本由错误消息映射函数提供，保持与仓库当前错误码约定一致。
5. 当前阶段不再生成额外租约号。`M3-04` 的注册成功响应只需要返回心跳参数与服务器时间；`M3-05` 的心跳请求也直接用稳定 `nodeId` 或当前控制链路定位活动节点，避免在还没有断线恢复和租约迁移需求时引入额外概念。
6. 注册表在结构层执行最小语义校验：`processType` 只能是 `Gate` 或 `Game`，`nodeId` 不能为空，`serviceEndpoint.host` 不能为空，`serviceEndpoint.port` 不能为 `0`。是否把冲突映射为 `3001 Control.NodeIdConflict`、是否接受同链路重注册等协议级规则，留给 `M3-04` 的请求处理层决定；但底层数据结构必须能稳定区分“参数非法”“NodeID 冲突”“节点不存在”等失败类别。
7. 为支持后续 `M3-12`、`M3-14` 与 `M3-15`，注册表导出的快照应具备确定性顺序，至少保证同一进程组内按 `nodeId` 稳定排序；同时为 `Game` 条目暴露构建路由目录所需的字段子集，为 ready 聚合暴露 `serviceReady` 与最新负载信息，避免后续条目再次回头重写基础容器。
8. 自动化验证以模块级单测为主：至少覆盖 Gate/Game 注册成功、`nodeId` 冲突拒绝、按主键与链路索引查询、移除后索引回收、心跳/负载更新、`serviceReady` 更新、活动快照排序与空表场景。这样 M3-04/M3-05 后续只需要把网络请求映射到该注册表接口，而不必重复验证容器语义。
