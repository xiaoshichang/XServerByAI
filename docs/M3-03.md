# M3-03 设计说明（GM 注册表数据结构）

1. 在 `src/native/node` 中引入可复用的 GM 注册表模块 `ProcessRegistry`，专门维护当前已完成注册的 `Gate` / `Game` 节点信息。`M3-03` 只交付数据结构、查询/更新接口与最小校验，不直接处理 ZeroMQ 收发。
2. 外部写入模型统一为 `ProcessRegistryRegistration`，内部稳定存储为 `ProcessRegistryEntry`。字段至少包括 `processType`、`nodeId`、`pid`、`startedAtUnixMs`、`innerNetworkEndpoint`、`buildVersion`、`capabilityTags`、`load`、`routingId`、`lastHeartbeatAtUnixMs` 与 `innerNetworkReady`；这样既能承接 `M3-04` / `M3-05` 的注册与心跳，也能为后续 ownership、ready 聚合和 Gate 目录投影保留所需信息。
3. 注册表以 `nodeId` 作为主键，以当前 Inner 链路的 `routingId` 作为辅助索引；同一时刻同一 `nodeId` 只允许存在一个活动注册项。当前不处理断线恢复、历史连接并存或节点迁移，语义保持为“`nodeId + 当前链路` 唯一”。
4. 对外接口至少覆盖：注册、按 `nodeId` / `routingId` 注销、按 `nodeId` / `routingId` 查询、按 `nodeId` / `routingId` 更新心跳时间与负载、按 `nodeId` / `routingId` 更新 `innerNetworkReady`，以及返回稳定排序的 `Snapshot()`。
5. `M3-04` 与 `M3-05` 只需要把 `Inner.NodeRegister` / `Inner.NodeHeartbeat` 请求映射到这些接口上：注册成功时插入条目，心跳成功时刷新 `lastHeartbeatAtUnixMs` 与 `load`，链路失效时按 `nodeId` 或 `routingId` 剔除。因此本条目要先把状态模型和最小错误集稳定下来，避免后续条目重复发明注册语义。
6. 最小校验规则保持稳定：`processType` 只允许 `Gate` 与 `Game`；`nodeId` 不允许为空；`innerNetworkEndpoint.host` 不允许为空；`innerNetworkEndpoint.port` 不允许为 `0`；重复 `nodeId` 映射为 `3001 Inner.NodeIdConflict` 对应的底层冲突；重复 `routingId` 视为链路级冲突并显式拒绝。
7. 为支撑后续 `M3-12`、`M3-14` 与 `M3-15`，`Snapshot()` 应保证稳定输出顺序，并保留 `innerNetworkReady` 与最新 `load` 等聚合所需字段。这样 `GM` 后续在生成 ownership 快照、ready 聚合或对 Gate 暴露 Game 目录时，可以直接复用同一份注册表视图。
8. 自动化验证以模块级单测为主：覆盖 Gate/Game 注册成功、主键/辅助索引冲突、查询、注销、心跳字段更新、`innerNetworkReady` 更新、`Snapshot()` 排序与空注册表行为。ZeroMQ 级链路行为由后续条目在 `M3-04` / `M3-05` 中承接。
