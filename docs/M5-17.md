# M5-17 C# 日志桥接到 C++ 日志设计

本文档定义 `M5-17` 的实现范围：在 `M2-01` 已有的 native 日志模块与 `M5-02` 已有的 `GameNative*` 入口绑定基础之上，为托管侧补齐“把 C# 日志回送到 C++ logger”的最小闭环。当前阶段目标是让 `Game` 进程内的 managed 代码能够复用 native 统一日志出口，而不是各自写独立文件、控制台或第三套日志框架。

## 目标与边界

1. 托管侧日志必须进入现有 C++ `Logger`，遵循同一份 `logging.minLevel`、文件输出、刷新与轮转策略。
2. 本里程碑只覆盖 `Game` 进程加载 `XServer.Managed.GameLogic` 时的日志桥接；`GM` 当前仅消费 catalog 导出，不在本次范围内扩展独立的 managed 初始化链路。
3. 当前阶段只传输基础日志字段：`level`、`category`、`message`。不在本里程碑内引入结构化 `context`、异常堆栈对象封送或跨语言日志作用域。
4. 桥接失败时不得影响 `GameNativeInit` 之后的基础运行；托管日志可以降级为 no-op，但不能破坏现有 ready/ownership ABI 行为。

## 实现清单

1. 扩展 managed↔native 日志回调 ABI
   - 在 `ManagedInterop.h` 与 `ManagedAbi.cs` 中新增托管日志桥接所需的 callback 签名、日志等级枚举与 UTF-8 参数约定。
   - `ManagedNativeCallbacks` 保持向后兼容的“struct_size + 回调指针”模式，新增日志回调字段时同步提升 `XS_MANAGED_ABI_VERSION` / `ManagedAbi.Version`。
   - 回调参数约定为只读 UTF-8 buffer + length，避免托管侧依赖 null-terminated 字符串。

2. 在 native `GameNode` 中提供日志回调落点
   - `GameNode::InitializeManagedRuntime()` 在构造 `ManagedNativeCallbacks` 时，把新的 managed log callback 一并传入 `GameNativeInit`。
   - 回调收到托管日志后，统一映射到 `xs::core::Logger::Log(...)`，并保留 managed 侧传入的 `category` 与等级。
   - native 侧对空消息、超长 UTF-8 或无效 level 做保守防御，避免托管 bug 直接污染进程稳定性。

3. 在 managed 侧建立统一日志桥接门面
   - `GameNativeInit` 保存 native 传入的日志 callback，并通过一个集中封装的 `NativeLoggerBridge` / `ManagedLogger` 暴露给托管运行时使用。
   - 日志门面负责 `LogLevel` 映射、UTF-8 编码与空回调降级，不让业务代码直接操作函数指针。
   - 日志门面默认提供最小 API：`Trace/Debug/Info/Warn/Error/Fatal` 或统一 `Log` 入口，便于后续 `M6-09` 继续扩展可观测性输出。

4. 为当前托管运行时接入基础日志点
   - 在 `GameNativeInit`、ownership apply/reset、ready callback 等已有托管运行时关键路径补最少量日志，确保桥接链路能被真实调用。
   - 日志内容优先覆盖“初始化成功/失败边界、ownership 应用结果、ready 状态变化”这些后续排障最需要的点，不把本特性扩展成大规模日志重写。

5. 补齐 host/interop 验证
   - 在 `tests/cpp/host_runtime_tests.cpp` 中增加日志 callback 捕获，验证 `GameNativeInit` 后托管日志能回调到 native，并带回正确的 level/category/message。
   - 如需覆盖 ABI 兼容性，更新现有 static assert 与 interop 结构大小断言，确保 native / managed 双边布局一致。
   - 若当前 `Game` 节点集成测试已具备读取日志文件能力，可补一条最小链路验证托管日志实际写入 native 日志输出。

## 验证出口

1. `ManagedInitArgs.NativeCallbacks` 能稳定把托管日志回送到 native，且不会破坏现有 `on_server_stub_ready` 回调。
2. 托管侧调用日志门面后，native 日志文件中能看到对应的等级、分类与消息文本。
3. ABI 版本、结构大小与字段偏移在 native / managed 双边保持一致；不一致时会按现有 interop 机制明确失败。
4. 当 native 未提供日志 callback 或托管日志编码失败时，运行时保持可继续执行，不影响现有 `GameNativeInit`、ownership 与 ready 逻辑。
