# M2-12 设计说明（注册消息编解码）

1. 在 `src/native/net/message` 内新增独立的 `RegisterCodec` 模块，专门承载 `Control.ProcessRegister`（`msgId = 1000`）消息体的编码、解码与基础语义校验；当前阶段只处理消息体，不把 `PacketHeader` 封包逻辑重复塞进该模块。
2. 将控制面共享结构抽到独立头文件中：`Endpoint`、`LoadSnapshot` 与 `ControlProcessType` 由注册消息与心跳消息共同复用，避免在多个 codec 中重复维护同一套字段定义。
3. 对外显式建模三类消息体：注册请求、注册成功响应、注册失败响应；请求体因包含变长 `string` / `string[]`，额外提供显式 wire-size 计算接口，继续要求调用方自行提供目标缓冲区，不在协议基础层隐藏堆分配。
4. 注册请求编码顺序固定为 `processType(uint16)`、`processFlags(uint16)`、`nodeId(string16)`、`pid(uint32)`、`startedAtUnixMs(uint64)`、`serviceEndpoint(Endpoint)`、`buildVersion(string16)`、`capabilityTags(uint16 count + repeated string16)`、`load(LoadSnapshot)`；其中 `Endpoint` 编码顺序固定为 `host(string16)`、`port(uint16)`。
5. 编解码阶段执行最小但稳定的语义校验：`processType` 只能取 `Gate(1)` 或 `Game(2)`，`processFlags` 当前必须为 `0`，`nodeId` 不能为空，`serviceEndpoint.host` 不能为空，`serviceEndpoint.port` 必须非 `0`，`capabilityTags` 数量不得超过 `32` 项；其余字符串继续遵循 `string16` 的长度边界。
6. 注册成功响应体编码顺序固定为 `registrationId(uint64)`、`heartbeatIntervalMs(uint32)`、`heartbeatTimeoutMs(uint32)`、`serverNowUnixMs(uint64)`；编解码阶段要求 `registrationId != 0` 且 `heartbeatIntervalMs < heartbeatTimeoutMs`，与 `docs/PROCESS_CONTROL.md` 的控制面约束保持一致。
7. 注册失败响应体编码顺序固定为 `errorCode(int32)`、`retryAfterMs(uint32)`；失败原因继续由 `errorCode` 承载，成功/失败语义由外层 `PacketHeader.flags` 区分，不在 codec 内派生新的 `msgId`。
8. 自动化验证至少覆盖：请求 wire-size 计算、请求/成功响应/失败响应 roundtrip、精确线缆字节布局、`processFlags` 非零拒绝、非法 `processType` 拒绝、空 `nodeId` / 空 `serviceEndpoint.host` / `serviceEndpoint.port = 0` 拒绝、`capabilityTags` 超上限拒绝、成功响应心跳时序非法拒绝、输入截断与尾部多余字节拒绝，以及空输出参数等参数错误路径。
