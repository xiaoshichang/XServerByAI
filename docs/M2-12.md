# M2-12 设计说明（注册消息编解码）

1. 在 `src/native/net/message` 中实现独立的 `RegisterCodec` 模块，专门负责 `Inner.NodeRegister`（`msgId = 1000`）消息体的编码、解码、wire-size 计算与最小语义校验。本条目只处理消息体本身，不把 `PacketHeader`、消息路由或节点生命周期耦合进 codec。
2. 注册消息共享一组稳定的基础结构：`Endpoint`、`LoadSnapshot` 与 `InnerProcessType`。这些定义统一放在共享消息类型头文件中，供注册与心跳 codec 共用，避免多个模块分别维护一份相似但不兼容的字段布局。
3. 注册消息固定分成 3 类：`RegisterRequest`、`RegisterSuccessResponse`、`RegisterErrorResponse`。其中 `RegisterRequest` 包含变长字符串与字符串数组，因此 codec 需要提供显式的 `GetRegisterRequestWireSize()`，由调用方先计算目标缓冲区大小，再执行正式编码。
4. `RegisterRequest` 的字段顺序固定为：`processType(uint16)`、`processFlags(uint16)`、`nodeId(string16)`、`pid(uint32)`、`startedAtUnixMs(uint64)`、`innerNetworkEndpoint(Endpoint)`、`buildVersion(string16)`、`capabilityTags(string16[])`、`load(LoadSnapshot)`。`Endpoint` 自身固定为 `host(string16)`、`port(uint16)`。
5. 当前阶段执行最小但稳定的语义校验：`processType` 只允许 `Gate` 与 `Game`；`processFlags` 当前必须为 `0`；`nodeId` 不允许为空；`innerNetworkEndpoint.host` 不允许为空；`innerNetworkEndpoint.port` 不允许为 `0`；`capabilityTags` 数量不允许超过 `32`；所有 `string16` 都必须满足 `uint16` 长度边界。
6. `RegisterSuccessResponse` 固定为 `heartbeatIntervalMs(uint32)`、`heartbeatTimeoutMs(uint32)`、`serverNowUnixMs(uint64)`，并要求 `heartbeatIntervalMs < heartbeatTimeoutMs`，与 `docs/PROCESS_INNER.md` 的时序约束保持一致。`RegisterErrorResponse` 固定为 `errorCode(int32)`、`retryAfterMs(uint32)`，成功或失败仍由外层 `PacketHeader.flags` 区分，不为错误响应分配新的 `msgId`。
7. codec 层统一返回 `RegisterCodecErrorCode`，同时区分二进制错误和协议语义错误，例如 `InvalidProcessType`、`InvalidProcessFlags`、`InvalidNodeId`、`InvalidInnerNetworkEndpointHost`、`InvalidInnerNetworkEndpointPort`、`TooManyCapabilityTags`、`InvalidHeartbeatTiming`、`TrailingBytes`。节点层收到这些结果后可以更稳定地映射到 Inner 错误码与日志。
8. 自动化验证至少覆盖：wire-size 计算、请求/成功响应/失败响应 roundtrip、稳定 wire bytes、非法 `processType`/`processFlags`/endpoint/标签数量 拒绝、成功响应时间参数非法拒绝、输入截断、尾部多余字节、空输出参数与缓冲区不足等失败路径。现有 `tests/cpp/register_codec_tests.cpp` 已按这组规则建立覆盖。
