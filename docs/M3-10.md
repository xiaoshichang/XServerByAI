# M3-10 设计说明（Game 角色启动骨架与 GM Inner 链路接入）

1. `M3-10` 的目标是把当前“初始化后立刻自停”的 `GameNode` 升级为可长期运行的骨架节点，并沿用 `M3-08` 已落地的 active-side `InnerNetwork` 能力主动接入 `GM`。本条目不提前实现注册、心跳、ownership 接收、`Game -> Gate` 注册或托管逻辑装载；这些分别留给 `M3-11`、`M3-12`、`M3-13`、`M5-*` 与 `M6-*`。
2. `Game` 侧不重新发明 Inner 传输层，而是复用当前 `InnerNetwork` 的 `ActiveConnector` 模式，与 `GateNode` 保持一致的节点边界：节点层只关心 `Init()` / `Run()` / `Uninit()`、连接状态查询、原始 payload 回调与后续可复用的发送挂点，不直接暴露 ZeroMQ connector 细节。
3. `GameNode::OnInit()` 需要严格校验并缓存最小运行时上下文：`gm.innerNetwork.listenEndpoint` 必填且合法，`game.<NodeId>.innerNetwork.listenEndpoint` 必填且合法，并统一标准化为 `tcp://host:port` 文本；`managed.assemblyName` 继续沿用配置解析层的默认值 `XServer.Managed.GameLogic`，在本条目只做缓存与日志输出，不触发 CLR / `nethost` 装载。
4. `GameNode` 需要补齐与 `GateNode` 对齐的可观测骨架状态：至少保留 `gm_inner_remote_endpoint`、`configured_inner_endpoint`、当前连接状态缓存、`started_at_unix_ms`、固定 `build_version`、`managed_assembly_name` 与最近一次协议/运行时错误文本。这样 `M3-11` 可以直接在现有节点状态上挂注册与心跳，而不用回头重做启动期上下文。
5. `GameNode::OnInit()` 的核心工作是创建 active-mode `InnerNetwork`，把本地配置得到的 `Game` Inner endpoint 作为本节点的已配置地址缓存下来，把 `GM` Inner endpoint 作为远端连接目标，绑定连接状态与消息回调，并在初始化成功后输出稳定日志字段。`Game` 当前没有 `ClientNetwork`，因此本条目的骨架重点应集中在 GM Inner 接入与托管配置元数据保留，而不是额外扩展本地监听组件。
6. `GameNode::OnRun()` 启动后不再主动 `RequestStop()`，而是进入等待外部停止的长期运行态；`OnUninit()` 负责按稳定顺序释放 inner connector、重置缓存状态并清空错误。连接状态变化需要记录 `nodeId`、`gmInnerRemoteEndpoint`、`configuredInnerEndpoint` 等关键上下文，便于后续联调时直接判断 `Game` 是否已经接入 `GM`。
7. 对来自 `GM` 的原始 Inner payload，本条目只要求 `GameNode` 能接到回调并形成“记录后忽略”的安全默认行为：暂不解析注册/心跳响应，不因为未知消息或暂未消费的消息而崩溃，但要留下最小可诊断日志与协议错误文本挂点。后续 `M3-11` 可以在同一回调入口上继续挂接 register / heartbeat 协议处理。
8. 自动化验证围绕“`Game` 骨架可持续运行并成功接入 `GM`”展开，建议新增 `tests/cpp/node_game_node_tests.cpp`，至少覆盖：缺失 `GM` Inner 配置时初始化失败、缺失 `game.<NodeId>.innerNetwork.listenEndpoint` 时初始化失败、`GM + Game` 联动启动后 `Game` 能观察到 active connection、`Run()` 在未请求停止前不会提前结束，以及 `RequestStop()` 后能够干净退出并在日志中留下关键 endpoint / 连接状态字段。
