# M2-10 设计说明（消息分发器）

1. 在 `src/native/net` 内新增独立 `MessageDispatcher` 模块，职责限定为“把已经通过 `M2-09` 解包得到的 `PacketView` 按 `header.msg_id` 路由到精确处理器”；本阶段不把 ZeroMQ 收包、业务解码或线程切换混入同一层。
2. 分发器只接收 `PacketView`，不重复承担线缆字节解析责任；调用方必须先通过 `DecodePacket()` 完成 `PacketHeader` 与 payload 校验，再把结果交给分发器，保持“封包解析”和“消息路由”边界清晰。
3. 注册接口采用精确 `msgId -> handler` 映射，不提前引入号段匹配、优先级链或多播语义；后续 `M3` Inner 网络消息与 `M4` Gate↔Game 中继只需要稳定的一对一路由能力即可落地。
4. 分发器显式拒绝非法注册：`msgId == 0`、空回调、重复注册都必须返回独立错误码；已注册消息可按 `msgId` 解除绑定，也支持一键 `Clear()` 以便进程生命周期切换或测试复位。
5. 分发阶段对未注册 `msgId` 返回明确的 `HandlerNotFound`；收到 `msgId == 0` 的 `PacketView` 直接视为非法输入，避免把“未初始化消息”静默路由到业务层。
6. 处理器签名保持最小化，只接收 `const PacketView&` 并在调用线程内同步执行；分发器不拥有业务上下文、不做异常恢复和异步排队，后续运行时如需串接日志、metrics 或线程切换，可在更高层包装。
7. 错误反馈采用独立 `MessageDispatchErrorCode` + `MessageDispatchErrorMessage()`，覆盖非法参数、非法 `msgId`、空处理器、重复注册与缺失处理器等场景，保持与现有 native 基础模块一致的“成功返回 `0`”约定。
8. 自动化验证覆盖：成功注册与精确路由、handler 接收到原始 `PacketView` 内容、重复注册拒绝、`msgId == 0` 注册与分发拒绝、空处理器拒绝、移除处理器与 `Clear()` 后不再命中，以及“未知消息返回 `HandlerNotFound`”等关键路径。
