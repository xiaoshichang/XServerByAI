# M3-05 设计说明（心跳处理与超时检测）

1. 本条目的目标是在 `GM` 侧补齐 `Inner.NodeHeartbeat`（`msgId = 1100`）的完整处理闭环，包括心跳请求解析、活动节点刷新、失效链路判定、超时剔除与错误响应。实现重心放在 `GM` Inner 协调面，而不是扩展 `Gate` / `Game` 侧发送逻辑。
2. 推荐在 `src/native/node` 中引入独立的 `GmInnerService` 或等价职责对象，统一持有 `ProcessRegistry`、默认心跳参数、超时扫描定时器、失效 `routingId` 记录与原始消息分发。`GmNode` 只负责生命周期装配与消息入口转发，不把心跳协议细节散落在节点入口层。
3. 心跳入口继续复用现有 `PacketCodec + HeartbeatCodec`：`GM` 只接受 `msgId = 1100`、不带 `Response/Error` 标记、且 `seq != 0` 的请求；packet 解包失败、payload 非法、保留位非零或 envelope 不符合约束时，必须形成可诊断失败路径，而不是把错误报文当成普通噪音忽略。
4. 心跳定位语义以当前 Inner 链路为准。收到请求后先按 `routingId` 命中活动注册项，命中后刷新 `lastHeartbeatAtUnixMs` 与最近一次 `LoadSnapshot`；如果该链路从未完成注册，则返回 `3003 Inner.NodeNotRegistered`；如果链路曾经有效但已被替换、超时剔除或显式失效，则返回 `3004 Inner.ChannelInvalid`。为支持这一区分，需要额外维护一份短期保留的“失效 routingId”集合。
5. 超时检测统一依赖 `MainEventLoop::timers()` 提供的 `TimerManager`。`GM` 在 owner-thread 上创建单个重复定时器周期扫描活动注册项，以 `serverNowUnixMs - lastHeartbeatAtUnixMs >= heartbeatTimeoutMs` 判定超时；超时后应把对应节点从 `ProcessRegistry` 剔除，并同步把其 `routingId` 放入失效集合，避免残余心跳把旧会话错误恢复成活动状态。
6. 成功响应统一回写 `heartbeatIntervalMs`、`heartbeatTimeoutMs` 与 `serverNowUnixMs`；失败响应统一回写 `errorCode`、`retryAfterMs` 与 `requireFullRegister`。当前阶段默认要求失败后完整重注册，因此 `3003` / `3004` 应返回 `requireFullRegister = true`；纯请求非法场景可返回 `3005` 且不强制完整重注册。
7. 自动化验证应覆盖两类场景：一类是服务级单测，验证超时扫描确实会剔除过期节点，并把旧 `routingId` 标记为失效；另一类是端到端 Inner 网络测试，验证活动节点心跳成功刷新、未知链路返回 `3003`、失效链路返回 `3004`、畸形请求返回 `3005`，以及回包的 `msgId` / `flags` / `seq` / payload 都符合协议约束。现有 `tests/cpp/node_gm_inner_service_tests.cpp` 已围绕这组语义建立覆盖。
