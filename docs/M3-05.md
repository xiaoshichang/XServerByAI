# M3-05 设计说明（心跳处理与超时检测）

1. 在 `src/native/node` 内为 GM 控制面补齐独立的心跳处理模块，建议命名为 `GmControlService` 或等价职责对象。该模块统一持有 `ProcessRegistry`、控制面消息分发、默认心跳参数（`5000ms / 15000ms`）以及超时扫描状态；`GmNode` 只负责生命周期装配，不再把心跳协议细节散落到 `InnerNetwork` 或节点入口里。
2. 扩展 `InnerNetwork` 的被动监听骨架，使其除了记录监听状态与收包日志外，还能把 `routingId + payload` 交给上层处理，并支持按 `routingId` 回发原始二进制响应。这样 `M3-04` 的注册请求处理与本条目的心跳处理都能复用同一条 GM 控制链路，而不是各自绕过现有网络封装。
3. 心跳入口统一走现有 `PacketCodec` + `HeartbeatCodec`。GM 只接受 `msgId = 1100`、`flags` 不带 `Response/Error`、`seq != 0` 的请求；解包失败、`msgId` 不匹配、payload 语义非法或保留位非零时，当前阶段按控制面错误响应返回，避免把损坏报文静默吞掉。
4. 心跳定位语义以当前控制链路为准：收到请求后先按 `routingId` 命中 `ProcessRegistry` 活动条目，命中后更新 `lastHeartbeatAtUnixMs` 与最新 `LoadSnapshot`。若当前链路没有活动注册，则返回 `3003 Control.NodeNotRegistered`；若链路曾经拥有节点但已被替换、超时剔除或显式失效，则返回 `3004 Control.ControlChannelInvalid`。为支持这一区分，本条目需要补一个轻量的“失效控制链路”记录，而不是只依赖现有活动注册表。
5. 超时检测统一依赖 `MainEventLoop::timers()` 提供的 `TimerManager`，由 GM 在 owner-thread 上创建单个重复定时器周期扫描活动注册。扫描时按 `serverNowUnixMs - lastHeartbeatAtUnixMs >= heartbeatTimeoutMs` 判定超时，把对应节点从 `ProcessRegistry` 剔除，并同步把其 `routingId` 标记为短期失效链路，确保超时后的残余心跳不会再次被当成“从未注册”处理。
6. 成功响应统一回写 `heartbeatIntervalMs`、`heartbeatTimeoutMs` 与 GM 当前 `serverNowUnixMs`；失败响应统一回写 `errorCode`、`retryAfterMs` 与 `requireFullRegister`。当前阶段默认要求失败后完整重注册，因此 `3003` / `3004` 都返回 `requireFullRegister = true`，`retryAfterMs` 先保持 `0`，后续如果 M3-09 / M3-11 需要退避策略再扩展。
7. 日志与测试都围绕“控制链路语义”展开。日志至少覆盖：心跳解包失败、未知链路心跳、有效心跳刷新、节点超时剔除。自动化验证至少覆盖：活动节点心跳成功并刷新负载、未知链路返回 `3003`、失效链路返回 `3004`、超时扫描剔除节点、超时后旧链路心跳不再恢复节点、以及 `GmNode`/`InnerNetwork` 端到端收包回包路径能够承载 `Control.ProcessHeartbeat`。
