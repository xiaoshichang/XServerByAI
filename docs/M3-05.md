# M3-05 设计说明（心跳处理与超时检测）

1. 本条目的目标是在 `GM` 侧补齐 `Inner.NodeHeartbeat`（`msgId = 1100`）的处理闭环，包括心跳请求解析、远端会话刷新、超时标记与成功响应。实现重心直接落在 `GmNode` 与 `ServerNode::inner_network_remote_sessions()`，不再引入额外的专用服务对象或独立的失效路由集合。
2. `GM` 通过 `InnerNetworkSessionManager` 管理所有远端节点会话，目录项类型统一为 `InnerNetworkSession`。注册成功会写入 `nodeId`、`processType`、`innerNetworkEndpoint`、`routingId`、负载与最近心跳时间；后续心跳继续在同一条 session 上刷新这些运行时字段，而不是再维护另一套 GM 专用会话状态。
3. `GmNode::HandleInnerMessage()` 先完成 packet header 预读，再按 `msgId` 分发到 `HandleRegisterMessage()` 与 `HandleHeartbeatMessage()`。心跳 handler 只接受 `msgId = 1100`、`flags = 0`、`seq != 0` 的请求；packet 解包失败、payload 非法、`statusFlags != 0` 或 envelope 不符合约束时，只记录日志并忽略。
4. 心跳定位语义以当前 `routingId` 命中 `InnerNetworkSession` 为准。若 channel 未注册，则记录 `3003 Inner.NodeNotRegistered` 并忽略；命中后刷新 `load`、`lastHeartbeatAtUnixMs`、`lastServerNowUnixMs`、`registered`、`connectionState`，并清除 `heartbeat_timed_out` 标记。
5. 超时检测统一依赖 `MainEventLoop::timers()` 提供的 repeating timer。`GM` 周期扫描 `inner_network_remote_sessions` 快照，以 `serverNowUnixMs - lastHeartbeatAtUnixMs >= heartbeatTimeoutMs` 判定超时；超时后只在 session 上标记 `heartbeat_timed_out = true`、`registered = false`、`connection_state = Stopped` 与最近一次错误文本，不从 manager 删除，也不额外维护单独的失效路由目录。
6. 成功响应统一回写 `heartbeatIntervalMs`、`heartbeatTimeoutMs` 与 `serverNowUnixMs`。当前心跳协议没有失败响应分支；未知、畸形或不合法请求只会被记录并忽略，不再通过独立的心跳失败 payload 或“要求全量重注册”语义驱动恢复。
7. 自动化验证应覆盖：活动节点心跳刷新；未知 channel 心跳被忽略；超时只修改 session 标记而不删除目录项；超时后允许后续注册替换旧 `routingId`；以及回包 envelope 始终保持 `msgId = 1100` + `Response` 的成功语义。现有 GM 节点与注册相关单测已围绕这组行为建立覆盖。
