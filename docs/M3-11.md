# M3-11 设计说明（Game→GM 注册与心跳）

1. `M3-11` 的目标是在 `M3-10` 已完成的 `Game`→`GM` active Inner 链路之上，补齐真正的启动期协议闭环：`Game` 在链路可用后先向 `GM` 发送 `Inner.NodeRegister (1000)`，注册成功后再进入 `Inner.NodeHeartbeat (1100)` 周期心跳。由于 `InnerNetwork` 现在支持同时维护多个 connector，`Game` 在本条目中需要同时连接 `GM` 与全部目标 `Gate`；但 `Game -> Gate` 的注册/心跳 payload 处理仍属于后续条目。
2. `GameNode` 的本地 runtime state 已简化为“进程启动骨架 + 托管程序集名”，核心字段只保留 `startedAtUnixMs` 与 `managedAssemblyName`。所有远端节点的连接/注册/心跳状态统一存放在 `ServerNode::inner_network_remote_sessions()` 的 `InnerNetworkSession` 中。`Game` 不再在 runtime state 中重复缓存 GM 连接状态、GM 远端 endpoint、构建版本或能力标签这类派生信息；连接状态从 session 查询，远端 endpoint 需要时从 cluster config 查询。
3. 初始化阶段，`Game` 需要先在 session manager 中登记一个稳定的 `GM` 会话项，再为每个配置中的 `Gate` 预登记一个远端会话项；随后用这些配置构造 `InnerNetwork` connectors。`GM` 对应的 session 用于本条目的注册与心跳闭环；各 `Gate` session 当前主要承载连接状态与后续扩展入口，为 `Game -> Gate` 阶段铺路。
4. 注册流程应绑定在 `GM` connector 的连接状态机上，而不是只在 `OnRun()` 启动时尝试一次：当 `GameNode` 观察到 `GM` Inner 连接进入 `Connected` 且当前未注册、也没有注册请求在途时，立即发送完整 `RegisterRequest`。请求字段与 `Gate` 侧保持同一协议语义，但 `processType` 必须是 `Game`，`nodeId` 使用 `Game0` 这类实例名，`pid` 取 `ServerNode::pid()`，`startedAtUnixMs` 固定为当前进程启动时间，`innerNetworkEndpoint` 使用配置文件中的 `game.<NodeId>.innerNetwork.listenEndpoint`，`buildVersion` 使用稳定占位 `dev`，`capabilityTags` 为空，`load` 先上报全零快照。
5. `GameNode::HandleGmMessage()` 需要按 Gate 相同的层次进行处理：先统一完成 `PacketCodec` 解包、flags / `seq` / `msgId` 校验，再把 `Inner.NodeRegister` 分发给 `HandleRegisterResponse()`、把 `Inner.NodeHeartbeat` 分发给 `HandleHeartbeatResponse()`。注册响应允许 success / error 两种 envelope；心跳响应只接受 `Response` 标记。遇到 `seq = 0`、flags 非法、解码失败或未知 `msgId` 时，只记录结构化日志并更新最近一次协议错误文本，不让 `Game` 因协议噪音崩溃。
6. 心跳逻辑必须严格以后置注册成功为前提：只有在解析到注册成功响应后，`Game` 才进入“已注册”态并创建 repeating timer，按 `GM` 返回的 `heartbeatIntervalMs` 周期发送心跳。每次成功心跳响应都允许刷新 GM session 中的 interval / timeout / `serverNowUnixMs`；如果参数发生变化，则重建定时器。当前不再存在独立的心跳失败 payload 或“要求全量重注册”的分支，GM 断线时只需取消心跳 timer 并清空 GM 会话态。
7. 生命周期与可观测性需要与现有 `GameNode` 骨架保持一致，并覆盖多连接器场景：GM/Gate 连接状态变化、注册发送/接受/拒绝、心跳发送/确认、定时器调度失败、关闭阶段资源释放等关键转移都应输出稳定字段，至少包含 `nodeId`、`seq`、`managedAssemblyName`、必要时带 `gateNodeId`。日志不再依赖运行时缓存的 endpoint 文本，而是使用稳定 `NodeID` 作为主要标识。
8. 自动化验证围绕“Game 成功完成 GM 注册心跳，并在多连接器与协议噪音下保持稳定”展开。测试至少应覆盖：`GM + Game` 联动启动后完成注册并收到一个以上成功心跳；带非法 envelope 的 GM heartbeat response 会被忽略；多个 `Gate` connector 的初始化与状态变化不会破坏 `Game` 的启动/退出流程；`Game` 停止时能干净取消心跳 timer 并回收 connector。现有 `tests/cpp/node_game_gm_session_tests.cpp` 已覆盖 GM 会话主路径，`tests/cpp/node_game_node_tests.cpp` 保留启动骨架覆盖。
