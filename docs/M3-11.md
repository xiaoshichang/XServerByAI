# M3-11 设计说明（Game→GM 注册与心跳）

1. `M3-11` 的目标是在 `M3-10` 已完成的 `Game`→`GM` active Inner 链路之上，补齐真正的启动期协议闭环：`Game` 在链路可用后先向 `GM` 发送 `Inner.NodeRegister (1000)`，注册成功后再进入 `Inner.NodeHeartbeat (1100)` 周期心跳。当前条目只覆盖注册、心跳、重注册、日志与测试，不提前实现 `M3-12` 的“所有节点已上线”通知接收、`M3-13` 的 `Game -> Gate` 闭环、`M3-14` 的 mesh ready / ownership 接收，或 `M3-15` 的 ready 上报与托管逻辑装载。
2. `GameNode` 需要从当前的“运行时骨架状态”升级为“运行时骨架 + GM 会话状态”两层上下文：继续保留 `M3-10` 已缓存的 `gm_inner_remote_endpoint`、`configured_inner_endpoint`、`managed_assembly_name`、`started_at_unix_ms`、`build_version` 与最近一次协议错误文本；同时新增最小但稳定的 GM 会话字段，例如 `next_seq`、`register_seq`、`heartbeat_seq`、`heartbeat_interval_ms`、`heartbeat_timeout_ms`、`last_server_now_unix_ms`、`heartbeat_timer_id`、`registered`、`register_in_flight` 与预留的 `capability_tags` / `load`。这样断线、要求全量重注册或节点退出时可以明确重置状态，而不会把旧通道数据带进新会话。
3. 注册流程应绑定在 active connector 的连接状态机上，而不是只在 `OnRun()` 启动时尝试一次：当 `GameNode` 观察到 `GM` Inner 连接进入 `Connected` 且当前未注册、也没有注册请求在途时，立即发送完整 `RegisterRequest`。请求字段与 `Gate` 保持同一协议语义，但 `processType` 必须是 `Game`，`nodeId` 使用 `Game0` 这类实例名，`pid` 取 `ServerNode::pid()`，`startedAtUnixMs` 固定为当前进程启动时间，`innerNetworkEndpoint` 使用配置文件中的 `game.<NodeId>.innerNetwork.listenEndpoint`，`buildVersion` 先沿用稳定占位值，`capabilityTags` 允许为空，`load` 先上报全零快照。
4. `GameNode` 需要从现在的“只解包后记录并忽略一切 payload”升级为最小可扩展的响应分发器：先统一完成 `PacketCodec` 解包、flags / `seq` / `msgId` 校验，再分别处理 `Inner.NodeRegister` 与 `Inner.NodeHeartbeat` 的 success / error 响应。遇到 `seq = 0`、flags 非法、解码失败、过期响应或未知 `msgId` 时，只记录结构化日志并更新最近一次协议错误文本，不让 `Game` 因协议噪音或未来功能消息崩溃。
5. `Game` 侧的收发 helper 应与现有 `GateNode` 规则保持一致：注册请求使用 `RegisterCodec` 计算大小并编码，再经 `PacketCodec` 包装为 `msgId = 1000` 的请求包；心跳请求使用 `HeartbeatCodec` 编码为固定大小 payload，再包装为 `msgId = 1100` 的请求包；发送仍走 active-mode `InnerNetwork::Send({}, packet)`。当前条目优先保证 `GameNode` 内部逻辑完整，不强制在此轮就抽象出跨 `Gate` / `Game` 共用类，但 envelope 规则、序号消费与错误处理必须与 `GateNode` 对齐，避免后续协议行为分叉。
6. 心跳逻辑必须严格以后置注册成功为前提：只有在解析到注册成功响应后，`Game` 才进入“已注册”态并创建 repeating timer，按 `GM` 返回的 `heartbeatIntervalMs` 周期发送心跳。每次成功心跳响应都允许刷新本地缓存的 interval / timeout / `serverNowUnixMs`；如果收到心跳错误且 `requireFullRegister = true`，`Game` 应先取消旧心跳 timer、清空会话态，再在当前 connector 仍连通时重新走完整注册流程；如果 connector 已断开，则只保留“待连接后重注册”状态，不做忙等重试。
7. 生命周期与可观测性需要与现有 `GameNode` 骨架保持一致，并补齐 GM 会话的关键转移日志：连接状态变化、注册发送/接受/拒绝、心跳发送/确认/拒绝、要求全量重注册、心跳 timer 创建失败或重置，都应输出稳定字段，至少包含 `nodeId`、`seq`、`gmInnerRemoteEndpoint`、`configuredInnerEndpoint`、必要时带 `managedAssemblyName`、`heartbeatIntervalMs`、`heartbeatTimeoutMs` 与错误码。`OnUninit()` 必须先取消心跳 timer、再释放 Inner connector 并清空会话状态，避免退出阶段出现悬挂 timer 或脏状态。
8. 自动化验证围绕“Game 成功完成注册心跳并可在异常响应下恢复”展开。测试建议在保留现有 `tests/cpp/node_game_node_tests.cpp` 启动骨架覆盖的基础上，新增专门的 `Game`↔`GM` 会话测试：至少覆盖 `GM + Game` 联动启动后 `GM` 注册表出现 `Game0` 且 `processType = Game`、`innerNetworkEndpoint` 与配置一致；等待一个以上心跳周期后 `lastHeartbeatAtUnixMs` 被刷新；`Game` 停止后能干净退出并清理 timer；对 `requireFullRegister = true` 的心跳错误场景，`Game` 会取消旧心跳并重新发送完整注册，而不是继续沿用失效会话发心跳。
