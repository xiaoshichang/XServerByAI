# M1-16 设计说明（配置与日志输出规范）

1. 将当前阶段进程配置组织方式、逻辑配置块、日志等级与文件输出规则沉淀为 `docs/CONFIG_LOGGING.md` 的长期维护文档，而不是把规范散落在 `PROJECT.md` 或各条目说明里。
2. 固定正式配置载体为单个 UTF-8 JSON 文件，`GM`、`Gate`、`Game` 同组实例的启动配置都放在同一文件中，由 `xserver-node <configPath> <selector>` 的 `selector` 选择具体实例。
3. 统一配置键命名为 `lowerCamelCase`，明确顶层至少包含 `serverGroup`、`logging`、`gm`、`gate`、`game`；其中 `gate` / `game` 以 `gate0`、`game0` 这类选择器作为子键。
4. 定义单文件内的解析与校验规则：代码默认值 < 顶层公共块 < 选中实例块，数组整体替换，未知顶层块与未知 `logging` 字段默认按配置错误处理，避免静默吞掉拼写错误。
5. 统一日志等级集合为 `Trace`、`Debug`、`Info`、`Warn`、`Error`、`Fatal`，并约定默认文件命名为 `GM-YYYY-MM-DD.log`、`Gate0-YYYY-MM-DD.log` 这类平铺格式，支持按天滚动、大小补充分卷、最小等级过滤与 `Error` / `Fatal` 的及时刷盘。
6. native 日志后端当前基线选型为 vendored `spdlog`，但配置键名、等级集合、文件命名与字段顺序不与单一库实现强绑定。
7. 固定日志记录的最小字段集合与文本渲染顺序，要求进程类型、实例标识、分类、消息以及 `errorCode` / 规范英文名在适用时稳定输出，为 `M5-17` 的 C# 日志桥接和 `M6-09` 的可观测性汇总提供统一边界。
8. 在 `docs/PROJECT.md`、`docs/CONVENTIONS.md`、`docs/KCP_CONFIG.md`、`docs/M1-03.md` 与 `docs/MANAGED_INTEROP.md` 中同步补充引用，确保项目总览、统一启动入口、配置命名、KCP 子块位置与托管配置路径约定保持一致。
