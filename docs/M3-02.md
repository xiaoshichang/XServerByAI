# M3-02 设计说明（GM 生命周期与节点网络骨架）

1. `src/native/node` 移除 `NodeGmRunner`、`NodeGateRunner`、`NodeGameRunner` 等 runner 入口，统一保留 `NodeRuntimeContext + RunNodeProcess(...)` 作为进程级调度壳。运行时只负责加载配置、初始化日志和事件循环、创建具体节点对象并驱动生命周期，不再承载角色专属逻辑。
2. 引入抽象基类 `ServerNode`，统一节点生命周期接口为 `Init(std::string*)`、`Run(std::string*)`、`Uninit() noexcept`。`Init` 负责资源装配与参数校验，`Run` 负责进入运行态并挂接事件循环，`Uninit` 负责关闭网络与释放资源，保证后续角色扩展继续沿同一边界演进。
3. 角色实现拆成 `GmNode`、`GateNode`、`GameNode` 三个 `ServerNode` 子类。`GmNode` 当前负责 GM 控制面监听；`GateNode` 预留集群内通信和客户端通信双通道骨架；`GameNode` 预留集群内通信骨架。这样后续 `M3-08`、`M3-10` 等条目可以直接在对应节点子类内增量实现，而无需再次调整进程入口。
4. 集群内部网络统一封装到 `InnerNetwork`。该类负责持有 `ZmqContext`、`ZmqPassiveListener`、监听状态、`configured_endpoint` / `bound_endpoint`、最近错误与 metrics，并暴露 `Init`、`Run`、`Uninit` 生命周期。当前 `M3-02` 只交付被动监听模式，满足 GM 控制面监听并为后续 Gate/Game 集群链路复用打底。
5. 面向客户端的网络行为统一预留在 `ClientNetwork`。本阶段只提供最小占位生命周期实现，让 `GateNode` 的结构先稳定下来；真正的客户端监听、会话管理与 KCP 行为留给 `M4-02` 及后续条目继续填充。
6. `RunNodeProcess(...)` 的启动顺序固定为：加载 `NodeRuntimeContext`、创建日志器、创建事件循环、按 `process_type` 构造 `ServerNode`、依次调用 `Init` 和 `Run`，停止时统一回调 `Uninit`。运行时错误按创建、初始化、运行三个阶段分别映射，便于定位节点生命周期问题。
7. 自动化验证覆盖两层：`node_runtime_tests` 校验节点分派与生命周期调度，`node_gm_node_tests` 校验 `GmNode` 对 GM 配置的约束，以及 `InnerNetwork` 的通配端口绑定、收包、状态收敛和日志输出。这样既保护新架构边界，也保留 M3-02 的行为回归。
