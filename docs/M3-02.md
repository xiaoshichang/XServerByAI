# M3-02 设计说明（GM 角色生命周期与内部控制端口监听）

1. 在 `src/native/node` 内把 GM 运行时从当前占位 `RunGmNode(...)` 提升为可管理的最小生命周期实现，但边界仍收敛在 GM runner 内部：统一由 `NodeRuntimeContext` 驱动启动，不改写 `node_main.cpp` 与已有 `RunNodeProcess(...)` 调度流程，保证后续 `M3-03`/`M3-04`/`M3-05` 只是在同一 GM 运行时上继续扩展，而不是重新拆入口。
2. GM 控制监听端口完全复用现有配置模型中的 `NodeConfig.control_listen_endpoint`，并在 GM runner 启动阶段显式校验“当前角色必须是 GM 且配置中必须存在 control listen endpoint”；若缺失或非法，则在进入稳定运行前返回清晰错误，避免把 GM 专属配置问题拖到后续注册消息处理阶段才暴露。
3. 监听实现直接复用 `M2-08` 的 `xs::ipc::ZmqPassiveListener`，并由 GM runner 在 `MainEventLoop` owner 线程上持有与驱动：启动时创建共享 `ZmqContext`、配置 `ROUTER` 监听、记录实际 bind 地址，停止时按 `MainEventLoop` 生命周期有序关闭 listener/context，确保 GM 进程具备明确的 start/stop 边界，而不是依赖进程退出硬回收。
4. 本条目只交付“控制端口可监听、可接收连接、可承载后续控制消息”的骨架，不抢跑实现注册表、注册响应、心跳租约或集群路由逻辑；收到的原始控制面 payload 当前阶段只做最小日志/计数/占位处理，把具体 `Control.ProcessRegister` / `Control.ProcessHeartbeat` 语义留给 `M3-04` 与 `M3-05`。
5. GM 运行时需要补齐最小可观测性：至少记录 GM 启动、监听地址、监听状态变化、接收到控制面消息以及关闭事件，并让日志字段继续沿用 `processType = GM` / `instanceId = GM` 语义，保证后续排查控制链路问题时能直接关联到 GM 生命周期而不是只能看到通用 runtime 占位日志。
6. 为避免把 ZeroMQ 生命周期细节散落在 `NodeGmRunner.cpp`，实现时优先抽出 GM 专属的小型运行时对象或 helper（例如 listener state / wiring 封装），负责保存 `bound_endpoint`、最近错误、消息计数与 stop 清理；但范围只覆盖 GM 内部控制监听，不提前抽象成所有角色共用的“万能节点服务容器”。
7. 自动化验证继续以 native 单元测试为主，新增对 GM runner 的最小覆盖：至少验证 GM 缺失控制端口配置时启动失败、合法配置下能成功启动监听并产生日志/监听状态、`tcp://127.0.0.1:*` 这类动态端口能暴露实际 bind 地址，以及 `MainEventLoop` 停止后 listener 会正确关闭、不污染既有 `node_runtime` 与 `ipc_passive_listener` 测试基线。
