# M3-09 设计说明（Gate→GM 注册与心跳）

1. `M3-09` 的目标是在 `M3-08` 已完成的 Gate→GM active Inner 链路之上，补齐真正的启动期协议闭环：`Gate` 在链路可用后先向 `GM` 发送 `Inner.NodeRegister (1000)`，注册成功后再进入 `Inner.NodeHeartbeat (1100)` 周期心跳。由于 `InnerNetwork` 现在支持同时作为 listener 与 connector，`Gate` 在本条目中需要一边连接 `GM`，一边监听后续来自 `Game` 的连接；但 `Game -> Gate` 的注册/心跳 payload 处理仍属于后续条目。
2. `GateNode` 的本地 runtime state 只保留最小骨架字段，例如 `startedAtUnixMs` 与 cluster ready 相关状态。`GM` 对端的注册/心跳状态统一存放在 `ServerNode::inner_network_remote_sessions()` 中，对应目录项类型为 `InnerNetworkSession`。`Gate` 不再在节点本地重复缓存 GM 连接状态或 GM 远端 endpoint 这类派生字段；连接状态从 session 读取，远端 endpoint 需要时从 cluster config 查询。
3. 初始化阶段，`Gate` 先在 session manager 中登记一个稳定的 `GM` 会话项，再构造同时具备 listener 与 connector 的 `InnerNetwork`：listener 绑定 `gate.<NodeID>.innerNetwork.listenEndpoint`，供未来 `Game` 主动接入；connector 则以 `GM` 的配置地址发起主动连接。当前阶段 listener 侧只保留必要的消息入口与日志，不在本条目里实现 `Game -> Gate` 协议闭环。
4. 注册流程绑定在 `GM` connector 的连接状态机上：当 `Gate` 观察到 `GM` Inner 连接进入 `Connected` 且当前未注册、也没有注册请求在途时，立即发送完整 `RegisterRequest`。请求字段包括 `processType = Gate`、`nodeId = Gate<index>`、`pid = ServerNode::pid()`、`startedAtUnixMs`、配置中的 `innerNetwork.listenEndpoint`、稳定占位 `buildVersion = dev`、空 `capabilityTags` 与全零 `LoadSnapshot`。注册成功后写回 GM session 的心跳参数并启动定时器；注册失败只记录可诊断日志并保持进程存活，等待后续连接事件重新驱动。
5. `GateNode::HandleGmMessage()` 需要按“统一解包 -> envelope 校验 -> 按 `msgId` 分发”的方式组织处理：`Inner.NodeRegister` 交给 `HandleRegisterResponse()`，`Inner.NodeHeartbeat` 交给 `HandleHeartbeatResponse()`，`Inner.ClusterReadyNotify` 交给 `HandleClusterReadyNotify()`。其中心跳响应只接受带 `Response` 标记的成功包；带错误标记的 heartbeat packet 属于非法 envelope，应记录并忽略。
6. 心跳逻辑必须严格以后置注册成功为前提：只有在解析到注册成功响应后，`Gate` 才创建 repeating timer 并发送 `HeartbeatRequest`。每次成功心跳响应都允许刷新 GM session 中的 `heartbeatIntervalMs`、`heartbeatTimeoutMs` 与 `serverNowUnixMs`；如果参数发生变化，则重建定时器。当前不再存在“心跳失败后要求全量重注册”的专门分支，GM 断线或 session reset 只会取消定时器、清空 GM 会话态并同步重置 cluster ready 状态。
7. 生命周期与可观测性需要与现有骨架保持一致：`Gate` 在 `Uninit()` 前必须先取消心跳 timer，再释放 `ClientNetwork` 与 `InnerNetwork`；连接状态变化、注册发送/接受/拒绝、心跳发送/确认、cluster ready 应用、listener 收到不支持的 `Game` payload 等关键转移都要打出稳定日志字段。日志以 `nodeId`、`seq`、`clusterReady`、`gameNodeId` 等稳定标识为主，不再依赖运行时缓存的 endpoint 字符串。
8. 自动化验证围绕“Gate 成功完成 GM 注册心跳，且在协议噪音下保持可恢复状态”展开。测试至少应覆盖：`GM + Gate` 联动启动后，`Gate` 完成注册并收到一个以上成功心跳；`clusterReady` 通知能驱动客户端网络开关；带非法 envelope 的 GM heartbeat response 会被忽略而不是误更新状态；`Gate` 停止时可以干净释放 timer、listener、connector 与 client network。现有 `tests/cpp/node_gate_gm_session_tests.cpp` 已围绕这组行为建立覆盖。
