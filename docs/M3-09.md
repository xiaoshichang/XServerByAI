# M3-09 设计说明（Gate→GM 注册与心跳）

1. `M3-09` 的目标是在 `M3-08` 已完成的 Gate→GM active Inner 链路之上，补齐真正的启动期协议闭环：`Gate` 在链路可用后先向 `GM` 发送 `Inner.ProcessRegister (1000)`，注册成功后再进入 `Inner.ProcessHeartbeat (1100)` 周期心跳。当前条目只覆盖注册、心跳、重注册与相关日志/测试，不提前实现 `M3-12` 的 ownership 下发、`M3-13` 的 `Game -> Gate` 闭环，也不改变 `ClientNetwork` 的受控开放语义。
2. `GateNode` 需要持有一份最小但稳定的运行时会话上下文，至少包含：当前是否已注册、最近一次注册成功返回的 `heartbeatIntervalMs` / `heartbeatTimeoutMs`、启动时间戳 `startedAtUnixMs`、下一个待发送 `seq`、心跳定时器 `TimerID`、最近一次 `GM` 返回的 `serverNowUnixMs`、最近一次协议错误文本，以及本地要上报给 `GM` 的固定注册字段（`nodeId`、`pid`、`configuredInnerEndpoint`、`buildVersion`、`capabilityTags`、初始 `LoadSnapshot`）。这份状态应在断线、要求全量重注册或节点退出时被明确重置，避免旧通道状态泄漏到新会话。
3. Gate 侧需要新增统一的“Inner 协议收发 helper”，集中负责：构造 `RegisterRequest` / `HeartbeatRequest`、调用 `RegisterCodec` / `HeartbeatCodec` / `PacketCodec` 编码完整 packet、通过 active-mode `InnerNetwork::Send({}, payload)` 发往 `GM`，以及对 `GM` 返回的 packet 做包头 flags / `seq` / `msgId` 级别校验。这样后续 `M3-12` 的 ownership 编排、`M3-15` 的 `clusterReady` 消费都可以复用同一套 envelope 规则，而不是把 `PacketHeader` 细节散落在 `GateNode` 各处。
4. 注册流程建议绑定在连接状态机上，而不是只在 `OnRun()` 启动时发一次：当 Gate 发现 active connector 进入 `Connected` 且当前未注册时，立即发送一次完整注册请求，其中 `processType = Gate`、`nodeId = Gate<index>`、`pid = ServerNode::pid()`、`startedAtUnixMs` 固定为本次 Gate 进程启动时间、`innerNetworkEndpoint` 使用配置文件中的 `gate.<NodeID>.innerNetwork.listenEndpoint`、`buildVersion` 先使用稳定占位字符串、`capabilityTags` 当前可为空、`load` 先全量上报 `0`。注册成功后再切换到“已注册”状态；注册失败只记录可诊断日志并保持进程存活，等待当前链路上的下一次显式重注册触发或后续断线重连。
5. 心跳逻辑必须严格以后置注册成功为前提：只有在解析到 `Inner.ProcessRegister` 成功响应后，Gate 才创建 repeating timer，并按 `GM` 返回的 `heartbeatIntervalMs` 周期发送 `HeartbeatRequest`。每次成功心跳响应都允许刷新本地缓存的 interval/timeout；如果 `GM` 返回心跳错误且 `requireFullRegister = true`，Gate 应先取消现有心跳 timer、清空“已注册”态，再在当前仍存活的 connector 上重新走完整注册流程；如果 connector 已断开，则只保留“待连接后重注册”状态，不做忙等重试。
6. `GateNode::HandleInnerMessage()` 需要从现在的“统一记录后忽略 payload”升级为最小可扩展的响应分发器：识别 `msgId = 1000` 的注册响应和 `msgId = 1100` 的心跳响应，只接受带 `Response` 标记的 `GM` 返回包，并把 success / error payload 分别解码到 `RegisterSuccessResponse`、`RegisterErrorResponse`、`HeartbeatSuccessResponse`、`HeartbeatErrorResponse`。遇到 `seq = 0`、flags 非法、解码失败或未知 `msgId` 时，只记录结构化日志并丢弃消息，不让 Gate 因协议噪音或未来功能消息提前崩溃。
7. 生命周期与可观测性需要与现有骨架保持一致：Gate 在 `Uninit()` 前必须先取消心跳 timer，再释放 `ClientNetwork` 与 `InnerNetwork`；连接状态变化、注册发送/接受/拒绝、心跳发送/确认/拒绝、要求全量重注册等关键转移都要打出稳定日志字段（至少带 `nodeId`、`seq`、`gmInnerRemoteEndpoint`、`configuredInnerEndpoint`、必要时带错误码）。`M3-09` 不得顺手开放客户端入口，也不得把 `Gate` 自身 `innerNetwork` 监听真的打开成 passive listener。
8. 自动化验证围绕“Gate 成功完成注册心跳并可在异常响应下回到可恢复状态”展开。测试至少应覆盖：`GM + Gate` 联动启动后，`GM` 注册表内出现 `Gate0` 且 `routingId` / `innerNetworkEndpoint` 正确；等待一个以上心跳周期后，注册表中的 `lastHeartbeatAtUnixMs` 被刷新；`Gate` 停止后可以干净退出并清理 timer；对需要 `requireFullRegister = true` 的心跳错误场景，Gate 会取消旧心跳并重新发起完整注册，而不是继续沿用失效会话发心跳。
