# M3-17 顶层 managed 配置、占位 ServerStub 与 GM 反射 Stub 目录设计

本文档定义 `M3-17` 的实现范围：把当前散落在 `game.<NodeID>.managed` 下的 managed 配置提升到顶层共享块，在 `src/managed/GameLogic` 中补齐 3 个占位 `ServerStubEntity` 类型，并让 `GM` 通过受控的 managed 反射目录获取全部 Stub 列表，替换 `GmNode.cpp` 中的原生硬编码 catalog，为后续 `M5-14`、`M5-15` 与 `M6-02` 建立统一的 Stub 真值来源。

## 目标与边界

1. 本特性只解决“共享 managed 配置来源”“占位 Stub 类型声明”“GM 获取 Stub 目录”三件事，不实现真实匹配、聊天、排行榜业务逻辑。
2. `M3-14` / `M3-15` 已经落地的 ownership 与 ready 协议主流程保持不变；但当前 `entityId` 语义更新为“启动期由 `GM` 先下发占位值 `unknown`，`Game` 在本地创建实体后生成真实 GUID ID，并在 `ready` 上报中回填给 `GM`”，本特性仍不扩展多实例 Stub 配置。
3. 当前阶段允许 `GM` 为目录发现而加载 CLR，但只做一次性的 catalog 读取，不在 `GM` 内运行 `GameNativeInit` / `GameNativeOnMessage` / `GameNativeOnTick` 业务循环。
4. 配置收敛后不继续维护“每个 `Game` 节点独立 `managed.assemblyName`”的重复写法，避免 `GM` 与 `Game` 对程序集来源出现分叉。
5. 反射结果只收集直接或间接继承 `ServerStubEntity` 的非抽象公开类型，且输出顺序必须稳定，避免 ownership 快照与测试结果抖动。

## 实现清单

1. 调整配置模型到顶层 `managed`
   - 在 `ClusterConfig` 顶层新增共享 `ManagedConfig`，由 `GM` 与全部 `Game` 共用；`GameNodeConfig` 继续保留该块的选中结果，但来源改为顶层复制，而不是 `game.<NodeID>` 内嵌。
   - 顶层 `managed` 需要承载程序集标识，以及 `GM` 读取 Stub 目录所需的 runtime asset 路径约定（如 `assemblyPath` / `runtimeConfigPath`）；路径解析规则统一相对配置文件或工作目录，避免 `GM` / `Game` 各自猜测路径。
   - 同步更新 `configs/local-dev.json`、配置解析校验与长期文档，使示例与代码一致。

2. 在 managed 侧补齐 3 个占位 `ServerStubEntity`
   - 在 `src/managed/GameLogic` 下新增 `MatchService`、`ChatService`、`LeaderboardService` 三个占位类型，全部继承 `ServerStubEntity`，暂不承载真实业务状态。
   - 三个类型名就是当前 ownership 协议中的 `entityType` 真值来源；启动期 catalog / ownership 里分别对应 `MatchService/unknown`、`ChatService/unknown`、`LeaderboardService/unknown` 占位条目，待本地实例 ready 后再绑定真实实体 ID。
   - 占位 Stub 必须位于正式业务程序集，而不是测试项目或临时文件中，避免后续反射目录混入非业务类型。

3. 在 managed 程序集中提供稳定的 Stub 目录导出
   - 由 `GameLogic` 程序集内部通过反射扫描 `ServerStubEntity` 派生类型，过滤抽象类、泛型定义与非公开类型，并按稳定的类型名顺序输出目录。
   - 目录项至少包含 `entityType` 与默认 `entityId`；当前阶段每个 Stub 类型只生成一条 `unknown` 记录，不扩展多实例 Stub 配置。
   - 不再引入额外 attribute 做手工标记或手工排序；哪些类型属于 Stub 只由类型继承关系决定。
   - 每个 `ServerEntity` 在构造时都生成自己的 GUID `EntityID`；`ServerStubEntity` 的真实 ID 由 `Game` 在 ready 上报时带回给 `GM`。

4. 扩展 native managed-host / interop 边界，供 `GM` 拉取 Stub 目录
   - 在现有 `ManagedRuntimeHost` / `ManagedInterop` 基础上，为 catalog 读取补充最小导出绑定能力；`GM` 只需加载 runtime、校验 ABI、调用 catalog 导出，并把结果转成 native stub 列表。
   - 若当前 `GameNative*` ABI 不足以表达目录读取，可新增独立的 catalog 导出与对应 native 结构，但必须保持现有 `GameNativeInit` / `GameNativeOnMessage` / `GameNativeOnTick` 兼容，不破坏已有 host runtime tests。
   - `GM` 读取 catalog 失败时应明确报错并停止生成 ownership，避免悄悄回退到空目录或继续使用过期硬编码表。

5. 用单一 Stub 状态表替换 `GM` 内置硬编码 Stub catalog
   - 删除或收敛 `GmNode.cpp` 中的 `kBootstrapStubCatalog` 硬编码列表，改为以 managed catalog 作为 `ServerStubOwnershipSync` 的唯一来源。
   - `GM` 内部只保留一张 Stub 状态表，同时维护 `entityType`、`entityId`、owner 与当前状态；状态只区分 `Init` 与 `Ready`。
   - 状态表中的 `entityId` 默认值为 `unknown`；当 `GM` 收到 `Inner.GameServiceReadyReport (1203)` 时，使用 ready 条目携带的真实 ID 更新对应 Stub 记录，并把状态切换到 `Ready`。
   - ownership 分配策略、`assignmentEpoch = 1` 与外部协议保持现状不变；本特性只替换“有哪些 Stub”以及 `GM` 内部如何聚合 ready 状态的实现方式。

6. 补充验证与文档
   - native 测试优先覆盖：顶层 `managed` 配置解析、`GM` 在 catalog 读取成功时产出 3 条 ownership assignment、读取失败时不给出错误分配。
   - managed 测试优先覆盖：反射目录只返回 3 个占位 `ServerStubEntity`，顺序稳定，且不会把抽象基类 `ServerStubEntity` 或测试类型混入目录。
   - 同步更新 `docs/CONFIG_LOGGING.md`、`docs/PROJECT.md`、`docs/MANAGED_INTEROP.md`，说明顶层 managed 配置与 `GM` catalog 发现的新边界。

## 验证出口

1. 配置文件改为顶层 `managed` 后，`GM` 与 `Game` 都能从同一处拿到 managed 程序集信息，旧的 `game.<NodeID>.managed` 不再是事实来源。
2. `GameLogic` 程序集中能稳定发现 `MatchService`、`ChatService`、`LeaderboardService` 三个占位 Stub，且不会把抽象基类或测试类型混入目录。
3. `GM` 在生成 ownership snapshot 时，assignment 数量与反射目录一致，`entityId` 全部为 `unknown`；随后 `Game` 会在 ready 上报中携带真实 GUID ID，`GM` 再把这些 key 回填到单一 Stub 状态表并维护 `Init` / `Ready` 状态。
4. 当 managed catalog 无法加载、ABI 不兼容或导出缺失时，`GM` 会显式报错并拒绝继续使用伪造或过期的 Stub 目录。
