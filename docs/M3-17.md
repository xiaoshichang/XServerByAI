# M3-17 管理命令脚本设计

本文档定义 `M3-17` 的实现范围：提供一套开发期管理脚本，用于在单机环境下按既定顺序启动 `GM` / `Game` / `Gate` 节点，以及在需要时强制杀掉同一套配置对应的全部 `xserver-node` 进程。当前阶段只落地 Windows 可用路径，但核心调度逻辑使用 Python 编写，便于后续扩展到 Linux/macOS。

## 目标与边界

1. 本特性只解决开发期脚本层面的“批量启动”和“强制清理进程”，不扩展标准的集群优雅退出流程；`GM /shutdown` 已存在，但当前阶段不把它接入为正式关闭方案。
2. 启动脚本以单个 JSON 配置文件为输入，默认面向 `configs/local-dev.json` 这类单机多进程配置，启动顺序固定为 `GM -> Game[*] -> Gate[*]`。
3. 启动成功判定只做最小检查：`GM` 进程拉起后通过现有本地 HTTP `GET /healthz` 探活，全部节点启动后再通过 `GET /status` 观察注册进程数是否达到配置期望值；不解析节点标准输出、标准错误，也不额外构建日志转发。
4. 强杀能力只针对当前配置对应的本机 `xserver-node` 进程，不尝试处理跨机节点，也不保证进程内资源优雅释放。
5. 错误处理保持简单直接：脚本优先保证可用与可读性，只在关键失败点返回非零退出码和简短错误信息，不引入复杂重试/补偿框架。

## 实现清单

1. 约定 Python 主入口与 Windows 包装脚本
   - 在 `tools/` 下新增一个 Python 主脚本，统一承载 `start` / `kill` 两类命令，例如 `python tools/cluster_ctl.py start --config configs/local-dev.json`。
   - 当前阶段同时提供 Windows `.bat` 包装脚本 `tools/start_local_cluster.bat`、`tools/kill_local_cluster.bat`，负责填充仓库内默认路径与参数；批处理文件本身只做薄包装，核心逻辑仍全部落在 Python 中。
   - Python 脚本默认支持显式传入 `--config`、`--node-bin`，并在未指定 `--node-bin` 时按仓库内常见输出目录自动查找 `xserver-node.exe`。

2. 从单配置文件推导本次启动计划
   - Python 脚本读取集群配置后，基于顶层 `gm`、`game`、`gate` 三个配置块生成节点清单，明确 `GM`、全部 `GameNodeId`、全部 `GateNodeId` 的启动顺序。
   - `Game` 与 `Gate` 的启动顺序按 `NodeID` 做稳定排序，保证同一配置文件下多次执行行为一致，便于日志和测试对齐。
   - 启动命令统一使用现有入口 `xserver-node <configPath> <nodeId>`；脚本只负责组装命令与工作目录，不改动节点配置格式。

3. 实现最小启动编排与 GM 探活
   - `start` 命令先启动 `GM`，轮询 `gm.controlNetwork.listenEndpoint` 的 `GET /healthz`，确认 `GM` 控制面已就绪后，再依次启动全部 `Game` 与 `Gate`。
   - 全部子进程启动完成后，脚本继续轮询 `GET /status`，直到 `registeredProcessCount` 达到配置内期望远端节点数（全部 `Game + Gate`），作为“启动脚本执行完成”的最小判定。
   - 节点进程使用独立进程方式启动，不接管标准输出/标准错误；运行期日志仍由 `xserver-node` 自身日志系统落盘。
   - 若中途启动失败，脚本只做当前阶段需要的最小回收：调用同一脚本内的强杀逻辑清理已经拉起的同配置节点，避免单机残留孤儿进程。

4. 实现单机强杀当前配置对应的全部节点进程
   - `kill` 命令读取同一份配置，推导出本次需要清理的目标 `nodeId` 集合：`GM`、全部 `Game`、全部 `Gate`。
   - Windows 下通过查询本机进程命令行，筛选出 `xserver-node.exe` 且命令参数同时匹配“当前配置绝对路径”和目标 `nodeId` 的进程，再使用强制终止方式结束它们。
   - 为避免误杀，脚本不按进程名全局扫杀，只清理能与当前配置、当前节点集合对应上的进程实例。
   - 当前阶段不扩展 Linux/macOS 杀进程实现；Python 代码层面保留平台分支，便于后续在 `M6-08` 演进为跨平台运行脚本。

5. 补充文档与验证出口
   - 更新 `docs/PROJECT.md`、`docs/TEST_PLAN.md` 或其他长期文档中与开发期管理脚本相关的描述，使其与最终命令入口保持一致。
   - 为脚本增加最小自动化验证，优先覆盖“按顺序启动”“GM 探活成功后继续拉起”“按配置精确匹配并强杀目标进程”“启动失败时会清理已拉起节点”这几类行为。
   - 保持实现足够简单，让 `M3-16` 能直接复用这套脚本完成 `GM -> allNodesOnline -> Gate↔Game -> meshReady -> ownership -> GameReady -> clusterReady` 的闭环集成验证。

## 验证出口

1. 在 Windows 本机执行启动命令后，`GM` 会先就绪，随后 `Game` / `Gate` 依次拉起，`GM /status` 中的 `registeredProcessCount` 最终达到配置内全部 `Game + Gate` 的数量。
2. 脚本不会接管节点标准输出/标准错误；节点日志继续写入项目既有日志目录。
3. 对同一配置执行强杀命令时，只会终止命令行匹配当前配置路径与目标 `nodeId` 的 `xserver-node` 进程，不会误杀其他配置实例。
4. 启动过程中任一关键步骤失败时，脚本能回收本次已启动的同配置节点，避免残留半启动状态影响下一次联调。
