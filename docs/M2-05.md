# M2-05 设计说明（主线程事件循环骨架）

1. 在 `src/native/core` 中新增 `MainEventLoop` 模块，定位为位于 `CoreLoopExecutor` 之上的主线程事件循环骨架：继续复用单线程 `asio::io_context`、现有线程命名与 work guard 约束，但把后续进程主入口真正需要的运行生命周期接口提前收敛为统一对象。
2. `MainEventLoop` 自身拥有一个 `CoreLoopExecutor` 和一个绑定同一 `io_context` 的 `TimerManager`；对外暴露 `Run()`、`RequestStop()`、`executor()`、`context()` 与 `timers()`，使 `GM` / `Gate` / `Game` 后续条目既能直接投递异步任务，也能基于统一定时器能力挂接心跳、超时与 Tick。
3. 当前条目明确采用严格单线程模型：`CoreLoopExecutor` 与 `MainEventLoop` 只允许 owner thread 使用，不承诺跨线程调用 `Start()` / `Stop()` / `Run()` / `RequestStop()` 的安全性，也不为未来可能的多线程停止请求预埋锁或跨线程同步层。
4. 生命周期采用显式 hooks：`on_start` 在事件循环线程真正进入 `run()` 后执行，适合完成启动期资源挂接；`on_stop` 在循环退出后执行，适合回收同一主线程上的轻量状态；当前阶段不引入额外工作线程、线程池或更复杂的调度框架。
5. 事件循环骨架额外支持可选固定 Tick：`MainEventLoopOptions` 提供 `tick_interval`，只有在同时提供 `on_tick` 回调时才允许为正值；Tick 回调携带当前 `steady_clock` 时间点、与上次 Tick 的 `delta` 以及递增的 `tick_count`，为后续 `Game` 进程驱动实体 Tick / managed Tick 提供基础时序数据。
6. 所有停止语义统一走 `RequestStop()`：它只负责当前 owner thread 上把主循环标记为 stopping 并请求底层执行器退出；`Run()` 收尾阶段再统一取消 `TimerManager` 中遗留的定时器，确保循环结束后不会残留悬挂的异步回调。
7. 启动失败、Tick 回调异常与停止回调异常都应转换为清晰的字符串错误返回；对于 `on_start` 返回 `false`、Tick 定时器创建失败、`on_tick` 抛出异常等情况，`Run()` 必须结束循环并把失败原因回传给调用方，而不是静默吞掉。
8. 当前阶段不把网络监听、ZeroMQ/KCP 句柄或业务对象直接塞进事件循环骨架；这些资源仍由后续 `M3`/`M4` 条目各自拥有，但其生命周期驱动必须依赖本条目提供的 `Run + RequestStop + timers` 基线，而不是各自拼装独立循环。
9. 自动化验证继续采用最小 native 测试可执行程序，至少覆盖：启动/停止 hooks 在主线程上执行、启动失败错误透传、Tick 回调与 `tick_count/delta` 生效、预先挂入 `TimerManager` 的定时器可以随事件循环一起驱动，以及无效 Tick 配置被显式拒绝。
