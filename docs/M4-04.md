# M4-04 Gate HTTP 鉴权与会话入口设计

本文档定义 `M4-04` 的实现范围：把当前“客户端先直连 Gate 的 KCP 入口、再发送占位登录包”的流程，调整为“客户端先通过 Gate 对外 HTTP 端口提交账号密码登录，登录成功后获取该 Gate 的 KCP 端口与分配好的 `conversation`，再用这个 `conversation` 建立后续长连接会话”。当前阶段重点是把 Gate 的 HTTP 鉴权入口、`conversation` 分配与保留、KCP 会话准入以及模拟客户端的新登录流程串起来；不会在本里程碑内补齐 `sessionId -> playerId` 权威会话表、`Gate -> Game` 业务路由或服务端 Avatar 权威创建。

## 目标与边界

1. `Gate` 需要新增一个对外 HTTP 鉴权端口，配置方式与现有 `clientNetwork.listenEndpoint` 保持同级，开放时机也与 KCP 客户端入口一致：只在收到 `clusterReady = true` 后真正开始对外监听。
2. 客户端登录入口改为 HTTP 账号密码登录，请求成功后返回“目标 Gate 的 KCP 地址 + 分配好的 `conversation`”；后续 KCP 长连接不再依赖手工指定 `conversation` 作为默认主路径。
3. `M4-04` 只负责“登录成功后发放可用的 KCP 会话入口”，不在本里程碑内落地 `M4-05` 的 `sessionId -> playerId` 持久会话映射，因此新的登录结果只能作为 KCP 准入凭据，不能提前扩展成完整业务会话表。
4. 当前阶段不接入真实账号系统、数据库或 TLS；鉴权规则维持在本地开发可用的最小语义，例如账号密码字段必填、失败返回明确 HTTP 错误码、成功分配临时 `conversation`。
5. 现有 `M4-03` 中“`login` 通过 KCP 发包并可本地套用 Avatar 成功态”的临时行为需要让位于真正的 HTTP 鉴权流程；Avatar 本地占位与业务命令演示不再与登录成功强绑定。

## 实现清单

1. 扩展 Gate 配置与节点生命周期
   - 在 `gate.<NodeID>` 下新增 `authNetwork.listenEndpoint` 配置块，和现有 `innerNetwork`、`clientNetwork` 并列，继续沿用 `listenEndpoint.host/port` 结构。
   - 扩展 `ClusterConfig` / `GateConfig` / `GateNodeConfig` 以及对应 JSON 解析逻辑、节点配置选择逻辑与样例配置，使 Gate 可以同时持有 KCP 对外端口和 HTTP 鉴权端口。
   - `GateNode` 在 `clusterReady = true` 时同时打开 `authNetwork` 与 `clientNetwork`；本里程碑继续沿用 `M4-02` 的边界，不额外处理运行期 `clusterReady` 回滚，只保证正常启动与正常退出路径。

2. 为 Gate 增加最小 HTTP 鉴权服务
   - 参考现有 `GmControlHttpService` 的实现风格，在 `src/native/node/` 下新增 Gate 侧 HTTP 服务，至少支持 `POST /login`，并可补一个最小 `GET /healthz` 便于联调观察。
   - `POST /login` 请求体使用 JSON，字段至少包含 `account` 与 `password`；非 JSON、缺字段、空字符串等输入要返回稳定的 `400` 错误，鉴权失败返回明确的失败响应。
   - 成功响应返回 Gate 节点标识、可拨号的 KCP 主机端口以及新分配的 `conversation`；当配置监听的是 `0.0.0.0` / `::` 这类通配地址时，响应中应返回客户端可实际拨号的规范化地址，而不是直接回传原始绑定地址。

3. 引入 `conversation` 保留表与临时鉴权结果
   - Gate 在 HTTP 登录成功后分配一个本地唯一、非零的 `conversation`，并写入“待建链路”的保留表，例如记录 `conversation`、账号标识、发放时间、过期时间和来源 IP。
   - 因为后续 KCP 使用的是 UDP，远端端口与 HTTP 请求端口不会一致，所以保留表只绑定来源 IP，不绑定来源端口；这样客户端可在同一台机器上先走 HTTP、再走 UDP 建链。
   - 该保留表只服务于 `M4-04` 的鉴权准入与 `conversation` 发放，不提前承担 `playerId`、`sessionId` 或路由信息的长期权威职责；这些语义继续留给 `M4-05` 及后续条目扩展。

4. 把 KCP 新会话创建收紧到“已鉴权 `conversation`”
   - 当前 `ClientNetwork` 会在收到首个 UDP/KCP 数据报后直接创建 `ClientSession`；`M4-04` 需要在这条路径上增加准入检查，只有命中有效保留表的 `conversation` 才允许新建会话。
   - 准入检查至少要同时校验 `conversation` 是否存在、是否未过期、来源 IP 是否与 HTTP 登录来源一致；未命中的新首包应被丢弃并记日志，避免任意客户端绕过 HTTP 自带 `conv` 直接入站。
   - 当首个合法 KCP 数据报命中保留表并成功建会话后，应消费这条保留记录；新建 `ClientSession` 可以转入更贴近真实语义的状态，例如直接视为“已通过登录准入”，但在 `M4-05` 之前仍不补 `playerId` 权威绑定。

5. 调整模拟客户端为“先 HTTP 登录，再用返回结果连 KCP”
   - `XServer.Client` 的 `login` 命令从“要求先 connect、再通过 KCP 发登录包”改为“直接通过 HTTP 提交账号密码登录”，默认参数从现有 `player/token` 切换为 `account/password`。
   - HTTP 登录成功后，客户端把返回的 Gate KCP 连接资料缓存到本地状态中；后续 `connect` 命令默认使用这份最近一次登录结果的主机、端口与 `conversation` 建立 KCP 长连接，也允许保留显式覆盖参数用于调试。
   - 现有 `move` / `buyWeapon` 等演示命令不能再依赖 `login localSuccess=true` 偷前置 Avatar；若仍需本地占位能力，应在实现时明确拆成独立的本地调试入口，而不是继续混在正式登录流程里。

6. 补齐测试与相关文档同步
   - Native 侧至少需要覆盖：配置解析新增 `authNetwork`、Gate 鉴权 HTTP 服务基本请求响应、`conversation` 保留表过期/命中/来源 IP 校验，以及未鉴权 `conversation` 不能触发 KCP 会话创建。
   - Client 侧至少需要覆盖：HTTP 登录请求构造、登录结果解析、客户端状态缓存、`connect` 默认复用最新登录结果，以及错误输入或鉴权失败时的可读提示。
   - 相关长期文档至少同步 `docs/CONFIG_LOGGING.md` 与 `docs/STARTUP_FLOW.md`，前者补 `gate.<NodeID>.authNetwork.listenEndpoint` 约定，后者明确 `Gate` 在 `clusterReady = true` 后同时开放 HTTP 鉴权入口与 KCP 客户端入口。

## 验证出口

1. `Gate` 只有在收到 `clusterReady = true` 后才会对外打开 HTTP 鉴权端口和 KCP 客户端端口；在此之前客户端无法通过 HTTP 登录拿到可用 `conversation`。
2. 客户端使用账号密码经 HTTP 登录成功后，能够拿到明确的 Gate KCP 连接信息和非零 `conversation`，并把这份结果保存为后续建链默认值。
3. 未经 HTTP 发放的 `conversation`，或者来源 IP 与登录来源不一致的 `conversation`，不能触发新的 `ClientSession` 创建。
4. 命中有效保留表的客户端首个 KCP 数据报可以成功创建会话并继续后续 KCP 收发，表明“HTTP 登录 -> KCP 建链”主链路闭环。
5. `M4-04` 不要求在本里程碑内验证 `sessionId -> playerId` 持久映射、`Gate -> Game` 业务转发、服务端 Avatar 权威创建或断线重连。
