# M4-03 控制台模拟客户端设计

本文档定义 `M4-03` 的实现范围：在仓库顶层新增独立的 `client/` 目录，使用 C# 落地一个简单但可扩展的控制台模拟客户端，用于连接 `Gate` 已开放的 KCP 入口、发送和接收基础协议包，并通过交互命令或脚本模拟客户端行为。当前阶段优先建立“可运行的模拟器骨架”和“可复用的行为驱动框架”，为后续 `M4-04` 登录鉴权、`M4-06` / `M4-07` 请求响应链路以及 Avatar/业务消息验证提供稳定载体。

## 目标与边界

1. `M4-03` 必须把模拟客户端作为独立的 C# 控制台程序放在 `client/` 目录，而不是混入 `src/managed/` 的服务端业务项目。
2. 当前阶段只要求模拟客户端能够建立到 `Gate` 的 UDP/KCP 通道、编解码基础包头、收发消息、打印日志，并提供后续行为扩展点；不要求在本里程碑内完成正式登录鉴权结果闭环。
3. 服务端 `Avatar` 的权威创建依赖后续 `Game` 侧业务链路与实体运行时，不应在 `M4-03` 中抢先实现；当前阶段只需要为“登录成功后本地展示 Avatar 视图 / 状态”预留客户端侧状态槽位。
4. “移动”“购买武器”等业务发包在本里程碑内可以作为演示命令或脚本动作存在，但默认属于测试 / 占位语义；如尚未有正式协议定义，不在 `M4-03` 中冻结正式业务 `msgId`。
5. `M4-03` 应尽量避免把协议、状态机和控制台交互逻辑耦死在 `Program.cs` 中，保证后续新增登录、推送、断线重连或自动化脚本时不需要重写主体结构。

## 实现清单

1. 建立 `client/` 目录下的独立项目结构
   - 新增独立控制台项目，例如 `client/XServer.Client/XServer.Client.csproj`，目标框架与仓库现有 .NET 项目保持一致。
   - 如需要单元测试，则在 `client/Tests/` 下增加独立测试项目，而不是挤进 `src/managed/Tests/`。
   - 当前阶段优先使用 `dotnet run --project ...` / `dotnet test ...` 直接驱动，不强制把客户端项目并入现有 `XServerByAI.Managed.sln`。

2. 在客户端侧建立最小 KCP 传输与包头编解码能力
   - 使用 C# 封装 UDP socket、KCP 会话驱动、发送队列、接收循环与关闭路径，使模拟器能够稳定与 `Gate` 的 `clientNetwork.listenEndpoint` 通信。
   - 落地与 [PACKET_HEADER](./PACKET_HEADER.md) 对齐的固定头编解码工具，至少支持 `magic`、`version`、`flags`、`length`、`msgId`、`seq` 的构造与解析。
   - 当前阶段不追求完整高性能实现，但需要把 transport、packet codec 和上层命令逻辑分层，避免后续修改业务消息时牵连底层收发代码。

3. 定义控制台命令面与本地客户端状态模型
   - 提供基础命令，例如连接、断开、发送测试消息、查看当前状态、加载脚本、退出等，让模拟器在开发期可以手动驱动。
   - 建立稳定的本地状态对象，至少区分“未连接 / 已连接 / 登录中 / 已登录 / Avatar 已就绪”等语义槽位；其中“已登录”“Avatar 已就绪”在当前阶段允许仍由占位结果驱动。
   - 命令处理层应能根据本地状态决定动作是否可执行，避免在后续补登录与业务链路时把状态检查散落到各处。

4. 为登录、Avatar 与业务消息预留可复用动作抽象
   - 把“登录请求”“移动请求”“购买武器请求”设计为可单独注册的客户端动作，而不是硬编码在一个巨大 `switch` 中。
   - 登录动作在 `M4-03` 中只负责组织请求、发送报文、打印响应或超时结果，为 `M4-04` 的正式鉴权语义预留挂点。
   - Avatar 相关逻辑在 `M4-03` 中只保留客户端本地视图模型与更新接口，不假定服务端已经具备权威 Avatar 创建能力。
   - 业务动作如“移动”“购买武器”优先走“可配置 `msgId` + 可配置 payload”的演示模型；待正式协议落地后再绑定稳定业务消息编号。

5. 增加脚本化场景驱动能力
   - 在手工 REPL 之外，支持一个最小脚本或场景文件格式，使模拟器可以按顺序执行 connect、login、move、buyWeapon 等动作，便于回归测试和联调复现。
   - 脚本执行结果应输出清晰的步骤日志、请求关联序号、响应摘要和失败原因，方便定位 Gate/KCP/协议问题。
   - 如某个动作当前仍是占位语义，应明确打印“仅本地模拟”或“已发包但服务端尚未实现”的状态，避免误判为功能完成。

6. 补齐单元测试与长期文档同步
   - 为包头编解码、命令解析、本地状态迁移、脚本解析与序号分配补齐客户端单测；网络环路可优先通过可替换抽象做最小验证，不强求在本里程碑内覆盖完整端到端。
   - 因为仓库会新增顶层 `client/` 目录，实施时需要同步更新 `docs/PROJECT.md` 与必要的目录规范说明，明确客户端代码与服务端 managed 代码的职责边界。
   - 如果为调试临时引入测试号段消息，必须遵守 [MSG_ID](./MSG_ID.md) 中的测试 / 联调号段约束，避免误占正式业务编号。

## 验证出口

1. 仓库中存在一个可以直接通过 `dotnet run --project ...` 启动的控制台模拟客户端，且代码位于独立的 `client/` 目录。
2. 模拟客户端能够连到已开放客户端入口的 `Gate`，完成最小 KCP 收发与固定包头编解码，并输出可读日志。
3. 模拟客户端既支持手工命令驱动，也支持最小脚本驱动，以便后续复用到登录、业务发包和回归验证。
4. 登录、Avatar、本地业务动作在代码结构上已经有清晰扩展点，但 `M4-03` 不把正式鉴权成功闭环、服务端 Avatar 权威创建或完整业务协议抢跑到本里程碑内。
