# M3-07 设计说明（GM HTTP 管理接口）

1. 本条目改为落地本地 HTTP 管理接口，不再使用本地 CLI。接口语义仍归属 `Control` 命名边界，因此实现、配置字段与后续文档统一使用 `ControlNetwork` / `controlNetwork` 命名，而不是复用 `Inner` 或 `Client` 术语。
2. `GM` 配置模型需要补充 `gm.controlNetwork.listenEndpoint`，用于声明 HTTP 管理接口监听地址。当前阶段默认要求绑定本机回环地址进行本地调试，不暴露远端公网管理能力；样例配置与长期维护文档在实现时同步更新。
3. 在 `src/native/node/` 内新增独立的 HTTP 管理服务，建议命名为 `GmControlHttpService` 或等价职责对象。该模块负责 TCP 监听、最小 HTTP/1.1 请求解析、路由分发、响应编码与生命周期回收；`GmNode` 继续只负责装配 `InnerNetwork`、`GmInnerService` 与该 HTTP 服务，不把协议细节直接塞进节点主类。
4. 整个 HTTP 服务必须复用 `GM` 现有 `MainEventLoop` 的同一个 `io_context`。监听、accept、读写、超时和请求处理都通过 `event_loop().context()` / `event_loop().executor()` 驱动，不新增专用读取线程或第二个事件循环；这样可以保持 owner-thread 语义一致，并避免后续停机时跨线程协调。
5. `M3-07` 先固定最小 HTTP 路由面：`GET /healthz`、`GET /status`、`POST /shutdown`。`/healthz` 用于最小存活探测；`/status` 返回 GM 当前节点标识、Inner 监听地址、Control 监听地址、已注册节点数量与基础运行状态；`/shutdown` 触发 `ServerNode::RequestStop()`，用于验证“HTTP 管理入口 -> GM 主循环停止”的闭环。更细的在线进程查询与强制下线能力继续保留到 `M3-17`。
6. 路由调度层需要抽象为稳定的请求/响应模型，而不是让 socket 回调直接拼接字符串。建议至少固定 `method`、`path`、`statusCode`、`contentType`、`body` 与 `requestStop` 这组结果语义，使后续 `M3-17` 新增进程查询 / 强制下线路由时，只扩展 handler 与状态访问，不重做连接管理和 HTTP 封装。
7. 生命周期上，HTTP 管理服务只在 `GM` 角色启用，并与 `GmNode` 同步初始化、运行和回收。`OnInit()` 负责读取 `controlNetwork` 配置并创建监听对象，`OnRun()` 启动 accept 链路，`OnUninit()` 必须取消未完成 accept、关闭活动连接并确保 `RequestStop()` 触发后的收尾不会破坏既有 `InnerNetwork` 与 `GmInnerService` 的退出顺序。
8. 自动化验证围绕“HTTP 接口建立且与同一事件循环协作正确”展开。测试至少覆盖：`gm.controlNetwork.listenEndpoint` 配置校验、`GET /healthz` 成功、`GET /status` 返回运行时快照、`POST /shutdown` 触发 GM 正常退出，以及非 `GM` 节点不会意外启用该接口。这样 `M3-07` 先把管理入口、配置和主线程模型做稳，`M3-17` 再在同一骨架上补齐更具体的运维路由。
