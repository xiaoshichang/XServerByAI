# M3-07 设计说明（ClusterConfig 收敛与网络命名规范）

1. 当前条目以配置模型收敛为目标，统一 native 配置入口只暴露两类接口：加载完整 `ClusterConfig`，以及基于 `ClusterConfig + nodeId` 选择具体 `NodeConfig`。调用方不再直接拼装零散节点配置对象。
2. `NodeConfig` 拆分为 `GmNodeConfig`、`GateNodeConfig`、`GameNodeConfig` 三个派生类型。节点角色由选中的派生类型表达，不再在 `NodeConfig` 内重复保存历史的进程类型字段与节点标识字段。
3. `logging` 与 `kcp` 上移到集群层，分别成为 `ClusterConfig.logging` 与 `ClusterConfig.kcp`。节点级配置只保留该节点真正独有的字段，避免 `NodeConfig` 与集群公共配置重复建模。
4. 实例选择统一使用 `NodeID` 语义：`GM`、`Gate<index>`、`Game<index>`。`gate` / `game` 的子键直接使用 `NodeID`，移除 `selector` 概念，也不再在实例块内部重复声明 `nodeId`。
5. 网络命名统一按职责分层：节点与节点之间为 `Inner`，ctrl 工具与 `GM` 之间为 `Control`，`Gate` 与客户端之间为 `Client`。后续配置字段、函数、类型和文档都必须遵守这一命名边界。
6. 结合当前职责，`gm` 只要求 `innerNetwork.listenEndpoint`；`gate.<NodeID>` 同时要求 `innerNetwork.listenEndpoint` 与 `clientNetwork.listenEndpoint`；`game.<NodeID>` 要求 `innerNetwork.listenEndpoint`，并允许可选 `managed.assemblyName`。
7. `GateNode` 的占位运行时不再只验证 schema，而是实际消费 `clientNetwork.listenEndpoint` 与集群级 `kcp`，把这组配置接入 `ClientNetwork` 占位骨架，为后续 `M4-02` 的真实客户端监听实现保留稳定边界。
8. 仓库内必须提供一份可工作的示例配置，当前固定为 `configs/local-dev.json`。测试夹具与文档示例都应复用同一套键名和网络命名，避免出现历史术语回流。
