# M2-09 设计说明（统一消息封包与解析逻辑）

1. 在 `src/native/net` 内新增独立的 `PacketCodec` 模块，统一承载内部节点消息的封包与解包逻辑；当前阶段只覆盖固定 `PacketHeader(20 bytes) + payload` 模型，不提前引入长度前缀、扩展头或压缩实现。
2. 编码侧继续复用 `M2-06` 的 `BinaryWriter`，对外暴露固定包头构造、header 校验、总线包长度计算与整包写入接口；调用方必须显式提供目标缓冲区，避免在协议基础层隐藏堆分配策略。除 `MakePacketHeader` 这类纯构造辅助外，所有可失败接口统一直接返回 `PacketCodecErrorCode`，其中 `0`/`None` 表示成功。
3. 解码侧对外暴露零拷贝 `PacketView`：解析结果包含一份按主机字节序还原后的 `PacketHeader` 副本，以及直接指向输入缓冲区的 `payload span`；后续 `M2-10`/`M2-11`/`M2-12` 可在此基础上继续解码消息体，而不重复复制 payload。输出对象只通过输出参数承载结果，不再额外使用 `bool + error_out` 双返回通道。
4. 包头固定字段校验集中在同一处完成：`magic` 必须等于 `0x47535052`，`version` 必须等于 `1`，`flags` 只能使用 `PacketDefinedFlagMask` 中已定义位；调用方不得在各处手写重复校验逻辑。
5. 整包解析必须要求“输入字节数恰好等于 `PacketHeaderSize + PacketHeader.length`”：包头不足、包体截断或尾部残留都视为失败，避免 ZeroMQ message 边界内混入半包或多包时被静默接受。
6. 错误反馈采用独立的 `PacketCodecErrorCode` + `PacketCodecErrorMessage()`，覆盖参数为空、缓冲区不足、payload 长度溢出、`magic/version/flags` 非法与 header/body 长度不匹配等场景；底层 `BinaryWriter` / `BinaryReader` 的序列化错误在模块内部统一映射，避免上层协议处理同时理解两套错误体系。调用方约定统一为“成功返回 `PacketCodecErrorCode::None`，失败返回具体含义的非零错误码”。
7. 自动化验证采用独立 native 测试可执行程序，至少覆盖：固定 header 构造与精确线缆字节布局、整包 roundtrip、零拷贝 payload 视图、保留 flag 非法、截断包体、尾部多余字节、编码缓冲区不足与 payload 长度越界等失败路径。
