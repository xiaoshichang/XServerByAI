# M3-13 Gate 被动 Inner 接入与 Game→Gate 注册/心跳闭环设计

本文档定义 `M3-13` 的实现范围：在 `Game` 收到 `GM` 下发的 `Inner.ClusterNodesOnlineNotify (1204)` 且 `allNodesOnline = true` 后，再由 `Game` 主动向配置中的全部 `Gate` 发起 `Inner.NodeRegister (1000)` / `Inner.NodeHeartbeat (1100)`；`Gate` 作为被动 `Inner` listener 接收这些请求、返回响应并维护本地 `Game` 会话目录。目标是形成当前服务器组内 `Game × Gate` 的二部图全连接，其中“全连接”只指每个 `Game` 都与每个 `Gate` 直连，不引入 `Gate↔Gate` 或 `Game↔Game` 直连。

## 目标与边界

1. 本特性只解决 `Game -> Gate` 的注册与心跳闭环、`Gate` 的被动接入，以及 `Game` 在启动期形成对全部目标 `Gate` 的直连事实，不在本里程碑内实现 `Inner.GameGateMeshReadyReport (1205)` 上报、ownership 下发、`clusterReady` 聚合或业务转发消息。
2. `Game` 启动后仍需立即完成 `Game -> GM` 注册与心跳，但不得在收到 `allNodesOnline = true` 前就主动向任意 `Gate` 发起连接或注册；`GM -> Game` 的该通知是 `Game -> Gate` 全连接的唯一启动信号。
3. `Gate` 必须始终保持被动 `Inner` 监听能力，允许在启动期先于 `Game` 建好 listener，但它只消费来自配置内期望 `Game` 的注册/心跳事实，不得自行推断 mesh 已完成。
4. 本条目当前优先覆盖主闭环 happy path 与基础协议校验，不引入“会话失效回滚、心跳超时扫描、结论撤销后自动重建”等增强鲁棒性处理；这些异常场景留待后续专门条目统一补强。

## 实现清单

1. 调整 `Game -> Gate` connector 的启动时机
   - `GameNode` 继续在初始化阶段收集全部目标 `Gate` 配置，但 `OnRun()` 只启动 `GM` 所需链路，不再在进程刚启动时就同时拉起全部 `Gate` connector。
   - `InnerNetwork` 需要补齐“已配置 connector 的延迟启动”能力，保证 `Game` 可以在收到 `allNodesOnline = true` 后再统一启动全部 `Gate` connector。
   - 重复收到 `allNodesOnline = true` 时不得重复启动已经运行中的 `Gate` connector；正常反初始化时仍按现有生命周期回收资源。

2. 在 `Gate` 侧建立期望 `Game` 会话骨架与被动接入入口
   - `GateNode` 在初始化阶段除 `GM` 会话外，还要根据集群配置为全部期望 `Game` 预登记 `InnerNetworkSession`，让后续 `routing_id -> GameNodeId` 的被动注册、状态查询和目录维护都有稳定落点。
   - `Gate` listener 收到来自 `Game` 的请求时，应把 `routing_id` 与声明的 `nodeId` 绑定到对应期望会话；未在配置中的 `Game`、错误的 `processType`、无效 endpoint 或协议封套错误都必须按现有 inner 错误语义拒绝。
   - `Gate` 本地目录的真值来源仍然是“当前 listener 上实际建立的 `Game` 注册事实”，不是 `GM` 转抄的目录快照。

3. 补齐 `Game -> Gate` 的注册请求/响应闭环
   - `GameNode` 对每个 `Gate` 连接建立后，仅在“允许启动 Gate mesh”且该 `Gate` 会话尚未注册时发送 `Inner.NodeRegister`，请求体沿用现有 `RegisterCodec`，声明自身 `Game` 身份与 `innerNetworkEndpoint`。
   - `GateNode::HandleGameMessage` 需要从“仅记录 unsupported packet”扩展为真实解析 `PacketHeader`、识别 `Inner.NodeRegister`、校验 envelope/payload，并返回成功或失败响应。
   - `Gate` 在接受注册后要写回该 `Game` 会话的 `routing_id`、`pid`、`startedAtUnixMs`、`buildVersion`、`capabilityTags`、`load`、最近心跳时间与 `registered` 状态；重复注册时先只要求同一路由幂等刷新，不在本条目中扩展更复杂的失效替换策略。

4. 补齐 `Game -> Gate` 的心跳请求/响应闭环
   - `GameNode` 需要把现有面向 `GM` 的注册/心跳状态机推广到每个 `Gate` 会话：每个 `Gate` 独立维护 `registerSeq`、`heartbeatSeq`、`heartbeatTimerId`、最近 `serverNowUnixMs` 与协议错误信息。
   - `GateNode` 需要在 listener 路径上识别 `Inner.NodeHeartbeat`，只接受已知且已注册的 `Game` 会话；成功后刷新 `lastHeartbeatAtUnixMs`、`load`、`innerNetworkReady` 等状态，并返回心跳成功响应。
   - `Game` 对 `Gate` 心跳响应的 envelope 校验、陈旧 `seq` 忽略、解码失败日志与配置变更后重建 timer 的行为，应与当前 `Game -> GM` 实现保持一致，避免形成两套语义。

5. 回归测试与验证补强
   - 为 `Game` 增加“收到 `allNodesOnline = true` 前不连接 Gate、收到后才开始连接全部 Gate”的测试，至少覆盖 `1 Game × 2 Gate`，确保不是只连上第一个 `Gate` 就提前成功。
   - 为 `Gate` 增加真实 listener 场景测试，覆盖“接受期望 `Game` 的注册与心跳”以及“拒绝未知 `Game` / 非法 envelope”这类基础协议行为。
   - 为 `Game` 增加 `Gate` 响应消费测试，覆盖“注册成功后进入心跳”与“重复 `allNodesOnline = true` 通知幂等”这两个主路径行为。
   - 若现有长期文档中的启动顺序或状态边界与实现细节发生偏差，需要同步更新 `docs/PROCESS_INNER.md`、`docs/STARTUP_FLOW.md` 与相关说明，保持“`Game` 在 `allNodesOnline` 之后启动 Gate 全连接”的单一语义。

## 验证出口

1. `Game` 在仅完成 `Game -> GM` 注册/心跳、但尚未收到 `allNodesOnline = true` 时，不得主动连接任意 `Gate`，也不得向 `Gate` 发送注册请求。
2. 一旦 `GM` 下发 `allNodesOnline = true`，每个 `Game` 都会主动向每个目标 `Gate` 建立 `Inner` 连接，并完成注册与周期心跳；`Gate` 本地可观察到对应 `Game` 会话已注册。
3. 在 `1 Game × N Gate` 或 `M Game × N Gate` 配置下，只要存在任意一个目标 `Gate` 尚未完成注册闭环，就不能把该 `Game` 视为“已完成 Gate 全连接”。
4. 当前条目的验收不要求覆盖链路断开、心跳超时、`allNodesOnline` 结论撤销后的自动回滚；这些鲁棒性场景留待后续专项实现与验证。
