# M3-14 Game→GM mesh ready、GM ownership 下发与 Game 本地 Stub 初始化挂点设计

本文档定义 `M3-14` 的实现范围：在 `M3-12` 的 `allNodesOnline` 与 `M3-13` 的 `Game -> Gate` 注册 / 心跳闭环之上，补齐 `Game -> GM` 的 `Inner.GameGateMeshReadyReport (1205)`、`GM -> Game` 的 `Inner.ServerStubOwnershipSync (1202)`，并在 `Game` 侧落地“按 ownership 驱动本地 Stub 初始化”的启动期挂点。

## 目标与边界

1. 本特性只解决 mesh ready 上报、GM 聚合与 ownership 下发，以及 `Game` 消费 ownership 后的本地初始化挂点；`Inner.GameServiceReadyReport (1203)`、`clusterReady` 聚合与 `Gate` 对客户端入口的开放仍留给 `M3-15`。
2. `meshReady = true` 的真值来源必须是“当前 `Game` 已对全部目标 `Gate` 完成注册且心跳链路可用”，而不是仅凭 connector 已连接、`allNodesOnline = true` 或局部 `Gate` 成功就提前推断。
3. `GM` 只有在当前 `allNodesOnline = true` 且全部期望 `Game` 都报告 `meshReady = true` 后，才能广播全量 ownership 快照；当前实现中的 bootstrap 分配表只构建一次，因此 `1202.assignmentEpoch` 固定为首轮值 `1`，旧轮次结果不得与当前结论混用。
4. 当前仓库尚未引入独立的 Stub 目录 / 配置源，也没有真实 `ServerStubEntity` 运行时，因此本里程碑只提供“ownership 协议 + 本地挂点 + 可观测状态”，不直接实例化 C# Stub；本地挂点先以可测试的 native 状态与日志落地，为 `M6-02` 的 managed 初始化留出稳定接口。
5. 为了让 ownership 闭环具备非空验证对象，本里程碑默认在 `GM` 内维护一个最小 bootstrap Stub 目录，先使用文档中已经出现的示例服务（建议 `MatchService/default`、`ChatService/default`）；当前实现只在首次满足条件时按最简单的随机策略生成一次 bootstrap 分配表，后续若引入真实配置、注册表或负载预估策略，可扩展不同分配策略，同时保持 `1202` 的全量表语义。

## 实现清单

1. 补齐 `1205` / `1202` 消息编解码与协议校验
   - 在现有启动编排消息模块上新增 `Inner.GameGateMeshReadyReport` 与 `Inner.ServerStubOwnershipSync` 的消息体结构、`msgId` 常量、编解码入口与错误消息。
   - `1205` 采用固定长度结构：`meshReady`、`statusFlags`、`reportedAtUnixMs`；`statusFlags` 当前必须为 `0`。该消息只表达“当前 `Game` 对全部目标 `Gate` 的全连接是否已经完成”，不携带额外业务元信息。
   - `1202` 采用“头部 + 变长 assignment 列表”结构，单条 `ServerStubOwnershipEntry` 至少包含 `entityType`、`entityKey`、`ownerGameNodeId`、`entryFlags`；`entryFlags` 当前必须为 `0`。
   - 回归测试覆盖正常 round-trip、非法 `bool`、非零 `statusFlags` / `entryFlags`、截断数据、尾随字节以及空 / 多条 assignment 列表。

2. 在 `GameNode` 建立 mesh ready 状态机并向 `GM` 上报
   - `GameNode` 新增启动编排状态：最近一次 `meshReady` 结论、最近一次上报时间、是否存在 `1205` 在途请求，以及“全部目标 `Gate` 是否都已注册且具备有效心跳参数”的聚合判断。
   - `HandleGateConnectionStateChanged`、`HandleGateRegisterResponse`、`HandleGateHeartbeatResponse`、`ResetGateSessionStates` 与 `HandleClusterNodesOnlineNotify` 统一收敛到“重算 mesh ready 并按需上报”的路径，保证 `true -> false` 与 `false -> true` 都能被 `GM` 观察到。
   - 只有在 `GM` 会话已注册、`allNodesOnline = true` 且当前没有在途 `1205` 上报时，`Game` 才发送 mesh ready 报告；收到重复结论时保持幂等，不重复刷包。
   - 一旦 `GM` 断线、`allNodesOnline` 被撤销、任一 `Gate` 会话掉线 / 失去注册闭环，就必须撤销本地 `meshReady` 结论，并为后续 ownership 失效留出统一回滚点。

3. 在 `GM` 聚合全部 `Game` 的 mesh ready 并生成 ownership 快照
   - `GM` 新增按期望 `GameNodeId` 维护的 mesh ready 注册表，记录最新 `meshReady`、`reportedAtUnixMs` 与最近一次可接受的路由通道；仅消费来自已注册 `Game` 会话的合法 `1205`。
   - `GM` 在注册完成、心跳超时、节点重连替换、`allNodesOnline` 变化或收到新的 `1205` 后，统一重算“全部期望 `Game` 是否 mesh ready”；只有结论从未满足变为满足时，才广播当前 `1202` 全量快照。若 bootstrap 分配表尚未构建，则在该时刻按策略构建一次；当前实现不在 `GM` 内重复构建该表，`assignmentEpoch` 固定为 `1`。
   - bootstrap ownership 目录先由 `GM` 内部维护，当前下发给 `Game` 的 `1202` 必须是一张完整的 `ServerStubEntity -> OwnerGameNodeId` 分配表；本里程碑先采用最简单的随机分配策略，后续可在不改变协议语义的前提下扩展为基于 Stub 负载预估或其他策略的分配。
   - 当 `allNodesOnline` 或 mesh ready 聚合条件失效时，`GM` 撤销“当前轮次已可用”的状态，但不提前实现 `clusterReady`；后续 `M3-15` 可直接复用该回滚点。

4. 在 `GameNode` 消费 ownership 并落地本地 Stub 初始化挂点
   - `GameNode` 新增 ownership 本地状态：最近一次接受的 `assignmentEpoch`、全量 assignment 快照、分配给当前 `GameNodeId` 的 owned Stub 列表，以及最近一次 ownership 同步时间。
   - `HandleGmMessage` 需要像处理 `1204` 一样优先识别 `1202` 单向通知，校验 envelope 后解码消息体；旧 `assignmentEpoch`、非法 payload 或注册前到达的通知都必须拒收并记录协议错误。
   - 成功消费 `1202` 后，`Game` 只根据属于自己的 assignments 触发本地 `ApplyStubOwnership()` 挂点；当前阶段挂点只更新可测试状态 / 日志，不直接创建 managed Stub。
   - 当 ownership 被新轮次覆盖，或支撑前提（`allNodesOnline` / Gate mesh / GM 注册链路）失效时，`Game` 需要清空旧 owned Stub 状态，避免 `M3-15`、`M4-05` 之后误用过期 ownership。

5. 回归测试与文档同步
   - 为 `1205` / `1202` 新增独立 codec 测试，覆盖固定长度与变长 assignment 列表场景。
   - 为 `GameNode` 新增测试：`1 Game × 2 Gate` 下只有全部 `Gate` 注册 / 心跳闭环成立后才向 `GM` 上报 `meshReady = true`；任一 `Gate` 失效或 `allNodesOnline = false` 时撤销结论；收到 `1202` 后只保留本节点 owned assignments，并忽略旧 epoch。
   - 为 `GM` 新增测试：未等到全部期望 `Game` mesh ready 前不得广播 ownership；全部满足后广播一次 `1202`；节点失效或 mesh ready 撤销后不沿用旧轮次。
   - 若实现过程中需要补充长期文档，应同步更新 `docs/PROCESS_INNER.md`、`docs/STARTUP_FLOW.md` 与 `docs/DISTRIBUTED_ENTITY.md` 中对 bootstrap ownership 来源与本地挂点边界的描述，保持与代码一致。

## 验证出口

1. `Game` 在仅完成 `Game -> Gate` 直连但尚未拿到全部 `Gate` 注册 / 心跳成功前，不得向 `GM` 报告 `meshReady = true`。
2. `GM` 只有在全部期望 `Game` 都报告 `meshReady = true` 时，才会向全部在线 `Game` 下发 `Inner.ServerStubOwnershipSync (1202)`；当前实现中的 `assignmentEpoch` 固定为 `1`。
3. `Game` 收到旧轮次或非法 envelope 的 ownership 通知时会拒收，且不会覆盖当前已接受的 `assignmentEpoch` 与 owned Stub 列表。
4. `Game` 成功消费 ownership 后，能够稳定暴露“当前轮次下本机负责哪些 Stub”的本地状态，并在支撑条件失效时撤销该状态，为 `M3-15` 的 ready 聚合留出干净起点。
