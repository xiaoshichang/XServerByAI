# M2-08 设计说明（ZeroMQ over TCP 监听端）

1. 在 `src/native/ipc` 内新增 `ZmqPassiveListener`，当前阶段只覆盖“节点间监听端”语义；socket 类型固定为 `ZMQ_ROUTER`，与 `M2-07` 的 `ZmqActiveConnector` 形成明确的 `DEALER -> ROUTER` 组合，不提前扩展为通用多模式收发层。
2. `ZmqPassiveListener` 继续遵循当前项目“单 owner 线程 + standalone `asio` 事件循环”的模型：不把 `ZMQ_FD` 直接桥接成 Asio descriptor，而是沿用 Asio 定时器驱动的非阻塞 pump，在 owner 线程上串行处理收发与生命周期。
3. 监听端构造时绑定 `asio::io_context` 与共享 `ZmqContext`，配置项只暴露当前已知需求：本地 bind endpoint、Asio 轮询间隔、`sndhwm/rcvhwm`、握手超时与基础 socket 选项；socket 统一设置 `ZMQ_LINGER = 0` 与 `ZMQ_ROUTER_MANDATORY = 1`，保证停止时不拖尾，且对未知 routing id 的发送会显式失败而不是静默丢弃。
4. `Start()` 同步完成 `zmq_bind()`，成功后通过 `ZMQ_LAST_ENDPOINT` 记录实际绑定地址；对 `tcp://127.0.0.1:*` 这类 wildcard bind，调用方可通过只读接口拿到实际端口，供后续测试或运行时注册复用。
5. 业务收发统一采用“routing id frame + 单帧 payload”约束：接收回调上抛 `routing id` 与完整消息字节，发送接口显式接收目标 `routing id` 与消息体；若收到缺少 payload 的异常帧序列，或收到超过两帧的 multipart 消息，则主动 drain 剩余 frame、记录错误并丢弃该条消息，保持当前阶段“一个内部包对应一个 ZeroMQ message”的模型稳定。
6. 公开 API 保持 owner-thread 语义，不引入 mutex、strand 或内部工作线程；状态只收敛为 `Stopped / Listening` 两种，状态回调与消息回调都在绑定的 Asio 线程上执行，便于后续 `M3-02` 的 GM 控制面监听和 `M4` 阶段的服务端链路复用。
7. 自动化验证继续采用最小 native 测试可执行程序，至少覆盖：无效配置与未启动发送、listener wildcard bind 后与 `ZmqActiveConnector` 建立 `DEALER -> ROUTER` 往返收发、`Stop()` 生命周期回落，以及异常 multipart 输入被拒绝且不会污染后续有效消息处理。
