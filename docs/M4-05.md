# M4-05 Gate 会话管理表与 Avatar 绑定设计

本文档定义 `M4-05` 的实现范围：在 `M4-04` 已有的 Gate HTTP 鉴权、`conversation reservation` 与 KCP 会话准入基础上，补齐 Gate 侧权威会话管理表，把 `sessionId`、`accountId`、`avatarId` 三者关联起来；同时补一条最小可运行的“账号登录后等待选择 Avatar，再在服务端创建对应 `AvatarEntity` 并绑定到 Gate”流程。当前阶段允许 Avatar 选择与 Avatar 创建使用占位实现，但数据关系必须先落稳。

## 目标与边界

1. `Gate` 需要拥有一份独立于 `conversation reservation` 的权威会话表；`conversation reservation` 继续只负责“HTTP 登录 -> KCP 首包准入”，而 `sessionId/accountId/avatarId` 的长期关系由 `M4-05` 新增的会话表维护。
2. `accountId` 是会话鉴权成功后的账号域标识；`avatarId` 是账号下当前进入游戏的角色标识；同一会话在完成 Avatar 选择前允许 `avatarId` 为空。
3. 客户端在 HTTP 登录成功后不能再默认直接进入“已选中 Avatar 并开玩”的状态，而是应进入“账号已登录、等待选择 Avatar”的状态；当前阶段 Avatar 列表与选择入口可以先做占位。
4. 服务端 Avatar 创建要落到真实的 `AvatarEntity` 上，而不是只在 Gate 或 client 本地做一个字符串绑定；同时 `Avatar` 的 `Proxy` 需要绑定到当前承载该玩家入口的 `GateNodeId`，并且所有新建 `AvatarEntity` 都必须向 `OnlineStub` 注册，作为后续广播、在线态维护与顶号处理的统一入口。
5. `M4-05` 只要求打通“会话建表、账号绑定、Avatar 选择占位、AvatarEntity 创建占位、Gate 侧会话绑定完成”这条闭环；不要求在本里程碑内完成完整的 `Client -> Gate -> Game` 通用业务转发框架，也不要求实现跨 Gate / 跨 Game 迁移。

## 实现清单

1. 在 Gate 内补齐权威会话表与索引
   - 基于现有 `ClientSession` 和 `ClientNetwork`，新增 `Gate` 持有的 `sessionId -> SessionRecord` 权威表，并补 `accountId -> sessionId`、`avatarId -> sessionId`、`sessionId -> avatar proxy metadata` 等最小索引。
   - `SessionRecord` 至少记录 `sessionId`、`conversation`、`accountId`、当前 `avatarId`、`gateNodeId`、目标 `gameNodeId`、鉴权时间、最近活跃时间与关闭状态，明确与 `conversation reservation` 的职责边界。
   - KCP 会话创建成功后，Gate 需要立刻把 `sessionId` 与 reservation 中携带的 `accountId` 绑定起来，形成真正的已鉴权会话记录。

2. 调整客户端登录后的状态机，进入“等待选择 Avatar”
   - `client` 侧保留当前 `Account` 与 `Avatar` 两层模型，但把 HTTP 登录成功后的默认行为改为“缓存 `Account` 与会话上下文，等待玩家选择 Avatar”，不再自动绑定本地 Avatar。
   - 新增一个占位的 Avatar 选择入口，例如无参的 `selectAvatar` 命令；当前阶段由 client 本地随机生成临时 `avatarId/avatarName` 并提交给 Gate，命令入口与状态切换必须是真实的。
   - `status` 输出需要明确区分“账号已登录但尚未选角”、“已发起选角但仍在等待服务端创建确认”以及“Avatar 已确认创建并绑定”几个阶段。

3. 在 Gate 侧补一条最小 Avatar 选择与绑定流程
   - Gate 需要提供一个最小的“选择 Avatar”入口，允许客户端在已有已鉴权会话上提交临时生成的 `avatarId`；实现形式可以是 Gate 本地控制消息或临时 HTTP/KCP 占位消息，但必须以 `sessionId/accountId` 为校验依据。
   - 选择 Avatar 时，Gate 要校验该会话已完成账号绑定、目标 Avatar 尚未被其他活跃会话占用，并把 `sessionId -> avatarId`、`avatarId -> sessionId` 关系写入权威表。
   - 当前阶段服务端不从 DB 读取 Avatar 列表；客户端与服务端大部分数据都可以使用临时随机占位数据，但接口语义要按“一个 Account 下可以选择一个 Avatar 进入游戏”来组织。

4. 让 Gate 驱动 Game 创建对应的 `AvatarEntity`
   - 在 Gate 选角成功后，补一条最小的 Gate -> Game 内部消息或直接调用挂点，驱动目标 `Game` 为该 `avatarId` 创建对应的 `AvatarEntity`。
   - `Game` 侧创建的必须是真实 `AvatarEntity` 实例，并注册到现有 `EntityManager` / runtime state 中，而不是只记录一条临时字符串映射。
   - 该 `AvatarEntity` 的寻址语义应使用 `Proxy`，且 `Proxy.routeGateNodeId` 绑定为当前承载会话的 Gate `NodeID`，为后续 `M5-09` 的 Proxy 消息路径做准备。
   - 所有新建 `AvatarEntity` 都必须同步向 `OnlineStub` 注册，先把“账号/角色已在线”的统一登记点建立起来，后续广播、在线列表、顶号与踢旧会话等逻辑都以这份 `OnlineStub` 注册关系为准。
   - `AvatarEntity` 创建完成后，`Game` 需要回送一条最小成功/失败结果给 `Gate`，再由 `Gate` 明确确认给 client，避免 client 只能做本地乐观假设。

5. 让 Gate 会话绑定上 Game 与 Avatar 归属关系
   - 选角成功后，Gate 的 `SessionRecord` 需要同步补齐 `avatarId`、承载该 Avatar 的 `gameNodeId`、以及后续转发所需的最小路由信息。
   - 若 Avatar 创建失败、选角失败或会话断开，Gate 必须回滚或清理未完成的 `sessionId/accountId/avatarId` 绑定，避免留下悬挂索引。
   - 当前阶段允许继续沿用单会话固定绑定单 Game 的简单策略，不引入迁移、重连接管或多端登录冲突迁移。

6. 补齐验证与文档
   - Native 测试至少覆盖：KCP 会话打开后 `sessionId <-> accountId` 绑定建表、未选角会话状态、Avatar 选择后 `sessionId/accountId/avatarId` 三向关联、重复占用 Avatar 拒绝、断开清理。
   - Managed / Game 测试至少覆盖：创建 `AvatarEntity` 并注册到 `EntityManager`、为 Avatar 生成 `ProxyAddress` 且 `routeGateNodeId` 正确指向 Gate、并验证 `OnlineStub` 收到了对应 Avatar 的注册。
   - Client 测试至少覆盖：HTTP 登录成功后处于“待选角”状态、`selectAvatar` 占位命令能进入“待确认”状态、服务端确认后本地 Avatar 才进入可用状态、未确认前不能继续依赖 Avatar 的本地业务命令。

## 验证出口

1. 客户端完成 HTTP 登录并建立 KCP 会话后，Gate 能生成稳定的 `sessionId`，并把该 `sessionId` 与 `accountId` 绑定进权威会话表。
2. 客户端在未选择 Avatar 前处于明确的“已登录、待选角”状态；`move` / `buyWeapon` 等依赖 Avatar 的操作不会误认为已经进入游戏。
3. 客户端提交无参的 `selectAvatar` 占位选择后，Gate 能把 `sessionId`、`accountId`、`avatarId` 关联起来，并拒绝非法或冲突的选择。
4. 服务端在选角成功后会真实创建对应的 `AvatarEntity`，且该 Avatar 的 `Proxy` 明确绑定到当前 Gate，并同步在 `OnlineStub` 中完成注册；随后 client 会收到明确的创建成功确认。
5. 会话关闭或创建失败时，Gate 不会残留脏的 `sessionId/accountId/avatarId` 索引关系。
