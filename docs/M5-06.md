# M5-06 EntityManager、Game 本地 Stub 初始化与 ready 回调设计

本文档定义 `M5-06` 的实现范围：在 `M5-04` / `M5-05` 已有的 `ServerEntity`、`ServerStubEntity` 基类之上，补齐 managed 侧的实体注册表、ownership 驱动的本地 Stub 实例化流程，以及由实体自身触发的 ready 回调语义，为后续 `M5-07` 的实体消息分发与 `M6-02` 的 native↔managed 宿主接线提供稳定基础。

## 目标与边界

1. 本特性优先解决 managed 侧“谁持有全部实体实例”“Game 如何依据 ownership 创建本地 Stub”“本地 Stub ready 结论从哪里来”三件事，不在本里程碑内完成客户端业务消息处理、Tick 驱动或 `GameNode -> GameNativeOnMessage/OnTick` 的完整宿主闭环。
2. `EntityManager` 必须成为 `Game` 本地所有 `ServerEntity` 的唯一注册表；后续实体消息分发、Tick、持久化与路由都只能基于这张表继续扩展，而不是重新引入平行单例容器。
3. `ServerStubEntity` 的真实 `EntityId` 仍由 managed `ServerEntity` 构造时生成 GUID；启动期 ownership 中的 `unknown` 只能作为 `GM` 下发的占位值，不能继续作为 `Game` 本地 ready 判定的真值来源。
4. 当前仓库中 `GameNode::ApplyStubOwnership()` 直接按 ownership 生成 GUID 并把 owned stub 立即标记 ready，这属于 `M3-15` 留下的占位路径；`M5-06` 需要把“创建实例与产出 ready 条目”的业务真值迁移到 managed `Game` / `EntityManager` 一侧，native 侧后续只保留桥接职责。
5. `M6-02` 仍负责把 `GameNativeInit` / `GameNativeOnMessage` / `GameNativeOnTick` 真正接入原生 `Game` 进程；因此 `M5-06` 的实现要优先沉淀可直接被 `M6-02` 调用的 managed API 与测试入口，避免把宿主细节硬编码进实体框架。

## 实现清单

1. 在 `Framework` 层建立统一的 `EntityManager`
   - 新增 `EntityManager`，以 `EntityId` 作为主键管理全部本地 `ServerEntity`，提供注册、注销、查找、按类型枚举与快照查询等基础能力。
   - `EntityManager` 在注册时必须阻止重复 `EntityId` 冲突，避免后续消息分发与 ready 聚合时出现二义性。
   - `EntityManager` 对外暴露的读取 API 以只读视图 / 快照为主，避免业务层绕过统一入口直接持有内部可变集合。

2. 为 `ServerStubEntity` 补齐 ready 生命周期钩子
   - 在 `ServerStubEntity` 上新增受保护的 `OnReady()` 生命周期钩子，以及由框架统一触发的 ready 入口，确保“是否 ready”与“ready 后要做什么”都归实体自身语义管理。
   - ready 入口必须保证同一实例只会进入一次 ready 完成态，重复调用保持幂等，避免 ownership 重放或重复初始化时重复触发业务副作用。
   - `ServerStubEntity` 不直接向 `GM` 发消息；它只负责本地生命周期与回调，集群上报仍由 `Game` / 宿主层统一聚合。

3. 在 `GameLogic` 层引入可复用的 `Game` 运行时对象
   - 新增 managed `Game` 运行时类型，持有本地 `nodeId`、`EntityManager`、已解析的 Stub 类型目录与当前轮次 ownership 快照。
   - `Game` 在消费 ownership 时，只为 `ownerGameNodeId == 当前 nodeId` 的条目创建本地 `ServerStubEntity`，未分配给本机的 Stub 只保留在快照中，不实例化。
   - `Game` 依据 `entityType` 从 `ServerStubCatalog` / 反射类型表解析具体 `ServerStubEntity` 类型，并通过统一工厂创建实例，而不是在代码里写死 `MatchService` / `ChatService` / `LeaderboardService` 的 `switch`。

4. 用 managed 实例化结果替换当前占位 ready 真值
   - 当 ownership 中的 `entityId` 为 `unknown` 时，本地 ready 条目必须使用 managed 新建实体实例的真实 `EntityId`，不再由 native 层临时生成独立 GUID。
   - `Game` 在本地 owned Stub 注册到 `EntityManager` 后，统一驱动其进入 ready，并基于 ready 完成结果聚合 `ServerStubReadyEntry` 所需的 `entityType`、真实 `entityId` 与 ready 状态。
   - 这样后续 native 宿主接线时，`GameNode` 不再需要自己猜测哪些 Stub ready，只需要把 ownership 输入交给 managed 运行时，再读取 ready 聚合结果即可。

5. 明确 ownership 重放、轮次切换与撤销行为
   - 对同一 `assignmentEpoch` 的重复 ownership，同一 managed `Game` 实例必须保持幂等：不重复创建相同 Stub，不重复触发 `OnReady()`，也不生成抖动的 ready 快照。
   - 当 ownership 轮次变化或当前节点失去某个 Stub 的 owner 身份时，`Game` 需要把对应实体从 `EntityManager` 中撤销，并清理旧轮次的 ready 聚合结果，避免混用过期实例。
   - 轮次撤销与重建逻辑要和 `docs/DISTRIBUTED_ENTITY.md` / `docs/STARTUP_FLOW.md` 中的 startup 编排语义保持一致，为后续 mesh/ready 回滚留出落点。

6. 补齐 managed 优先的测试覆盖
   - 在 `Framework.Tests` 中新增 `EntityManager` 基础测试：注册、重复 ID 拒绝、注销、按类型查询、快照稳定性。
   - 新增 ownership 驱动的 `Game` 运行时测试：只实例化本机 owned Stub、ready 后产出真实 GUID `entityId`、重复 ownership 幂等、失去 owner 时撤销实体。
   - 现有 `node_game_gm_session_tests` 中关于 ready GUID 的原生占位断言，后续要改为消费 managed 运行时输出；本特性先把可替换的 managed API 与测试基线准备好。

## 验证出口

1. 任意 `ServerEntity` / `ServerStubEntity` 进入 `EntityManager` 后，都能通过稳定的 `EntityId` 被唯一查询到；重复 `EntityId` 注册会被拒绝，而不是悄悄覆盖。
2. `Game` 在收到 ownership 快照后，只会为本机负责的 Stub 创建本地实例，并把这些实例纳入 `EntityManager`；未分配给本机的 Stub 不会被错误实例化。
3. 本地 ready 条目中的 `entityId` 必须来自真实 managed `ServerStubEntity.EntityId`，不能继续依赖 native 占位 GUID 或 `unknown`。
4. 同一 `assignmentEpoch` 的 ownership 重放不会重复创建实体或重复触发 `OnReady()`；轮次切换或 owner 撤销后，旧实体与旧 ready 快照会被清理。
5. `M6-02` 后续只需要负责宿主接线与消息投递，即可复用本特性沉淀下来的 `Game` / `EntityManager` / ready 聚合能力，而不必重写实体初始化语义。
