# M1-09 设计说明（进程注册与心跳消息结构）

1. 本条目为 `Gate` / `Game` 向 `GM` 建立启动控制面时使用的注册与心跳消息定义稳定结构，并以 `docs/PROCESS_INNER.md` 作为后续长期维护的协议总文档；本条目只固定消息语义与字段，不承担运行时实现。
2. 在 `msgId` 登记层固定两类共享控制消息：`Inner.NodeRegister`（`1000`）与 `Inner.NodeHeartbeat`（`1100`）。注册请求、成功响应与失败响应都复用同一 `msgId`；心跳则只保留请求与成功响应，不再定义错误响应体。
3. `Inner.NodeRegister` 请求用于声明发起节点身份、`NodeID`、`Inner` 发布地址、构建版本、能力标签与负载快照；注册成功响应返回 `heartbeatIntervalMs`、`heartbeatTimeoutMs` 与 `serverNowUnixMs`；注册失败响应返回 `errorCode` 与 `retryAfterMs`。
4. `Inner.NodeHeartbeat` 请求用于维护已建立的注册会话，并附带轻量负载刷新；心跳成功响应继续返回最新心跳参数与 `serverNowUnixMs`。当前协议只定义成功响应；未知、畸形或 envelope 非法的心跳由接收方记录并忽略，恢复流程交由连接状态变化与后续注册逻辑处理。
5. 注册与心跳消息共享一组基础结构：`Endpoint`、`LoadSnapshot` 与 `ProcessType`。这些共享结构由协议层统一定义，避免多个 codec 或运行时模块各自维护相似但不兼容的字段布局。
6. 当前默认拓扑为 `1 GM + N Gate + M Game`。`Gate` / `Game` 的稳定逻辑身份统一使用 `NodeID`，格式为 `Gate0`、`Gate1`、`Game0`、`Game1`；`nodeId` 既是协议字段，也是后续注册表、路由和 ready 聚合的主键。
7. 当前默认心跳参数为 `heartbeatIntervalMs = 5000`、`heartbeatTimeoutMs = 15000`。发起方必须接受响应方在成功响应中的覆盖值，并保证后续实现满足 `heartbeatIntervalMs < heartbeatTimeoutMs`。
8. 本条目需要与 `docs/PROJECT.md`、`docs/MSG_ID.md`、`docs/ERROR_CODE.md` 和 `docs/PROCESS_INNER.md` 同步维护，保证节点角色、消息名、错误返回与共享结构只有一个权威来源。
