# M5-07 SpaceEntity / AvatarEntity 占位设计

本文档定义 `M5-07` 的实现范围：在 `M5-04` 已有的 `ServerEntity` 基类之上，补齐两个后续业务会直接依赖的实体类型占位：`SpaceEntity` 与 `AvatarEntity`。当前阶段只建立类型与语义边界，不在本里程碑内实现消息收发、迁移协议、持久化或业务属性细节。

## 目标与边界

1. `SpaceEntity` 代表场景实体，负责承载“一个场景/房间/地图”的业务上下文；当前阶段只要求具备可被实例化、可被识别的实体类型身份。
2. `AvatarEntity` 代表玩家角色实体，负责承载“一个玩家角色”的业务上下文；当前阶段只要求具备可被实例化、可被识别的实体类型身份。
3. `SpaceEntity` 必须明确为不可迁移实体，`AvatarEntity` 必须明确为可迁移实体；这两个结论应通过实体类型语义直接表达，而不是依赖外部配置或文档约定。
4. 本里程碑只做占位，不实现实际业务字段、跨节点迁移流程、消息处理函数、Tick、持久化读写或客户端同步。

## 实现清单

1. 新增 `SpaceEntity` 类型
   - 继承 `ServerEntity`。
   - 对外暴露稳定的实体类型名，供后续 `Mailbox`/`Proxy`、日志与调试输出复用。
   - 覆写迁移能力语义，使其始终返回不可迁移。

2. 新增 `AvatarEntity` 类型
   - 继承 `ServerEntity`。
   - 对外暴露稳定的实体类型名，供后续 `Mailbox`/`Proxy`、日志与调试输出复用。
   - 覆写迁移能力语义，使其明确返回可迁移。

3. 为后续消息寻址预留最小接入点
   - 两种实体都应能被 `EntityManager` 统一持有与查询。
   - 后续 `M5-08` / `M5-09` 不应再通过字符串硬编码去猜测实体是否可迁移，而应直接读取实体类型语义。

## 验证出口

1. `SpaceEntity` 与 `AvatarEntity` 都可以作为合法的 `ServerEntity` 派生类型被构造与注册。
2. `SpaceEntity.IsMigratable()` 的语义稳定为 `false`，`AvatarEntity.IsMigratable()` 的语义稳定为 `true`。
3. 后续里程碑在判断“该实体应该走 `Mailbox` 还是 `Proxy`”时，已经可以直接基于这两个占位类型的迁移语义继续实现。
