# M5-04 ServerEntity 基类（Id/属性/生命周期）设计

本文档定义 `M5-04` 的实现范围：在 `M5-03` 已经提供的 managed 项目骨架之上，把当前仅含 `Guid EntityId` 占位的 `ServerEntity` 扩展为可承载实体基础元数据、`EntityProperty` 属性标记与生命周期状态机的正式基类，为后续 `M5-05`、`M5-06`、`M5-10`、`M5-11` 与 `M5-12` 提供统一落点。

## 目标与边界

1. 本特性只解决 `ServerEntity` 的基础身份、实体属性标记与生命周期管理，不实现 `ServerStubEntity` 的 ready 语义、EntityManager 注册表、消息分发、Tick 驱动、脏标记同步或持久化适配器。
2. `EntityId` 继续由 `ServerEntity` 在构造时生成 `Guid`，并在实例整个生命周期内保持稳定；后续 `GM` 与 `Game` 之间回填的真实实体 ID 仍以这里生成的值为准。
3. 属性方案采用“实体自身 CLR 属性 + `EntityProperty` attribute 标记”的方式，而不是额外的运行时属性容器；业务代码继续直接访问真实属性，当前阶段只负责完成标记声明，不实现运行时 metadata 发现、缓存或属性操作接口。
4. 生命周期能力覆盖 managed 侧实体实例的本地激活、迁移中、停用与销毁边界；其中“迁移中”需要进入正式状态机，但 `OwnerGameNodeId` 切换、路由解析、跨节点迁移协议与启动编排仍由后续里程碑负责。
5. 实现必须与 `docs/DISTRIBUTED_ENTITY.md` 中“同一实体状态修改应串行回到同一 ExecutionLane”的约束保持兼容，因此基类需要明确非法生命周期转换的保护，而不是让调用方自由拼接状态。

## 实现清单

1. 补齐实体基础元数据
   - 保留现有 `EntityId` 生成逻辑，并补充 `EntityType`、`IsMigratable()` 与当前 `LifecycleState` 等基础查询能力，作为诊断、后续路由与持久化扩展的统一入口；其中基类 `EntityId` 自身也通过 `EntityProperty(EntityPropertyFlags.AllClients | EntityPropertyFlags.Persistent)` 声明为实体基础属性。
   - 实体迁移能力不再通过单独的 `MobilityType` 枚举或构造参数表达，而是改为 `ServerEntity` 上可覆写的 `IsMigratable()`；默认实体返回 `true`，`ServerStubEntity` 等特殊实体可以通过覆写返回 `false`。

2. 建立实体属性标记模型
   - 新增 `EntityPropertyAttribute`，用于标记 `ServerEntity` 派生类上需要参与迁移和存盘序列化的真实 CLR 属性；未标记属性默认视为运行期辅助状态，不进入这两类序列化边界。
   - `EntityPropertyAttribute` 允许通过 flags 参数声明属性特性，例如同步范围与是否持久化；当前约定的同步相关值包括 `ServerOnly`、`ClientServer` 与 `AllClients`，并可与 `Persistent` 组合使用。典型写法为 `EntityProperty(EntityPropertyFlags.AllClients | EntityPropertyFlags.Persistent)`。
   - 实体属性继续保留为普通属性定义与直接访问，不额外引入字典式 property bag，避免热路径读写因为统一容器而产生额外开销。
   - 当前阶段不在运行时实现属性 metadata 发现、缓存、校验或统一操作入口；后续如需高效访问，可再通过 source generator 生成针对具体实体的 property 操作代码。

3. 建立包含迁移态的生命周期状态机
   - `ServerEntity` 需要从“已构造”进入“已激活/可运行”，并在需要时进入“迁移中”，最终进入“已停用/已销毁”等终态；当前阶段先提供统一的状态表达与切换入口，不额外实现严格的状态转换合法性校验。
   - 可迁移实体与非迁移实体的差异在本里程碑先体现在 `IsMigratable()` 上；更严格的“哪些状态允许进入 `Migrating`”约束延后到后续里程碑再补。
   - 提供受保护的生命周期钩子，允许派生类在激活、进入迁移、迁移完成/取消、停用与销毁阶段补充自身逻辑；但当前阶段不把真实迁移协议、Stub ready、Tick 注册或持久化回调直接塞进这些钩子。
   - 生命周期公共 API 应保持同步、轻量与可测试，后续如需异步持久化、跨语言驱动、真正的跨节点迁移编排或更严格的状态机守卫，可在此基础上继续扩展。

4. 为后续里程碑预留稳定扩展点
   - `M5-05` 可以在不重写基础状态机的前提下，让 `ServerStubEntity` 复用身份、属性与生命周期能力，并补充全局服务语义。
   - `M5-06` 可以围绕基类公开的生命周期状态与只读元数据构建 EntityManager，而不需要再猜测实体是否完成本地初始化。
   - `M5-10`、`M5-11` 与 `M5-12` 后续分别在 Tick、脏标记与持久化场景复用这套“真实属性 + 标记元数据”模型，而不是再引入一套独立的属性容器语义。

5. 补充 Framework 层测试
   - 在 `src/managed/Tests/Framework.Tests` 中新增测试，覆盖 `EntityId` 稳定性、默认元数据、`EntityProperty` 标记可声明性，以及包含 `Migrating` 在内的生命周期状态切换与钩子行为。
   - 需要补一组派生测试实体，验证生命周期钩子触发顺序与次数，确保后续 `ServerStubEntity`、`PlayerEntity`、`SpaceEntity` 接入时不会因为基类状态机缺陷而表现不一致。

## 验证出口

1. 新建任意 `ServerEntity` 派生类实例时，都能获得非空且稳定的 `EntityId`，并能读取到一致的基础元数据。
2. 实体类型可以稳定地使用 `EntityProperty` 对真实 CLR 属性进行标记，并通过 flags 组合声明同步范围、持久化等特性，为后续迁移/存盘生成代码提供语义来源；当前阶段不额外引入运行时 metadata 管线。
3. 生命周期切换入口会统一更新当前状态，并触发对应钩子；更严格的合法性约束与异常校验留待后续里程碑补充。
4. 现有 `ServerStubEntity` 与占位服务类在引入新的基类能力后仍能保持可编译，并为下一步 `M5-05` 扩展全局服务语义留出直接落点。
