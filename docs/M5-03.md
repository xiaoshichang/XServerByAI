# M5-03 设计说明（C# Framework 基础项目）

1. 在 `src/managed/` 下新增独立类库项目 `Framework`，程序集与根命名空间统一为 `XServer.Managed.Framework`；将它加入 `XServerByAI.Managed.sln`，并建立 `Framework -> Foundation`、`GameLogic -> Framework` 的依赖方向。`Foundation` 继续承载通用契约与互操作共享类型，实体框架抽象不直接塞入 `GameLogic`，避免后续业务项目各自复制基类。
2. 在 `Framework` 项目中先落位分布式实体框架的最小公共类型骨架，但暂不引入独立的 `EntityType` 或 `EntityKey` 描述对象；实体类别直接通过 `ServerEntity` / `ServerStubEntity` 及其后续派生类型反映，唯一身份统一约定为 `Guid` 类型的 `EntityId`。当前阶段的基础地址抽象需要显式区分 `Mailbox` 与 `Proxy`：二者都先检查本地实体，远程时统一经 Gate 转发；`Mailbox` 额外携带目标 `GameNodeId` 或等价目标地址，供 Gate 直接转发；`Proxy` 额外携带路由表所在 Gate 信息，供 Gate 查表解析当前 owner。
3. 新增 `ServerEntity` 抽象基类文件与目录骨架，但明确将 `EntityId(Guid)`、属性容器、生命周期入口、框架上下文、Tick、脏标记与持久化钩子留给 `M5-04`、`M5-10`、`M5-11`、`M5-12` 等后续条目实现。`M5-03` 的职责是把“未来所有状态型实体都应继承同一个基类”这件事定下来，而不是一次性做完整功能。
4. 新增 `ServerStubEntity` 抽象基类文件，使其从 `ServerEntity` 继承，并在类型层面固定“全局服务 / 共享服务入口”的语义位置；同时通过注释、命名或最小元数据接口把“由 `GM` 在启动阶段统一分配承载 Game、运行期不可迁移”的约束沉淀下来，但不抢跑实现 `M5-05` 的 ready、ownership 聚合或业务能力。
5. 为 `Framework` 项目补充最小可编译的装配标记与命名空间布局，并同步调整 `GameLogic` 占位代码对新项目的引用，确保后续 native 导出层可以逐步接入框架类型，而不是继续只依赖 `Foundation`。如果后续还会增加 managed 业务项目，也统一通过 `Framework` 复用实体抽象。
6. 在 `src/managed/Tests/` 下新增面向 `Framework` 的测试项目，占位最小编译与继承关系验证：至少覆盖 `ServerStubEntity` 继承自 `ServerEntity`、`Mailbox` / `Proxy` 的地址元数据与本地优先约定可被验证，以及基础公共类型可被 `GameLogic` 引用。测试当前只验证结构、地址语义与依赖方向，不提前验证生命周期或完整路由行为。
7. 同步更新与 managed 项目结构强相关的文档，例如 `docs/PROJECT.md` 与需要体现依赖方向的设计说明；同时在实现中严格遵守 `docs/CONVENTIONS.md` 的 C# Allman 风格。整个 `M5-03` 只建立“项目 + 抽象 + 寻址语义 + 依赖方向 + 编译验证”的基础，不越权实现后续里程碑的运行时语义。
