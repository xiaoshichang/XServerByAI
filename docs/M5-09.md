# M5-09 通过 Proxy 发送消息

本文档定义 `M5-09` 的实现范围：在 `M5-08` 已有的实体消息发送框架之上，补齐“通过 `Proxy` 给实体发送消息”的统一路径。`Proxy` 适用于当前 owner 可能变化、调用方不能长期缓存 `GameNodeId` 的实体，典型场景是 `AvatarEntity`。

## 目标与边界

1. 调用方持有 `ProxyAddress` 时，应能通过统一入口向目标实体投递消息，而不需要自己知道该实体当前实际归属哪个 `Game`。
2. `Proxy` 只解决“动态 owner 解析 + 投递”的问题，不重定义业务实体生命周期，也不实现完整的迁移协议状态机。
3. `Proxy` 依赖 `Gate` 侧已有的会话与路由数据模型；因此本里程碑必须复用 `M1-12` / `M4-05` 的路由语义，而不是再引入一套新的 owner 目录格式。

## 实现清单

1. 统一 `Proxy` 发送入口
   - 输入至少包含 `EntityID`、`routeGateNodeId`、`msgId` 与消息负载。
   - 发送入口先尝试本地实体查找；命中时直接走本地实体分发。
   - 本地未命中时，把请求交给 `ProxyAddress.RouteGateNodeId` 指向的 `Gate`，由其负责解析当前 owner 并继续转发。

2. 明确路由解析语义
   - `Gate` 根据本地路由目录或后续 owner 解析结果找到当前目标 `Game`。
   - 调用方不得缓存某一次解析出来的 `GameNodeId` 并长期复用；每次远程 `Proxy` 发送都应允许重新解析。
   - 解析失败时，应优先归类为路由/寻址错误，而不是目标实体业务错误。

3. 与实体模型的关系
   - `AvatarEntity` 默认通过 `Proxy` 寻址。
   - `SpaceEntity` 与 `ServerStubEntity` 不应强制复用 `Proxy`；它们在目标固定时优先走 `MailBox`。

## 验证出口

1. 本地存在目标实体时，`Proxy` 发送能够直接命中本地实体，不经过远程 owner 解析。
2. 本地不存在目标实体时，`Gate` 能够根据 `Proxy` 提供的路由上下文完成 owner 解析并转发。
3. 目标 owner 变化后，后续 `Proxy` 发送无需修改调用方持有的地址对象，也能继续命中新的 owner。
