# M5-09 通过 Proxy 发送消息

本文档定义 `M5-09` 的实现范围：在 `M5-08` 已有的 `CallStub(...)`、`GameNodeRuntimeState` 与 transport 抽象基础上，补齐“通过 `Proxy` 给普通实体发送消息”的统一路径。当前代码里 `AvatarEntity` 已经绑定了 `ProxyAddress`，并且在创建时会向 `OnlineStub` 做一份在线登记，但运行时还没有一条完整的“通过 `Proxy` 投递实体消息”闭环，也没有一条“由 GM 触发，经 `OnlineStub` 广播给所有在线 Avatar”的统一路径；本里程碑要把这两条链路一起补齐。

## 目标与边界

1. 调用方持有 `ProxyAddress` 时，应能通过统一入口向目标实体投递消息，而不需要自己知道该实体当前实际归属哪个 `Game`。
2. 所有 `AvatarEntity` 在 `Game` 上创建完成时，都必须把自己的 `Proxy` 语义注册到 `OnlineStub`，作为后续在线广播的统一数据源，而不是只保留一份“账号在线”的弱信息。
3. `GM` 需要提供一个控制 HTTP API，按你的要求先使用 `GET /boardcase?msg=...` 这个入口，由 `GM -> OnlineStub -> 在线 Avatar` 路径完成广播。
4. `Proxy` 只解决“动态 owner 解析 + 投递”的问题，不重定义业务实体生命周期，也不实现完整的迁移协议状态机。
5. 当前里程碑优先复用 `M5-08` 已有的“本地优先 + transport 抽象 + `GameNativeOnMessage` 中继”模式，不重新引入通用 `DispatchMessage(...)` / `OnMessage(...)` 大而全框架；更完整的 RPC 语义继续留给 `M4-05A` / `M4-06`。
6. `GM` 当前不运行 managed 业务循环，因此 `/boardcase` 不能假设 `GM` 本地直接持有 `OnlineStub` 实例；它必须先基于当前 Stub ownership 找到 `OnlineStub` 所在 `Game`，再由该 `Game` 上的 `OnlineStub` 继续广播。

## 实现清单

1. 在 `ServerEntity` 上增加专用 `Proxy` 发送契约
   - 为普通实体补一个与 `CallStub(...)` 平行的发送入口，例如 `CallProxy(ProxyAddress address, uint msgId, ReadOnlyMemory<byte> payload)`。
   - 同时补最小接收端消息模型与错误码语义，范围只覆盖“实体收到一条通过 `Proxy` 投递的消息”，不扩展成完整 RPC 框架。
   - 调用参数校验至少要覆盖空地址、空 `EntityId`、空 `RouteGateNodeId`、`msgId = 0` 等基础错误。

2. 在 `GameNodeRuntimeState` 内实现本地优先的 `Proxy` 路由
   - `GameNodeRuntimeState` 需要像处理 `CallStub(...)` 一样，先校验源实体是否已注册，再优先按 `EntityId` 查找本地目标实体。
   - 当目标实体就在本地时，直接命中本地实例，不经过远程 transport，也不依赖 `Gate` owner 解析。
   - 当本地查找未命中时，运行时只能把消息交给 `ProxyAddress.RouteGateNodeId` 对应的 transport 边界，由外层 `Gate` 按最新会话/路由目录继续解析 owner。

3. 把 Avatar 的 `Proxy` 注册升级成 `OnlineStub` 的正式在线目录
   - `AvatarEntity` 在创建成功时，除了现有在线登记外，还要把自己的 `ProxyAddress` 语义同步注册到 `OnlineStub`；注册项至少应稳定表达 `EntityId`、`RouteGateNodeId`、`SessionId`、`GateNodeId`、`GameNodeId` 与展示名。
   - `OnlineStub` 需要对外提供稳定的在线 Avatar 快照读取能力，供后续广播逻辑遍历当前在线玩家。
   - 这份注册表的主语要从“Avatar 作为在线记录”提升为“Avatar 可通过哪条 Proxy 路径被寻址”，避免广播逻辑再自行拼接地址。

4. 增加与 `CallStub(...)` 对齐的 `Proxy` transport / interop 边界
   - 新增 `IProxyCallTransport` 及其 managed/native 实现，职责与 `IStubCallTransport` 平行，但输入改为 `EntityId + RouteGateNodeId + msgId + payload`。
   - 为 forwarded `Proxy` 消息增加独立 codec / envelope，并在 `GameNativeOnMessage` 中补一条解码和分发路径，保持后续原生宿主只需要负责中继。
   - envelope 中不直接写死目标 `GameNodeId`，避免调用方把一次 owner 解析结果误当成长期真值。

5. 给 `OnlineStub` 增加“广播给所有在线 Avatar”的最小能力
   - `OnlineStub` 增加一个明确的广播入口，输入至少包含广播 `msgId` 与消息体文本 / 二进制负载。
   - 广播执行时，`OnlineStub` 遍历当前在线 Avatar 的 `Proxy` 快照，并通过本里程碑新增的 `CallProxy(...)` 路径向每个在线 Avatar 投递消息。
   - 当前阶段只要求 managed 内部广播链路闭环，不要求一口气把 `Avatar -> Client` 的最终用户可见消息协议全部做完；对客户端真正展示的那一跳，后续可以继续挂到 `M4-07` / `M6-03`。

6. 给 `GM` 控制 HTTP 增加 `/boardcase` 广播入口
   - `GM` 控制 HTTP 服务新增 `GET /boardcase?msg=...` 路由，按你给的 URL 形式接收广播文本。
   - 路由处理时，`GM` 需要先根据当前 Stub ownership 找到 `OnlineStub` 的 owner `GameNodeId`，再把广播命令转发到该 `Game` 上的 `OnlineStub`，而不是在 `GM` 本地直接执行 managed 逻辑。
   - HTTP 返回至少要能明确区分参数错误、`OnlineStub` owner 不存在 / 不可达、以及广播命令已成功投递到 `OnlineStub` 三类结果。

7. 用测试锁定 Avatar 注册、GM 广播入口与 fan-out 语义
   - Framework tests 需要覆盖“创建 Avatar 后，`OnlineStub` 注册表里能读到对应 `Proxy` 信息”。
   - Framework / loopback transport tests 需要覆盖“`OnlineStub` 接到广播命令后，能对所有在线 Avatar 逐个走 `Proxy` 路径投递”。
   - Native tests 需要覆盖 `GM` 控制 HTTP `/boardcase?msg=...` 的参数校验、`OnlineStub` owner 解析与转发行为。
   - 失败测试至少覆盖未知目标 Avatar、不可达 route gate、`OnlineStub` owner 缺失，以及空 `msg` 请求四类情况。

## 当前实现落点

1. `AvatarEntity.Proxy` 已经能稳定保存 `EntityId + RouteGateNodeId`，并且 `GameNodeRuntimeState.TryCreateAvatarEntity(...)` 已经会向 `OnlineStub` 注册在线 Avatar；`M5-09` 需要把这份注册提升成可直接用于广播的 `Proxy` 目录。
2. `GameNodeRuntimeState` 当前只具备 `CallStub(...)` 的本地短路和远程转发能力；`M5-09` 需要在同一层补齐普通实体的 `Proxy` 路由。
3. `GameNativeExports`、`ManagedNativeStubCallTransport` 与 `RelayStubCallCodec` 已经为跨语言 stub 转发提供了一套模式，`Proxy` 应尽量沿用这套组织方式。
4. `GM` 现有已经具备 `GmControlHttpService`，并支持 `/healthz`、`/status`、`/shutdown`；`/boardcase` 会在同一服务里扩展，但它的业务执行目标应是 `OnlineStub` owner `Game`，不是 `GM` 本地 managed runtime。

## 验证出口

1. 本地存在目标实体时，`Proxy` 发送能够直接命中本地实体，不经过远程 owner 解析。
2. 所有新建 `AvatarEntity` 都会把自己的 `Proxy` 信息同步登记到 `OnlineStub`，后续广播不需要临时拼地址。
3. `OnlineStub` 收到广播命令后，能够遍历当前在线 Avatar，并通过 `Proxy` 路径逐个投递。
4. `GM` 的 `GET /boardcase?msg=hello` 能把广播请求准确转发到 `OnlineStub` owner，并得到可区分的成功 / 失败结果。
5. 对未知目标实体、不可达 route gate、`OnlineStub` owner 缺失与空广播消息等失败路径，调用结果能够稳定区分原因。
