# M2-11 设计说明（心跳消息编解码）

1. 在 `src/native/net/message` 中实现独立的 `HeartbeatCodec` 模块，专门负责 `Inner.NodeHeartbeat`（`msgId = 1100`）消息体的编码、解码与最小语义校验。本条目只覆盖消息体，不把 `PacketHeader` 封包、消息分发或 ZeroMQ 收发耦合进 codec。
2. 心跳消息固定分成 3 类：`HeartbeatRequest`、`HeartbeatSuccessResponse` 与 `HeartbeatErrorResponse`。字段定义与顺序必须与 `docs/PROCESS_INNER.md` 保持一致，保证后续 `GM`、`Gate`、`Game` 可以直接复用同一套稳定协议布局。
3. `LoadSnapshot` 作为共享嵌套结构参与心跳请求编码，字段顺序固定为 `connectionCount`、`sessionCount`、`entityCount`、`spaceCount`、`loadScore`，全部使用网络字节序 `uint32`。当前阶段 unknown 值统一写 `0`，不为 codec 再引入额外可选语义。
4. `HeartbeatRequest` 的 wire layout 固定为 `sentAtUnixMs(uint64)`、`statusFlags(uint32)`、`load(LoadSnapshot)`；`statusFlags` 当前必须为 `0`，否则编码或解码都应返回 `InvalidStatusFlags`。`HeartbeatSuccessResponse` 固定为 `heartbeatIntervalMs(uint32)`、`heartbeatTimeoutMs(uint32)`、`serverNowUnixMs(uint64)`，并要求 `heartbeatIntervalMs < heartbeatTimeoutMs`。
5. `HeartbeatErrorResponse` 固定为 `errorCode(int32)`、`retryAfterMs(uint32)`、`requireFullRegister(bool)`。其中 `bool` 必须使用 `uint8(0/1)` 编码；解码时遇到非法布尔值必须显式返回 `InvalidBoolValue`，避免上层把损坏 payload 当成真实错误响应继续处理。
6. codec 层统一返回 `HeartbeatCodecErrorCode`，对底层二进制读写错误进行稳定映射，并额外补充协议语义错误：`InvalidStatusFlags`、`InvalidHeartbeatTiming`、`TrailingBytes`。这样节点层可以把“缓冲区不够”和“协议字段不合法”区分开来，日志与测试也能直接复用这些错误码。
7. 自动化验证至少覆盖：请求、成功响应、失败响应 roundtrip；稳定 wire bytes；`statusFlags != 0` 拒绝；`heartbeatIntervalMs >= heartbeatTimeoutMs` 拒绝；非法布尔值拒绝；输入截断、尾部多余字节与空输出参数等失败路径。现有 `tests/cpp/heartbeat_codec_tests.cpp` 已按这组语义建立覆盖。
8. 本条目交付完成后，后续 `M3-05`、`M3-09`、`M3-11` 只需关注心跳业务时序与会话状态，不再重复定义心跳 payload 结构。
