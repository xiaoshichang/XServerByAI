# M2-11 设计说明（心跳消息编解码）

1. 在 `src/native/net/message` 内新增独立的 `HeartbeatCodec` 模块，专门承载 `Control.ProcessHeartbeat`（`msgId = 1100`）消息体的编码、解码与基础语义校验；当前阶段只处理消息体，不把 `PacketHeader` 封包逻辑重复塞进该模块。
2. 对外显式建模三类消息体：心跳请求、心跳成功响应、心跳失败响应；三者分别对应 `docs/PROCESS_CONTROL.md` 中登记的字段集，避免调用方直接按裸字节位置手写读写。
3. 共享负载快照 `LoadSnapshot` 在该模块内作为稳定结构复用，字段顺序固定为 `connectionCount`、`sessionCount`、`entityCount`、`spaceCount`、`loadScore`，统一使用网络字节序 `uint32` 编码。
4. 请求体编码顺序固定为 `registrationId(uint64)`、`sentAtUnixMs(uint64)`、`statusFlags(uint32)`、`load(LoadSnapshot)`；其中 `statusFlags` 当前为保留字段，编解码阶段要求其必须为 `0`，防止保留位被静默带入后续协议演进。
5. 成功响应体编码顺序固定为 `registrationId(uint64)`、`heartbeatIntervalMs(uint32)`、`heartbeatTimeoutMs(uint32)`、`serverNowUnixMs(uint64)`；编解码阶段要求 `registrationId != 0` 且 `heartbeatIntervalMs < heartbeatTimeoutMs`，与控制面文档约束保持一致。
6. 失败响应体编码顺序固定为 `errorCode(int32)`、`retryAfterMs(uint32)`、`requireFullRegister(bool)`；`bool` 继续复用 `BinaryReader` 的 `0/1` 约束，非法布尔值直接判为解码失败。
7. 所有可失败接口统一返回 `HeartbeatCodecErrorCode`，内部对 `BinaryWriter` / `BinaryReader` 的 `BufferTooSmall`、`LengthOverflow`、`InvalidBoolValue` 与 `InvalidArgument` 做集中映射；心跳特有校验另外补充 `InvalidRegistrationId`、`InvalidStatusFlags`、`InvalidHeartbeatTiming` 与 `TrailingBytes`。
8. 自动化验证至少覆盖：请求/成功响应/失败响应 roundtrip、精确线缆字节布局、保留 `statusFlags` 非零拒绝、`heartbeatIntervalMs >= heartbeatTimeoutMs` 拒绝、布尔字段非法值拒绝、输入截断与尾部多余字节拒绝，以及空输出参数等参数错误路径。
