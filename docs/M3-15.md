# M3-15 Game 服务 ready 上报与 Gate 开放通知设计

本文档定义 `M3-15` 的实现范围：在 `M3-14` 的 ownership 下发与本地 Stub 初始化挂点之上，补齐 `Inner.GameServiceReadyReport (1203)`，并由 `GM` 基于 ready stub 条目聚合后向 `Gate` 下发 `Inner.ClusterReadyNotify (1201)`。当前阶段只实现启动闭环所需的最小路径，暂不扩展 `GM` / `Gate` 断线、重连替换与 ready 回滚鲁棒性。

## 目标与边界

1. 本特性只解决 `Game -> GM` 的 stub ready 上报，以及 `GM -> Gate` 的开放通知，不在本里程碑内实现客户端鉴权、业务转发或真实 managed `ServerStubEntity` 生命周期。
2. `Game` 向 `GM` 上报时，只携带当前已经 ready 的 stub 条目；如果当前 `Game` 在该 `assignmentEpoch` 下没有负责任何 stub，则不发送 `1203`。每条 ready 条目都必须携带本地实例的真实 GUID `entityId`。
3. `GM` 是否开放 `Gate`，以 ownership 表中的 stub 是否都已经收到 ready 条目为准，而不是以“每个 `Game` 是否都上报过 `localReady = true`”为准。
4. 当前仓库尚未接入真实 managed Stub 初始化，因此 `M3-15` 继续以 native 占位状态生成 ready 条目；后续 `M6-02` 再把 ready 结论替换为真实业务初始化结果。
5. `ClusterReadyNotify (1201)` 在当前阶段只承担“open gate”通知职责；`Gate` 只消费该结论，不参与本地推导。

## 实现清单

1. 补齐 `1203` 协议、编解码与校验
   - 在 `InnerClusterCodec` 中新增 `kInnerGameServiceReadyReportMsgId`、`ServerStubReadyEntry` 与 `GameServiceReadyReport` 结构，以及对应的 wire-size / encode / decode 入口。
   - `ServerStubReadyEntry` 至少包含 `entityType`、`entityId`、`ready`、`entryFlags`；`GameServiceReadyReport` 至少包含 `assignmentEpoch`、`localReady`、`statusFlags`、`entries`、`reportedAtUnixMs`。
   - `statusFlags` 与 `entryFlags` 当前必须为 `0`，回归测试覆盖 round-trip、非法 bool、非法 flags、截断数据、尾随字节与长度溢出。

2. 在 `GameNode` 建立最小 stub ready 上报路径
   - `GameNode` 在消费 ownership 后，基于分配给自己的 stub 列表生成当前 ready 条目；当前阶段占位实现默认把本机 owned stub 立即标为 ready。
   - `1203.entries` 只上报当前 ready 的 stub 条目，不附带其他 `Game` 的 stub；如果本机没有任何 owned stub，则直接跳过本次 `1203` 上报。
   - `Game` 收到 `1202` 时如果 `entityId` 仍为占位值 `unknown`，则在本地创建实体实例时生成真实 GUID ID，并在 `1203` 中回传给 `GM`。
   - `Game` 只在 `GM` 会话已注册、`meshReady = true` 且已经拥有当前 `assignmentEpoch` 的 ownership 时发送 `1203`；同一 `assignmentEpoch` 不重复刷包。

3. 在 `GM` 按 stub 条目聚合 ready 并下发 open gate 通知
   - `GM` 在收到合法 `1203` 后，按 `nodeId + assignmentEpoch` 记录该 `Game` 当前上报的 ready 条目，并把匹配到的 Stub 记录 `entityId` 从 `unknown` 更新为 ready 条目中的真实 ID。
   - `GM` 使用当前 ownership 表逐条校验：只要每一个 `ServerStubOwnershipEntry` 都能在对应 owner `Game` 的 ready 条目里找到匹配且 `ready = true` 的记录，就认为集群已 ready。
   - `GM` 在首次满足全部 stub ready 后，向全部在线 `Gate` 广播 `ClusterReadyNotify (1201)`，驱动 `Gate` 打开客户端入口。

4. 复用现有 `GateNode` 的 `clusterReady` 消费逻辑
   - `GateNode` 继续只消费 `1201` 单向通知，不增加任何本地推导分支。
   - 当前阶段只验证 `clusterReady = true` 的 open gate 路径，不扩展因 `GM` / `Gate` 断线导致的关闭或回滚逻辑。

5. 回归测试与文档同步
   - 扩展 `inner_cluster_codec_tests` 覆盖 `GameServiceReadyReport` / `ServerStubReadyEntry`。
   - 为 `node_game_gm_session_tests` 新增验证：`Game` 在收到 ownership 后上报全部 ready stub；当没有 owned stub 时不发送 `1203`。
   - 为 `node_gm_register_tests` 新增验证：`GM` 只有在 ownership 表中的所有 stub 都出现在 ready 条目里后，才向 `Gate` 下发 `clusterReady = true`。
   - 如需补充长期文档，同步更新 `docs/PROCESS_INNER.md`、`docs/STARTUP_FLOW.md` 与 `docs/DISTRIBUTED_ENTITY.md` 中与 stub ready 聚合相关的描述。

## 验证出口

1. `Game` 在拿到 ownership 后，只上报本机当前 ready 的 stub；如果本机没有负责任何 stub，则不会发送 `1203`。ready 条目中的 `entityId` 必须是本地实例 GUID，而不是占位值 `unknown`。
2. `GM` 不会因为收到单个 `Game` 的 `localReady = true` 就提前开放 `Gate`；只有 ownership 表中的全部 stub 都有 ready 条目时，才会下发 open gate 通知。
3. `Gate` 在收到 `clusterReady = true` 后打开客户端入口，形成 `Game -> GM -> Gate` 的最小启动闭环。
