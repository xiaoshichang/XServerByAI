# M5-08 通过 MailBox 发送消息

本文档定义 `M5-08` 的实现范围：在 `M5-06` 已有的实体注册表与本地实体查询能力之上，补齐“通过 `MailBox` 给实体发送消息”的统一路径。`MailBox` 适用于目标 `GameNodeId` 已知、且实体语义上不需要动态 owner 解析的场景，例如 `SpaceEntity` 与已完成 ownership 分配的 `ServerStubEntity`。

## 目标与边界

1. 调用方持有 `MailBoxAddress` 时，应能通过统一入口向目标实体投递消息，而不需要自己区分“本地调用”还是“远程经 Gate 转发”。
2. `MailBox` 只解决“已知目标 `GameNodeId` 的静态寻址”问题，不负责 owner 解析；需要动态解析 owner 的场景属于 `M5-09`。
3. 当前阶段只定义实体消息发送闭环，不扩展新的会话路由模型，不引入 `Game -> Game` 直连业务链路。

## 实现清单

1. 统一 `MailBox` 发送入口
   - 输入至少包含 `EntityID`、`targetGameNodeId`、`msgId` 与消息负载。
   - 发送入口先尝试本地实体查找；命中时直接走本地实体分发。
   - 本地未命中时，转为远程转发请求，并把 `targetGameNodeId` 作为静态目标交给 `Gate`。

2. 明确远程转发语义
   - `Gate` 只根据 `MailBoxAddress.TargetGameNodeId` 做单跳转发，不再额外解析 owner。
   - 远程投递失败时，应区分“目标节点不存在/不可达”与“目标实体本身拒绝处理”两类问题，避免把寻址错误混入实体业务错误。

3. 与实体模型的关系
   - `SpaceEntity` 默认通过 `MailBox` 寻址。
   - `ServerStubEntity` 在 ownership 已知时，也可以复用同一条 `MailBox` 发送路径。
   - `AvatarEntity` 不在本里程碑内强制走 `MailBox`；它的动态寻址场景留给 `M5-09`。

## 验证出口

1. 本地存在目标实体时，`MailBox` 发送能够直接命中本地实体，不经过远程转发。
2. 本地不存在目标实体但 `targetGameNodeId` 有效时，消息能够经 `Gate` 转发到指定 `Game`。
3. `MailBox` 发送失败时，可以区分地址无效、节点不可达与实体业务拒绝三类基础失败。
