# M5-08 通过 MailBox 发送消息

本文档定义 `M5-08` 当前实现范围：在 `M5-06` 已有的 `EntityManager`、`ServerStubEntity` ownership 快照与本地 Stub 实例化能力之上，先落地“通过静态 owner 给 `ServerStubEntity` 发消息”的闭环。

当前版本不再走通用 `ServerEntity.DispatchMessage(...)` / `OnMessage(...)` 方案，而是收敛成 Stub 语义明确的 `CallStub(...)`：

1. 调用方传入目标 Stub 的类型名称、`msgId` 与消息负载。
2. 运行时根据当前 ownership 快照判断目标 Stub 属于哪个 `Game`。
3. 如果目标 Stub 就在本地，直接命中本地实例。
4. 如果目标 Stub 在远端，则把消息转发到对应 owner `Game`。

`Proxy` 所需的动态 owner 解析仍然属于 `M5-09`，不在本里程碑内实现。

## 目标与边界

1. `ServerEntity` 应具备统一的 `CallStub(...)` 发送入口，调用方不需要自己区分“本地调用”还是“远程转发”。
2. 当前寻址键使用 Stub 的类型名称，而不是实体 GUID；这样可以直接复用启动期由 `GM` 下发的 `ServerStubEntity -> OwnerGameNodeId` 分配表。
3. 当前里程碑只覆盖 `ServerStubEntity` 的静态 owner 调用链路，不扩展 `AvatarEntity` 等可迁移实体的动态寻址。
4. 当前里程碑只定义 managed 侧消息路由与可测试闭环；native / Gate 的真实接线继续留给后续宿主接入阶段。
5. 当前路由层不额外携带来源实体类型；如果业务需要这类上下文，应由消息体自己显式表达。

## 实现清单

1. 统一 `CallStub(...)` 入口
   - 在 `ServerEntity` 上提供 `CallStub(string targetStubType, uint msgId, ReadOnlyMemory<byte> payload)`。
   - 调用时只要求目标 Stub 类型名称、`msgId` 与消息负载，不再要求目标实体 GUID。

2. 基于 ownership 的本地优先寻址
   - `GameNodeRuntimeState` 维护当前节点本地 owned Stub 实例与 ownership 快照。
   - 如果目标 Stub 当前 owner 就是本节点，并且实例已存在，则直接调用本地目标 Stub。

3. 远程转发抽象
   - 本地未命中时，由 `GameNodeRuntimeState` 根据 ownership 解析目标 owner `GameNodeId`。
   - 远程调用通过 `IStubCallTransport` 抽象转发，保持“统一经外层路由/宿主发送”的边界。
   - 路由层只关心目标 Stub 类型、目标 `GameNodeId` 与消息体，不额外注入来源类型元数据。

4. 错误语义
   - 对外返回 `StubCallResult` / `StubCallErrorCode`。
   - 当前基础失败至少区分 `UnknownTargetStub`、`TargetNodeUnavailable` 与 `StubRejected`。

5. 验证用占位 Stub 行为
   - `MatchStub` 当前接受并记录收到的 Stub 调用，作为 `M5-08` 的最小可验证接收端。
   - `OnlineStub` 通过基类 `CallStub(...)` 即可向 `MatchStub` 发起调用。

## 当前实现落点

1. `ServerEntity` 提供统一的 `CallStub(...)` 发送入口，`ServerStubEntity` 提供 Stub 接收钩子 `OnStubCall(...)`。
2. `GameNodeRuntimeState` 负责 ownership 查询、本地短路与远程转发决策。
3. `IStubCallTransport` 负责把“源 Stub 类型 + 目标 Stub 类型 + 目标 GameNodeId + 消息体”交给外层链路。
4. `MatchStub` 当前会记录收到的调用内容，便于测试验证 `OnlineStub -> MatchStub` 的消息路径。

## 验证出口

1. 当 `OnlineStub` 与 `MatchStub` 同属一个 `Game` 时，`CallStub(nameof(MatchStub), ...)` 能直接命中本地 `MatchStub`，且不经过远程转发。
2. 当 `OnlineStub` 与 `MatchStub` 分别归属于不同 `Game` 时，`OnlineStub` 能依据 ownership 将消息转发到 `MatchStub` 所在 `Game`，并由远端 `MatchStub` 成功接收。
3. 当目标 Stub 类型不存在 ownership、目标节点不可达或目标 Stub 拒绝处理时，调用结果能够稳定区分失败原因。
