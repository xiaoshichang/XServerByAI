# M3-01 设计说明（统一 `xserver-node` 运行时入口与节点角色分派）

1. 保持统一入口命令行形式为 `xserver-node <configPath> <selector>`，但把参数校验、配置加载与运行时启动拆成独立 helper；`main()` 只负责组装调用链与统一返回退出码，避免后续 GM/Gate/Game 生命周期逻辑继续堆在单个函数里。
2. `selector` 继续只用于配置选择，真正的运行时身份统一从 `NodeConfig` 派生：至少显式收敛出 `processType`、稳定 `nodeId`、`configPath` 与原始 `selector`；其中 `GM` 运行时实例标识固定为 `GM`，Gate/Game 复用已完成配置阶段校验过的 `nodeId`（如 `Gate0`、`Game1`），避免入口层再手写一套字符串推导逻辑。
3. 在 `src/native/node` 内新增独立的运行时上下文与入口 API，例如 `NodeRuntimeContext` / `RunNodeProcess(...)`；该层负责承接配置加载结果、进程参数、当前进程 `pid` 与后续角色启动所需的共享对象，让 `main()` 与具体角色实现之间只有一个稳定边界。
4. 角色分派统一以 `processType` 为准，显式映射到 `GM`、`Gate`、`Game` 三类 role runner；本条目先在 `src/native/gm`、`src/native/gate`、`src/native/game` 放置最小占位 runner 或 dispatcher 入口，保证后续 `M3-02`、`M3-08`、`M3-10` 只扩展各自角色实现，而不再改写 CLI 与总入口控制流。
5. 统一入口在进入 role runner 之前完成基础日志初始化，`LoggerOptions` 直接复用 `NodeConfig.process_type` 与 `NodeConfig.instance_id`，确保从进程启动、配置失败、角色分派失败到正常退出都能带上正确的 `processType` / `instanceId` 语义。
6. 运行生命周期统一依赖 `MainEventLoop`：`RunNodeProcess(...)` 负责创建主事件循环并把运行时上下文交给选中的 role runner；本阶段 role runner 只需要提供可测试的最小 start/stop 骨架，不抢先实现 GM 控制监听、Gate/Game 控制链路或注册心跳。
7. 错误传播采用两层边界：参数/配置错误继续在进入事件循环前以清晰字符串返回；角色分派或事件循环启动失败则通过统一 runtime error 返回到 `main()`，由入口打印消息并返回非零退出码，避免后续每个角色各自发明一套退出约定。
8. 自动化验证优先覆盖可复用的 runtime API，而不是依赖真实子进程拉起：至少验证参数不足与非法 `selector` 失败、`gm`/`gate0`/`game0` 能派生出正确的 `processType + nodeId`、角色分派命中正确 runner、日志/运行时上下文字段与配置一致，以及 role runner / `MainEventLoop` 启动失败时错误能透传回入口层。
