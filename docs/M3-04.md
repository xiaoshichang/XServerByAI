# M3-04 设计说明（GM 注册请求处理与响应）

1. 在 `src/native/node` 内把当前只负责收包日志的 GM 控制监听继续向上收敛成“可处理控制消息的运行时入口”。`InnerNetwork` 需要暴露稳定的“收到路由消息”回调与“按 routingId 回包”能力，但仍保持自己只负责 ZeroMQ ROUTER 传输，不直接感知注册表或控制面业务语义。
2. `GmNode` 在本条目内引入最小控制面状态：持有 `ProcessRegistry`、注册请求处理入口以及固定的默认心跳参数（当前沿用 `5000ms / 15000ms`）。所有注册请求的解析、注册表写入和响应发送都在 GM 主事件循环线程内同步执行，不提前引入额外工作线程、异步队列或跨模块 runtime 包装。
3. 注册请求处理只接管 `Control.ProcessRegister`（`msgId = 1000`）。GM 收到消息后先用 `PacketCodec` 校验外层包头，再用 `RegisterCodec` 解码 payload；只有“`msgId = 1000`、非 `Response`、非 `Error`、`seq != 0`”的消息会进入本条目处理路径。无法安全回显上下文的损坏包继续记录日志并丢弃；能够明确归因到注册语义的失败，则进入稳定错误响应分支。
4. 本条目需要把注册语义与现有错误码对齐。已在 `docs/PROCESS_CONTROL.md` / `docs/ERROR_CODE.md` 中登记的 `3000 Control.ProcessTypeInvalid`、`3001 Control.NodeIdConflict`、`3002 Control.ServiceEndpointInvalid` 必须直接落地；如果实现时发现 `nodeId` 为空、`processFlags` 非零这类场景无法用现有码值准确表达，则先补充文档登记，再把映射固化到代码与测试，避免运行时出现“日志报错但协议语义无编号”的灰区。
5. 对于通过校验的注册请求，GM 需要把 `RegisterRequest` 转换为 `ProcessRegistryRegistration` 并写入现有 `ProcessRegistry`。其中 `routingId` 直接取自 ROUTER 链路、`lastHeartbeatAtUnixMs` 初始化为 GM 处理当前注册时的服务器时间、`serviceReady` 初始为 `false`。本条目只负责“建立活动注册”，不提前实现断线剔除、重连接管或心跳超时回收，这些后续分别留给 `M3-05` / `M3-15`。
6. 注册成功响应统一复用 `msgId = 1000` 与原始 `seq`，设置 `PacketHeader.flags.Response`，payload 使用 `RegisterSuccessResponse`，返回当前心跳间隔、超时阈值与 `serverNowUnixMs`。注册失败响应统一设置 `Response | Error`，payload 使用 `RegisterErrorResponse`，并保持错误码、`retryAfterMs`、日志字段与文档约定一致；响应封包继续复用 `PacketCodec + RegisterCodec`，不在 GM 侧手写二进制布局。
7. 为了让后续 `M3-05` / `M3-06` / `M3-12` 能直接复用本条目的输出，GM 侧日志至少要在成功与失败路径记录 `nodeId`、`processType`、`routingIdBytes`、`serviceEndpoint`、`seq` 与结果错误码；同时保留查询当前注册表快照的内部访问点，便于单测直接验证注册成功后条目内容，而不是只靠抓包判断。
8. 自动化验证以端到端的 GM 控制面测试为主：至少覆盖 `Gate/Game` 注册成功并写入注册表、成功响应 flags/msgId/seq 正确、重复 `nodeId` 返回冲突错误、非法 `processType` / `serviceEndpoint` 返回预期错误、无法形成稳定协议上下文的损坏包不会污染注册表，以及同一 GM 监听端口上连续处理多次注册请求的行为稳定。必要时补充 `InnerNetwork` 或独立控制处理单测，但不把纯协议编解码测试重复堆回 `RegisterCodec` 模块。
