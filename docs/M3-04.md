# M3-04 设计说明（GM 注册请求处理与响应）

1. 本条目的目标是在 `GM` 侧真正接住并处理 `Inner.NodeRegister`（`msgId = 1000`）请求，把当前只负责监听与收包的 GM Inner 骨架升级成可写注册表、可回注册响应的最小协调节点。
2. `InnerNetwork` 需要向上层暴露稳定的原始消息入口：除了把 payload 交给 `GmNode` 之外，还要同时带回 ZeroMQ ROUTER 的 `routingId`，以便 `GM` 把“同一条 Inner 链路”映射到注册表项，而不是只根据消息内容猜测来源。
3. `GmNode` 在本条目中维护最小但稳定的注册上下文：`ProcessRegistry`、默认心跳参数（当前为 `5000ms / 15000ms`）、注册请求处理、错误映射与响应发送。请求处理、注册表写入与响应回包都统一在 GM 事件循环线程上执行，不额外引入后台线程或异步队列。
4. `GM` 只处理 `msgId = 1000`、不带 `Response/Error` 标记、且 `seq != 0` 的请求包。外层先用 `PacketCodec` 校验包头，再用 `RegisterCodec` 解码 payload；解码失败、flags 非法、`seq = 0` 或 payload 语义非法时，必须按协议返回可诊断的失败响应或记录稳定日志，而不是把损坏请求静默写入注册表。
5. 注册成功时，`GM` 要把 `RegisterRequest` 转换为 `ProcessRegistryRegistration` 写入 `ProcessRegistry`：保存 `processType`、`nodeId`、`pid`、`startedAtUnixMs`、`innerNetworkEndpoint`、`buildVersion`、`capabilityTags`、`load`、`routingId`，并把 `lastHeartbeatAtUnixMs` 初始化为当前 GM 时间；`innerNetworkReady` 在此阶段先置为 `false`。
6. 注册失败时，`GM` 统一把协议错误映射为稳定 Inner 错误码。当前实现重点覆盖：`3000 Inner.ProcessTypeInvalid`、`3001 Inner.NodeIdConflict`、`3002 Inner.InnerNetworkEndpointInvalid`、`3005 Inner.RequestInvalid`。失败响应仍复用 `msgId = 1000` 与原始 `seq`，只通过 `Response | Error` 标志位区分。
7. 成功响应统一回写 `RegisterSuccessResponse`，即 `heartbeatIntervalMs`、`heartbeatTimeoutMs` 与 `serverNowUnixMs`；失败响应统一回写 `RegisterErrorResponse`，即 `errorCode` 与 `retryAfterMs`。这一层只负责注册协议闭环，不提前承担心跳刷新、超时剔除、ownership 编排或 `clusterReady` 聚合。
8. 自动化验证围绕“GM 可以稳定接收注册、写入注册表并返回正确响应”展开。现有 `tests/cpp/node_gm_register_tests.cpp` 已覆盖：`Gate`/`Game` 顺序注册成功、重复 `nodeId` 冲突、非法 `processType`、非法 `innerNetworkEndpoint`、非法 payload、损坏 packet 无响应，以及注册表中 `routingId` / endpoint / load 等字段的实际落库结果。
