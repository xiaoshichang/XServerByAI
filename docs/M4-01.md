# M4-01 集成 KCP 库并封装会话对象设计

本文档定义 `M4-01` 的实现范围：在现有 `Gate` 侧 `ClientNetwork` 占位骨架之上，补齐 vendored KCP 依赖接入、KCP 控制块包装层，以及后续 `Gate -> Client` 链路复用的客户端会话对象。当前阶段只建立“可被 `M4-02` 继续使用的传输与会话基础设施”，不在本里程碑内打开真实 UDP 监听、接入客户端握手或实现业务消息转发。

## 目标与边界

1. `M4-01` 必须把 KCP 作为仓库内 vendored 第三方依赖接入构建系统，保证后续 native 目标可以稳定复用，而不是在 `M4-02` 中再临时引入外部源码或平台特判。
2. `M4-01` 只负责 KCP 会话基础设施与数据模型，不负责 `Gate` 监听器开放、连接受控开启、会话创建时机、客户端鉴权、心跳踢出或 Gate/Game 转发链路。
3. 会话对象语义必须对齐 [SESSION_ROUTING](C:\Users\shichang.xiao\.codex\worktrees\1a7f\XServerByAI\docs\SESSION_ROUTING.md) 中的 `sessionId`、`SessionState`、`RouteState` 与时间戳约定，哪怕当前阶段尚未把完整路由表落地。
4. `ClientNetwork` 当前占位实现可以继续保持“未真正收包”的状态，但要完成对新会话封装层的持有与最小生命周期挂点，为 `M4-02` 直接接 listener / session create 链路预留接口。

## 实现清单

1. 接入 vendored KCP 依赖与项目侧 CMake 包装
   - 在 `3rd/` 下引入 KCP 上游源码，保持与现有 `spdlog`、`zeromq`、`asio` 一致的 vendored 管理方式。
   - 在 [3rd/CMakeLists.txt](C:\Users\shichang.xiao\.codex\worktrees\1a7f\XServerByAI\3rd\CMakeLists.txt) 中增加项目侧别名目标，例如 `xs::thirdparty::kcp`，避免业务代码直接绑定上游 target 名称。
   - 只暴露 `ikcp` 所需的最小编译入口，不在当前阶段额外引入示例程序、测试程序或平台专属脚本。

2. 在 native 层建立 KCP 控制块包装与配置映射
   - 新增项目自有的 KCP 包装类型，负责 `ikcp_create` / `ikcp_release` 生命周期、`KcpConfig` 到 `ikcp_*` API 的参数映射，以及输入输出缓冲的最小约束。
   - 包装层需要显式支持 [KCP_CONFIG](C:\Users\shichang.xiao\.codex\worktrees\1a7f\XServerByAI\docs\KCP_CONFIG.md) 已定义的全部字段；如果某个字段暂时无法生效，必须在实现说明和测试中明确记录，不允许静默忽略。
   - 包装层只处理“单个 conversation 的 KCP 语义”，不负责 UDP socket、远端地址发现、监听绑定或多会话分发。

3. 定义 Gate 侧客户端会话对象
   - 新增 `ClientSession` 一类的会话封装，至少持有 `sessionId`、远端地址、KCP conversation、`SessionState`、`RouteState`、`playerId` 以及关键时间戳。
   - 初始状态要与 `SESSION_ROUTING` 约定保持一致：会话创建后进入 `Created`，`playerId = 0`，路由默认为 `Unassigned`，未鉴权与未关闭时间戳保持 `0`。
   - 会话对象应把“传输层状态”和“逻辑会话状态”集中在一个稳定抽象内，避免 `M4-02` 以后把 KCP 控制块、地址信息、状态位分散塞回 `ClientNetwork`。

4. 给 `ClientNetwork` 补齐最小会话基础设施挂点
   - 现有 [ClientNetwork.h](C:\Users\shichang.xiao\.codex\worktrees\1a7f\XServerByAI\src\native\node\ClientNetwork.h) / [ClientNetwork.cpp](C:\Users\shichang.xiao\.codex\worktrees\1a7f\XServerByAI\src\native\node\ClientNetwork.cpp) 需要从纯日志占位，升级为“知道如何持有 KCP 配置与会话对象”的骨架。
   - 当前阶段不要求 `ClientNetwork` 真正创建会话，但应预留注册、查询或销毁单个 `ClientSession` 的最小内部接口，使 `M4-02` 可以直接在监听器收到首包后接入。
   - `GateNode` 与 `ClientNetwork` 的既有 `clusterReady` 开关关系保持不变；`M4-01` 不改变“只有收到 `clusterReady = true` 后才运行客户端入口”的产品语义。

5. 补齐回归测试与长期文档同步
   - 为 KCP 包装层新增独立 native 单测，覆盖配置映射、空输入、分片/窗口基础行为、非法参数与资源释放路径。
   - 为会话对象新增单测，覆盖初始状态、状态迁移约束、时间戳更新以及与 KCP 包装实例的绑定关系。
   - 如接入方式、目录职责或日志分类发生变化，同步更新 `3rd/README.md`、`docs/PROJECT.md`、`docs/CONFIG_LOGGING.md` 等长期文档中的 KCP 描述。

## 验证出口

1. 仓库可以在不依赖系统级 KCP 安装的前提下完成配置与编译，native 目标通过项目侧 `xs::thirdparty::kcp` 复用依赖。
2. `KcpConfig` 中列出的参数都能被包装层显式处理，并有测试验证关键映射结果。
3. `ClientSession` 可以在不依赖真实网络监听的情况下独立构造、驱动状态变化，并保持与 `SESSION_ROUTING` 文档一致的默认语义。
4. `ClientNetwork` 在当前阶段仍可作为 `GateNode` 的客户端入口骨架初始化，但内部已经不再只是纯日志占位，而是具备承接 `M4-02` 会话创建流程的基础结构。
