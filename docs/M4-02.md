# M4-02 Gate KCP 监听骨架、受控开放与会话创建设计

本文档定义 `M4-02` 的实现范围：在 `M4-01` 已落地的 `KcpPeer` 与 `ClientSession` 基础上，把 `Gate` 侧 `ClientNetwork` 从“仅持有配置与内存会话索引”的骨架，补齐为真正受 `clusterReady` 控制的 UDP/KCP 监听入口，并在客户端首个 KCP 数据报到达时创建本地会话。当前阶段只建立“可被后续鉴权、转发与心跳继续复用的监听与会话基础设施”，不在本里程碑内实现客户端鉴权、`Gate -> Game` 业务转发、客户端心跳踢出或断线重连。

## 目标与边界

1. `GateNode` 继续只消费 `GM -> Gate` 的 `ClusterReadyNotify (1201)` 结论；`ClientNetwork` 是否真正对客户端开放，必须完全受 `clusterReady` 控制，不能在 `Gate` 启动时自行提前绑定并接受客户端数据报。
2. `M4-02` 负责补齐真实 UDP/KCP 监听、首包驱动的会话创建、最小 KCP 输入与输出循环，以及必要的日志与测试；但不会在本里程碑内落地 `M4-03` 的模拟客户端、`M4-04` 的登录鉴权、`M4-05` 的 `sessionId -> playerId` 会话表、`M4-06` / `M4-07` 的请求响应链路或 `M4-15` 的客户端心跳超时。
3. `ClientSession` 的逻辑语义仍以 [SESSION_ROUTING](./SESSION_ROUTING.md) 为准：`sessionId` 是 `Gate` 本地权威标识，`conversation` 只表示单条 KCP 传输会话，不得反向替代 `sessionId`。
4. 当前 `ClientNetwork` 仅按 `conversation` 建索引，这在真实 UDP 场景下不足以区分“不同客户端恰好使用相同 conv”的情况；`M4-02` 需要把传输层匹配键升级为“远端地址 + conversation”的组合语义，避免跨客户端误复用同一会话。
5. 当前阶段不新增 `clusterReady = false`、GM 断线或其他异常情况下的客户端入口回滚逻辑；若运行期收到这类异常信号，只要求保留可观测日志。正常进程退出时仍沿用既有 `Uninit()` 资源释放路径。

## 实现清单

1. 把 `ClientNetwork` 升级为真实 UDP 监听骨架
   - 在 `ClientNetwork` 内引入基于 `asio::ip::udp::socket` 的监听器状态，使用 `MainEventLoop` 的 `io_context` 完成 bind、异步收包与停止时的 socket 关闭。
   - `Init()` 只做配置校验与监听器对象准备，不提前开放客户端入口；`Run()` 才真正绑定 `gate.<NodeID>.clientNetwork.listenEndpoint` 并启动异步接收，`Stop()` 负责停止接收、关闭 socket 并撤销与监听器相关的运行态。
   - `ClientNetwork` 需要保留当前 `initialized` / `running` 语义与日志分类，避免破坏 `GateNode` 现有 `clusterReady -> Run/Stop` 的控制路径。

2. 明确 Gate 受控开放语义
   - `GateNode` 现有 `HandleClusterReadyNotify()` 继续作为唯一入口：只有在收到 `clusterReady = true` 后才调用 `ClientNetwork::Run()`；本里程碑不扩展 `clusterReady = false`、GM 断线等异常回滚处理，若运行期出现这类情况仅记录日志，不把它们作为 `M4-02` 的实现阻塞项。
   - 在 `ClientNetwork` 尚未 `Run()` 前，不允许任何客户端数据报创建会话，也不允许通过“先建会话后等开放”的方式绕过 `clusterReady`。
   - 本里程碑按单次正常生命周期实现：`ClientNetwork::Run()` 与 `ClientNetwork::Stop()` 在进程生命周期内各只考虑调用一次，不处理重复调用、重入、异常恢复或其他非正常时序。

3. 定义首包驱动的会话创建与传输层索引
   - 当监听器收到新的 UDP 数据报时，先从 KCP 头部提取 `conversation`，再使用“远端地址 + conversation”查找现有会话；若未命中，则分配新的本地 `sessionId` 并创建 `ClientSession`。
   - `ClientNetwork` 内部索引需要从“仅 conversation”扩展到“`sessionId` 权威表 + endpoint/conversation 传输索引”；后续 `M4-05` 可以继续围绕 `sessionId` 扩展更高层会话表，而不会受 KCP `conv` 复用限制。
   - 会话创建时继续复用 `M4-01` 已定义的默认初始状态：`SessionState = Created`、`RouteState = Unassigned`、`playerId = 0`，并记录远端地址与连接时间戳。

4. 补齐最小 KCP 输入、输出与驱动循环
   - 收到数据报后，把字节流喂给对应 `ClientSession::kcp()`；若 `KcpPeer` 产出待发送数据报，则立即通过 UDP socket 回写给该客户端，保证基础 ACK/控制流可以闭环。
   - `ClientNetwork` 需要为已存在会话补齐最小 update/flush 驱动，可使用 `MainEventLoop::timers()` 基于 `kcp.interval_ms` 调度重复任务，统一推进所有活跃会话的 `ikcp_update`。
   - 当前阶段只需要把 KCP 解出的完整 payload 视为“客户端业务消息占位输入”并更新 `lastActiveUnixMs` / 日志；不在本里程碑内继续解析应用协议或转发到 `Game`。

5. 补齐回归测试与长期文档同步
   - 扩展 `node_client_network_tests`：覆盖单次 `Run()` 后真实监听启动、单次正常 `Stop()` 路径下监听关闭、首包创建会话、重复数据报复用同一会话，以及不同远端在相同 `conversation` 下不会冲突。
   - 扩展 `node_gate_gm_session_tests`：验证 `Gate` 在收到 `clusterReady = true` 前不会对客户端入口开放；收到通知后客户端首个 KCP 数据报能够触发会话创建。异常回滚路径暂不纳入 `M4-02` 的测试范围，最多补充日志可观测性断言。
   - 如日志字段、监听职责描述或 KCP 配置落地方式发生变化，同步更新 `docs/CONFIG_LOGGING.md`、`docs/KCP_CONFIG.md` 与相关长期说明，确保后续条目直接复用本里程碑结论。

## 验证出口

1. `Gate` 启动并完成与 `GM` 的 inner 注册前，客户端入口不会提前开放；只有在消费 `clusterReady = true` 后，`ClientNetwork` 才进入真实监听状态。
2. 客户端首个有效 KCP 数据报到达后，`Gate` 能稳定创建一条新的 `ClientSession`，并把其状态保持在 `Created / Unassigned` 的默认语义。
3. 来自同一远端、同一 `conversation` 的后续数据报会复用既有会话；来自不同远端但使用相同 `conversation` 的客户端不会被错误合并成同一会话。
4. `ClientNetwork` 能把 KCP 输出的数据报回写给客户端，并通过定时 update 驱动维持基础 KCP 传输闭环，即使当前阶段尚未接入更高层业务协议。
5. `M4-02` 不要求验证 `clusterReady` 回落、GM 断线或其他运行期异常场景下的监听回滚；若后续遇到这类情况，当前阶段只要求保留清晰日志。正常 `Gate` 进程退出时资源仍应随既有 `Uninit()` 路径释放。
