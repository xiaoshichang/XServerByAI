# M2-14 设计说明（基础 metrics 结构）

1. 在 `src/native/ipc` 内新增独立的 listener metrics 模块，显式建模“当前连接数 + 吞吐累计值”的快照结构；本阶段只提供数据采集与读取能力，不引入日志输出、聚合上报或 Prometheus/HTTP 暴露。
2. metrics 对外统一以只读快照表达，至少包含 `activeConnectionCount`、`receivedMessageCount`、`receivedPayloadBytes`、`sentMessageCount`、`sentPayloadBytes` 与 `snapshotUnixMs`；吞吐先记录为单调递增累计值，由调用方通过两次快照差值换算区间速率，避免在基础层强绑定固定时间窗口。
3. 连接数不依赖 libzmq draft API；`ZmqPassiveListener` 在启动时附带创建稳定版 `zmq_socket_monitor()` 监控通道，只订阅 `ZMQ_EVENT_ACCEPTED`、`ZMQ_EVENT_DISCONNECTED` 与 `ZMQ_EVENT_MONITOR_STOPPED`，把连接建立/断开事件折算为内部 gauge，避免把“见过 routing id”误当成“当前在线连接”。
4. `ZmqPassiveListener` 的 metrics 更新边界保持明确：成功收到一条业务消息后累加 `receivedMessageCount` 与 `receivedPayloadBytes`，成功发送回复后累加 `sentMessageCount` 与 `sentPayloadBytes`；ZeroMQ monitor 事件与空 payload 通知不进入业务消息处理器，也不计入吞吐。
5. listener 对外新增同步快照读取接口，返回复制后的 metrics 结构而不是可变内部引用；调用方可以在运行中或停止后读取最后一次快照，成功 `Start()` 时重置统计，`Stop()` 只终止采集并保留当前值，方便测试与故障诊断。
6. monitor 通道与业务 ROUTER socket 共享同一 `asio::steady_timer` 驱动循环，不额外创建线程；实现需在 `Start()` / `Stop()` 生命周期里完整管理 monitor endpoint、PAIR socket 与残余事件清理，保证不会因为 monitor 建立失败而泄漏资源或让 listener 处于半启动状态。
7. 自动化验证至少覆盖：初始快照全零、业务收发后消息数/字节数累计正确、`Start()` 后重置历史统计、`Stop()` 后快照仍可读取、monitor 驱动的连接数随 dealer connect/disconnect 正确增减，以及 monitor 事件不会伪造成业务消息进入 `SetMessageHandler()`。
