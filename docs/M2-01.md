# M2-01 设计说明（native 日志模块）

1. 在 `src/native/core` 内落地独立日志模块，对外暴露不依赖 `spdlog` 类型的接口，固定提供 `LogLevel`、`ProcessType`、`LoggingConfig`、`LoggerOptions` 与 `Logger`，避免后续配置加载与 C# 日志桥接直接耦合第三方库细节。
2. `Logger` 绑定单个进程实例的静态身份信息：`processType`、`instanceId`、`pid`。每条日志调用只补充 `category`、`message`、可选 `errorCode` / `errorName` 与结构化 `context`，确保输出字段顺序与 `docs/CONFIG_LOGGING.md` 一致。
3. 底层继续使用 vendored `spdlog` 做等级过滤与线程安全封装，但文件输出由项目侧自定义 sink 控制，实现 UTC 日切、同日按大小补充分卷、缺失目录自动创建，以及按实例前缀清理超出上限的旧文件。
4. 日志模块内部统一以 UTC 时间生成文件名与日志行时间戳，文件名格式固定为 `<instanceId>-YYYY-MM-DD.log` 和 `<instanceId>-YYYY-MM-DD.<n>.log`；文本行渲染固定为单行，避免后续采集端解析复杂度上升。
5. 刷盘策略分为两层：普通日志由 `flushIntervalMs` 驱动后台周期刷盘，`Error` / `Fatal` 则在写入后立即触发刷盘，满足规范中的及时落盘要求。
6. 预留字符串与枚举互转辅助函数，至少覆盖日志等级解析、进程类型字符串渲染与基础参数校验，为后续 `M2-02` 的 JSON 配置加载直接复用。
7. 当前阶段先以 native 工程编译通过作为验收基线；自动化日志行为测试后续在统一测试基线完善时补齐，但实现本身要避免把测试前提写死在生产接口里。
