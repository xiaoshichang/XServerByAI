# M5-01 设计说明（`nethost` 初始化与 CLR 加载）

1. 将 .NET 官方 native hosting 编译期依赖以 vendoring 方式纳入 `3rd/dotnet_host/`，统一维护 `nethost.h`、`hostfxr.h`、`coreclr_delegates.h` 与平台对应的 `nethost` 链接库；版本固定为与当前 managed 项目 `net10.0` 基线对齐的 `10.0.0`，不得依赖开发机本地 SDK/host pack 路径完成构建。
2. 在 `3rd/CMakeLists.txt` 中新增项目侧包装 target，统一通过 `xs::thirdparty::nethost` 暴露头文件、平台链接库与 Linux 侧所需的系统库；native 业务代码不得直接依赖上游 host pack 的目录布局或文件名细节。
3. 将 `src/native/host/` 从占位实现扩展为跨平台 CLR 宿主模块，提供显式输入路径的运行时加载 API：调用方传入 `runtimeconfig.json` 与根程序集 `.dll` 路径，宿主模块负责执行 `nethost -> hostfxr -> load_assembly_and_get_function_pointer` 这条装载链路，但在本条目内只完成 CLR 装载与委托准备，不解析具体业务导出入口。
4. 宿主模块应严格复用 `docs/MANAGED_INTEROP.md` 约定的装载顺序：先通过 `nethost` 定位 `hostfxr`，再动态装载 `hostfxr` 导出，调用 `hostfxr_initialize_for_runtime_config` 创建上下文，随后获取 `hdt_load_assembly_and_get_function_pointer` 运行时委托，并将其作为后续 `M5-02` 的函数绑定基础能力缓存到宿主状态中。
5. `M5-01` 仅负责“运行时是否成功装起”这一层，不承担 ABI 校验、`GameNative*` 导出名解析或 `Game` 进程业务接入；这些能力分别留给 `M5-02` 与 `M6-02`。为避免后续条目重复实现，本条目需要同时提供错误码枚举、错误消息映射、平台动态库装载封装与宿主状态生命周期管理。
6. 宿主模块的公开可失败 API 应遵循项目既有约定，返回稳定错误码并落入 `docs/ERROR_CODE.md` 的 `4000-4099` 号段；至少区分 vendored host 依赖缺失、路径无效、`get_hostfxr_path` 失败、`hostfxr` 动态装载失败、运行时初始化失败与运行时委托获取失败等场景。
7. 为保证该条目在不依赖 `M5-02` 的前提下可验证，测试应覆盖成功与失败两类路径：失败路径验证无效 `runtimeconfig` / 根程序集路径或缺失 host 依赖时的稳定错误码；成功路径通过最小 `runtimeconfig.json` 与当前 managed 根程序集产物完成一次 CLR 装载，并断言 `load_assembly_and_get_function_pointer` 委托可用。相关文档需同步更新 `3rd/README.md`、`docs/PROJECT.md` 与必要的构建说明，明确 `dotnet_host` 已成为仓库固定 vendored 依赖的一部分。
